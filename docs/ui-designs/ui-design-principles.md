# UI 设计原则：设计风格约束

本文档用于约束后续使用 pencil.dev 与网页技术栈实现的游戏 UI 原型。这篇文档主要研究了 Kittens Game 的设计风格。

## 1. 总体定位

目标不是做一个现代、华丽、强视觉冲击的游戏界面，而是做一个“文字驱动的系统界面”。玩家面对的不是电影化 HUD，而是一套逐步解锁、逐步理解的控制台、档案柜、通讯面板和资源账本。

核心美学可以概括为：

- 低装饰：界面主要由文本、边框、列表、按钮和简单分区组成。
- 高信息密度：同一屏幕允许同时存在资源、日志、行动入口、状态反馈。
- 慢热：初始内容少，随着机制推进逐渐出现更多入口。
- 系统感强：每个按钮背后都是一个可计算、可积累、可被优化的系统。
- 有黑色幽默和轻微荒诞感：文字可以克制，但不必完全严肃。

对于本项目，可以将整体方向定义为：

> 一座前沿基地的低保真控制系统。玩家不是“看见宇宙”，而是通过控制台、通讯记录、地图格子、资源数字和事件日志来推断宇宙。

## 2. 机制设计原则

设计原则重点不在视觉，而在系统组织方式。后续 UI 原型也应服务这些原则。

### 复用已有资源和建筑

优先让新功能从已有对象中生长出来，而不是不断增加完全独立的新模块。

例如：

- “咖啡机”和“冰箱”在本阶段只提供沉浸反馈，不影响精力、时间、补给或其他数值系统。
- “通讯台”不要只是联系人列表，可以承担任务、事件、交易、紧急消息、情报来源。
- “地图”不要只是导航，可以暴露资源、风险、可行动目标和长期规划，但不直接发起指令。

### 鼓励主动操作，但不强迫持续点击

玩家可以主动点击获得收益或推进事件，但游戏不应只依赖频繁点击。适合的交互节奏是：

- 玩家主动检查控制中心。
- 做出几项关键选择。
- 系统产生累积、等待、冷却、反馈。
- 玩家通过日志和状态变化理解后果。

UI 上要避免过度强调“立即点击很多次”。按钮更像命令、请求、派遣、确认，而不是街机式操作。

### 每个瓶颈应有多种解决路径

如果某个资源、角色、区域或关系成为瓶颈，应该提供多个解决方向。

例如：

- 缺少能源：可以修理设备、探索资源点、联系外部援助、降低设施耗能。
- 通讯中断：可以等待窗口期、修复通讯链路、派人去地图节点、通过联系人转接。
- 角色状态差：可以休息、补给、对话安抚、调整任务安排。

UI 需要让这些路径看起来属于同一个系统，而不是彼此孤立的菜单。

### 每个解法都制造新问题

解决一个问题后，不应只是单纯变强。它应该带来新的消耗、风险、依赖或管理对象。

例如：

- 修好卫星雷达后，玩家获得更多地图情报，但也看到更多威胁。
- 接通新联系人后，玩家获得任务线索，但也引入关系压力或资源请求。
- 处理矿床事件后，资源线索增加，但带来维护、污染、人员安全或运输问题。

这种原则会让 UI 更有生命力：界面不是静态菜单，而是不断展开的管理系统。

### 初期困难，后期用升级缓解

不要一开始就给玩家完整能力。初期界面可以显得简陋、受限、不完整：

- 地图只有少量格子可见。
- 通讯录只有少数联系人在线。
- 控制中心部分设备处于锁定、损坏、未供电状态。
- 统计面板只显示基础数值，复杂指标后续解锁。

随着升级推进，UI 可以逐渐增加：

- 新的状态字段。
- 新按钮。
- 新筛选器。
- 新日志分类。
- 新地图标记。

这种“界面本身逐渐成长”的感觉，是增量游戏的重要美学。

### 设计时考虑后期规模

不要只按原型阶段的 3 个联系人、4 个地图格、5 个资源来设计。界面要假设后期会出现：

- 大量资源。
- 大量地点。
- 大量联系人。
- 多条同时运行的任务。
- 很大的数字。

因此从一开始就要采用可扩展结构：

- 列表可滚动、可折叠。
- 面板可分组。
- 状态行可复用。
- 数字格式要预留长文本空间。
- 地图格子要能承载多层信息，而不只是一个地形名。

### 变化应留在既有规则内

新机制可以有变化，但最好使用玩家已经理解的动词和结构。

例如：

- “通话”“请求”“派遣”“回收”“升级”“查看日志”可以在多个模块重复出现。
- 地图探索、通讯请求、设备维修都可以共享“消耗资源 -> 等待/判定 -> 产出结果”的节奏。
- 不同页面的按钮、状态、反馈方式应保持一致。

这会让复杂系统变得可学习。

## 3. 视觉美学约束

### 文本优先

界面的第一信息载体是文字，而不是插画。图片、图标和色块只用于辅助理解。

建议：

- 地图格可以先用文字表达“平原 / 水 / 森林 / 铁矿床 / 采矿厂”。
- 联系人卡片优先显示姓名、身份、位置、状态、行动按钮。
- 控制中心设备优先显示名称和用途，而不是复杂拟物图标。

避免：

- 大面积装饰性背景图。
- 过多图标替代文本。
- 用视觉特效承担关键反馈。

### 低保真控制台感

网页界面有一种近似旧网页、调试面板、文本游戏的质感。可以使用：

- 浅灰或深色背景。
- 细边框。
- 简单矩形按钮。
- 明确的面板标题。
- 紧凑行距。
- 少量状态色。
- 可见的系统日志。

整体应像“仍在运行的前沿基地软件”，而不是精装修的商业 SaaS。

### 色彩克制

主色建议保持少而稳定。

推荐色彩角色：

- 背景色：灰白、深蓝黑、炭黑或低饱和冷灰。
- 边框色：中性灰。
- 文本色：黑、浅灰或单色终端绿。
- 强调色：只用于可交互项、警告、在线状态、稀有发现。
- 危险色：红或橙只用于真正需要注意的状态。

当前通讯录草图中的蓝色面板可以保留，但应作为“当前活动系统”或“选中条目”的强调色，而不是全局大量铺满。

### 信息密度高，但层级清楚

画面可以密集，但必须让玩家知道先看哪里。

建议层级：

1. 页面标题或当前位置。
2. 关键状态与资源。
3. 可执行动作。
4. 详情说明。
5. 系统日志或次要反馈。

Pencil 原型中可以故意保留边框、留白和模块标题，让界面看起来像可操作的系统面板。

### 保留一点粗糙感

不要把界面打磨得过于圆润、过于现代。适度的粗糙感能强化“边境、故障、临时系统”的主题。

可用方式：

- 简单线框。
- 方形或小圆角卡片。
- 等宽字体用于数字、日志、坐标、资源。
- 普通无衬线字体用于中文说明。
- 少量不完全对称的模块布局。

## 4. 交互与布局原则

### 主界面像房间，也像菜单

控制中心可以同时是空间布局和功能入口。玩家点击“通讯台”“卫星雷达”“中控台”“冰箱”等设备进入子界面。

关键约束：

- 每个设备都是一个明确入口。
- 设备位置可以暗示功能关系。
- 未解锁或损坏设备可以先显示在画面上，但用虚线、灰色或状态标签表示不可用。
- 点击设备后出现二级窗口或切换到对应页面。

### 子界面使用面板叠加

通讯录展开后悬浮在通讯台页面之上，这个方向很好。建议统一以下叠加规则：

- 二级窗口不完全遮挡主上下文。
- 窗口有明确标题。
- 右上或底部提供关闭/收起。
- 窗口内操作完成后，主页面状态或日志产生变化。

这样能形成“操作控制台”的感觉，而不是普通网页跳转。

### 日志是叙事核心

在我们的参考游戏中，Kittens Game 页面中有明显的日志区域，“You are a kitten in a catnip forest.” 这类短句承担了叙事开场。你的原型也应有一个系统日志或事件记录。

日志可以用于：

- 记录玩家操作。
- 提示资源变化。
- 暗示世界观。
- 展示通讯结果。
- 告知失败原因。
- 埋入轻微幽默或不安感。

日志文字应短、具体、可累计。例如：

- “卫星雷达返回了 3 个低置信度信号。”
- “Amy 拒绝接入，她说森林里有东西在听。”
- “冰箱完成了一次毫无必要的自检。咖啡味道像旧电池。”

### 折叠与解锁优先于复杂导航

不要过早引入复杂路由、抽屉导航或大型菜单。增量游戏更适合：

- 折叠面板。
- 标签页。
- 小型弹窗。
- 局部展开。
- 状态驱动的入口出现。

玩家应感觉“这个控制台逐渐长出来”，而不是一开始就面对完整后台系统。

### 操作反馈要即时可见

每次点击至少改变一处可见内容：

- 按钮进入冷却或禁用。
- 资源数字变化。
- 联系人状态变化。
- 日志新增一行。
- 地图格出现新标记。
- 设备显示新的状态标签。

如果操作需要等待，也要显示“进行中”“预计完成”“信号搜索中”等状态。

## 5. 页面级约束

### 控制中心

控制中心是总入口，负责表达“基地仍在运转，但不完整”。

建议保留的视觉元素：

- 大空间线框。
- 固定设备位置。
- 可点击设备块。
- 虚线表示未修复、未供电、未解锁。
- 中控台显示核心资源、状态和统计。

建议设备语气：

- 窗户：观察外部情况，偏叙事与气氛。
- 咖啡机：轻量幽默和沉浸反馈，不影响数值。
- 音频终端：音乐、情绪、基地氛围。
- 冰箱：食物氛围和沉浸反馈，不影响补给数值。
- 通讯台：联系人、请求、任务、事件入口。
- 卫星雷达：地图发现、信号、异常。
- 研究台：科技树、解锁、升级。
- 星际贸易：资源交换、价格波动、关系。
- 星际之门：高风险请求、抽卡或重大事件。

### 通讯台

通讯台应像一个“关系与任务系统”，而不只是电话簿。

联系人卡片建议包含：

- 头像或占位符。
- 名称与称号。
- 当前位置。
- 当前状态。
- 可用动作。
- 关系或信任等级。
- 最近状态摘要。

状态文字比图标更重要。颜色只负责强调：

- 在线 / 可接通：普通或绿色。
- 忙碌 / 行进中：灰色或蓝色。
- 危险 / 紧急：橙色或红色。
- 未知 / 失联：低对比度。

### 通话页面

通话页面不需要做成真实聊天软件。它更适合像任务事件窗口：

- 左侧或上方显示联系人信息。
- 中央显示对话、请求或事件描述。
- 底部提供 2-4 个选择。
- 选择后写入日志，并改变联系人或资源状态。

选项不宜太多。每个选项最好有明显代价或风险。

### 地图

地图可以继续使用像素网格或表格作为基础。关键是让每个格子既是地形，也是系统节点。

每个格子可承载：

- 地形：平原、水、森林、山、沙漠。
- 地块对象：铁矿床、水源、遗迹、信号点。
- 特殊状态：危险、异常、生产中。
- 状态：未探索、已侦察、危险、可派遣、生产中。
- 坐标或编号。

地图不需要一开始漂亮，但要可扩展。后续可以加入：

- hover 详情。
- 点击选中格子。
- 右侧详情面板。
- 可行动目标提示。
- 资源产出预估。
- 探索日志。

## 6. 文案语气

文案应短、冷静、略带异样感。不要过度解释世界观，让玩家从系统反馈中推断故事。

推荐语气：

- “雷达找到了一片没有反射的区域。”
- “Mike 说湖泊今天比昨天更近。”
- “中控台建议你不要再问这个问题。”
- “咖啡机完成了自检。它拒绝说明上一次维护日期。”

避免语气：

- 过度热血。
- 过度可爱。
- 大段设定说明。
- 现代 SaaS 式提示，例如“恭喜你完成配置！”

## 7. Web 实现约束

后续网页原型建议遵守：

- 优先使用 HTML 原生结构：button、dialog、section、ul、table。
- CSS 保持简单，可读，可快速调整。
- 不依赖复杂动画表达核心状态。
- 所有关键反馈都必须有文本形式。
- 数值、日志、联系人状态、地图格数据应数据驱动渲染。
- 组件应按系统拆分：控制中心、设备入口、弹窗、通讯录、联系人卡片、地图格、日志。
- 字体建议使用系统无衬线 + 等宽字体组合。

## 8. Pencil 原型约束

在 pencil.dev 中制作视觉稿时，优先建立可复用组件：

- 设备块。
- 面板容器。
- 小按钮。
- 状态标签。
- 联系人卡片。
- 地图格。
- 日志行。
- 弹窗标题栏。

不要把每个页面画成完全不同的视觉系统。所有页面应像同一套基地软件的不同模块。

Pencil 中的灰盒、线框和占位头像可以保留，不需要过早精绘。这个项目的质感来自系统关系、文本反馈和逐步展开，而不是插画完成度。

## 9. 总结原则

最终 UI 应满足以下约束：

- 像控制台，不像营销页。
- 像系统管理，不像单纯菜单。
- 像逐渐修复的基地，不像一开始完整的软件。
- 用文字讲故事，用数字制造压力，用按钮表达选择。
- 新功能从旧模块里长出来。
- 每个解决方案都引出新的管理问题。
- 视觉克制，交互清楚，反馈即时。

一句话方向：

> 让玩家通过一套粗糙、密集、可扩展的前沿基地控制系统，逐步理解一个荒诞、危险、资源紧张的星际边境。
