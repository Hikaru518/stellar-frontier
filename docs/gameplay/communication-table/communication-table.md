# 通讯台 Gameplay

## 1. 概述（What & Why）

通讯台 gameplay 是玩家与队员、地图、行动和事件系统之间的主要指令入口。玩家通过主动通话给队员下达调查、移动、原地待命和地块对象交互等基础指令；队员在行动、抵达、调查完成或遭遇事件后，也会以事件来电的形式把世界变化反馈给玩家。

这个系统把地图探索、队员行动、地块对象和事件触发串成可验证的网页闭环：玩家打电话，了解队员处境，下达一个清楚行动，在地图与状态中看到行动发生，再通过日志或来电获得行动结果。

## 2. 设计意图（Design Intent）

通讯台要让玩家感觉自己是在远程指挥一支真实行动中的探索队，而不是在菜单里切换角色状态。日常体验的重点是“联系队员 -> 确认现场 -> 下达行动 -> 等待行动推进 -> 接收结果反馈”。

系统优先制造三种体验：玩家能用通话稳定推进探索；地图上的未知对象会因为调查逐步变成可互动内容；事件来电会把世界变化重新拉回通讯台。系统避免把地图做成直接操作面板，也避免为不同队员维护各自硬编码的行动按钮。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **普通通话**：玩家主动联系队员的通话入口，用于查看队员当前处境并下达基础行动指令。通讯台按钮显示为“通话”。
- **事件来电**：由事件系统产生的队员主动联系玩家的入口。普通事件和紧急事件都可以显示为“接通”，但通过标签、阻塞状态和倒计时区分严重度。
- **基础行动**：所有队员都可用的通用行动，包括调查当前区域、移动到指定区域、原地待命、停止当前行动，以及由已揭示地块对象提供的对象行动。
- **地块对象**：地图地块上的可发现内容。对象在未调查前可能隐藏；被发现或调查揭示后，才会为普通通话提供额外动态按钮。
- **对象行动**：由地块对象的 `candidateActions` 声明并由通话页渲染的行动。当前内容支持调查、采集、建设、回收和扫描。
- **简单携带物**：对象交互获得的轻量物品或资源记录，当前复用队员 `inventory`；系统不展示重量、容量、丢弃或交付回基地规则。
- **原地待命**：立即停止当前行动并进入待命；如果队员已经待命，则记录一次 `idle_time` 事实信号。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家从通讯台选择一名队员，点击“通话”主动联系，或在来电时点击“接通”。
2. 通话页显示队员当前位置、当前行动状态、地块已知信息，以及可用行动。
3. 队员待命时，玩家可以选择调查当前区域、移动到指定区域、原地待命，或选择已揭示地块对象生成的动态按钮。
4. 队员忙碌时，通话页展示当前行动状态，并只提供停止当前行动或原地待命。
5. 玩家选择移动时，通话页进入地图选点流程；地图只记录候选目的地，最终派遣仍回到通话页确认。
6. 玩家选择调查或对象行动时，系统创建真实队员行动，通讯台、地图和队员状态展示行动进度。
7. 行动完成后，系统结算位置、调查披露、携带物或地块对象状态，并发出 `arrival`、`action_complete` 或 `idle_time` 等事实信号。
8. 事件系统根据事实信号判断是否生成事件来电；玩家从通讯台“接通”并处理普通或紧急事件。

### 4.2 典型情境

- **高光时刻**：玩家让 Garry 调查矿床并执行采集，采集完成后获得铁矿石，同时触发矿床异常来电，玩家能看到对象交互和事件反馈连接在一起。
- **低谷 / 摩擦点**：队员正在移动或调查时，玩家不能直接追加新任务，只能查看当前行动并选择停止或原地待命，避免行动并行造成状态混乱。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

- 队员同一时间只能执行一个主行动；移动、调查、采集、建设、回收、扫描和事件处理互斥。
- 普通通话按队员是否存在 `activeAction` 决定按钮：没有行动时显示完整基础行动和对象行动；已有行动时只显示 `availableWhenBusy = true` 的通用行动。
- 当前通用行动包括 `survey`、`move`、`standby` 和 `stop`；其中 `standby` 和 `stop` 可以在忙碌时使用。
- 普通事件来电不默认阻塞队员行动；紧急或 blocking 事件可以占用队员行动或通讯，并要求玩家在倒计时或有效期内处理。

### 5.2 规则与公式

- 普通通话的基础行动由 `content/call-actions/*.json` 定义，并由 `src/callActions.ts` 生成按钮视图；不按角色硬编码。
- “调查当前区域”创建 `survey` 行动。完成后揭示当前地块中 `visibility = onInvestigated` 的对象，写入地图运行时状态，并发出 `action_complete`。
- “移动到指定区域”进入地图选点；地图只读，最终移动必须回到通话流程确认。
- “原地待命”立即清除当前行动，写入日志，并发出 `idle_time`。
- “停止当前行动”立即清除当前行动并进入待命；当前配置保留 `10` 秒时长字段，但处理器即时结算。
- 已揭示地块对象的 `candidateActions` 会映射成动态按钮。按钮是否出现还要满足行动定义的 `category = object_action` 与 `applicableObjectKinds`。
- 对象交互完成后，系统按 handler 结算。`gather` 会把产物加入队员 `inventory`；`build`、`extract` 和 `scan` 当前写入行动完成日志并发出 `action_complete`。
- 事件选项如果表达移动、调查、等待或处理目标，应桥接到真实队员行动、目标或地块状态变化，而不是只写日志。

### 5.3 参数与默认值

| 参数 | 当前规则 |
| --- | --- |
| 调查当前区域 | `survey`，`120 秒` |
| 调查地块对象 | object `survey`，`120 秒` |
| 采集地块对象 | `gather`，`180 秒`，默认按对象 `legacyResource` 或 action params 给队员 `inventory` 加产物 |
| 建设地块对象 | `build`，`300 秒` |
| 回收地块对象 | `extract`，`180 秒` |
| 扫描地块对象 | `scan`，`90 秒` |
| 原地待命 | `standby`，即时结算并发出 `idle_time` |
| 停止当前行动 | `stop`，当前处理器即时结算，内容配置保留 `10 秒` |
| 地图移动 | 按地图系统路径和地形耗时计算，通话页只负责确认目的地 |

## 6. 系统交互（System Interactions）

- **依赖于**：通讯台与通话页负责入口和行动确认；地图系统提供队员位置、地块已知信息、对象可见性和移动目标；队员行动系统负责移动、调查、对象交互、停止和待命；事件系统负责监听行动事实并生成来电。
- **被依赖于**：事件内容依赖此系统把选项落成真实行动；地图探索依赖此系统推动地块调查和对象交互；队员详情和通讯台列表依赖此系统展示当前行动、简单携带物和来电状态。
- **共享对象 / 状态**：`GameState` 中的队员位置、队员 `activeAction`、地图 runtime state、地块对象可见性、runtime call、队员 `inventory`。
- **事件 / 信号**：移动抵达发出 `arrival`；调查、采集、建设、回收、扫描等完成发出 `action_complete`；原地待命发出 `idle_time`；玩家在事件通话中选择选项产生 `call_choice`。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：主动调查地块**：队员待命在某地块，玩家主动通话并选择“调查当前区域”；队员进入调查行动，完成后揭示隐藏对象，地图详情和通话选项发生变化。
- **S2：通话派遣移动**：队员待命，玩家通话中选择“移动到指定区域”；地图进入选点模式，玩家返回通话页确认目的地，队员按路线移动，并在抵达后触发 `arrival`。
- **S3：对象交互**：地块对象已揭示且声明 `candidateActions`；通话页显示“采集 / 扫描 / 回收”等动态按钮；队员执行对象行动，完成后获得携带物或触发事件检测。
- **S4：事件来电**：行动完成或抵达触发事件；通讯台显示“接通”入口和普通 / 紧急标签；玩家进入通话处理事件选项。
- **S5：紧急事件处理**：队员进入 blocking 事件并出现倒计时；玩家接通并选择应对方案；事件结果解除阻塞、创建后续行动或造成后果。

### 7.2 边界 / 失败场景

- **F1：队员忙碌中**：队员正在移动、调查或采集时，普通通话只展示当前行动和停止 / 原地待命，避免直接叠加新行动。
- **F2：地块对象未揭示**：地块存在隐藏对象但未调查时，普通通话不显示对象交互按钮。
- **F3：普通事件未处理**：普通事件来电出现后，队员行动不默认被阻塞；玩家可稍后接通，但事件不应丢失。
- **F4：紧急事件超时**：紧急来电在有效期内未处理时，系统按事件定义结算失败或自动后果，并将结果反馈到队员、地图或日志。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- **取舍 1**：基础调查、移动、待命和对象交互由通话 / App 层生成真实行动，而不是把所有普通动作都事件图化。事件系统消费行动结果信号并负责反馈。
- **取舍 2**：已揭示对象直接生成动态按钮，而不是先做对象子菜单。当前地块对象数量有限，直接按钮更容易让玩家看见“调查后解锁交互”的变化。
- **取舍 3**：简单携带物复用队员 `inventory`，不扩展完整背包、负重或交付链。对象交互需要有获得感，但资源运输不是通讯台玩法的职责。
- **取舍 4**：普通和紧急事件都可以表现为“接通”。普通 / 紧急的差异由标签、blocking 状态和倒计时承担。
- **要避免的反模式**：为具体队员或具体地块写死按钮；地图上直接下达指令；事件选项只追加日志但不改变队员行动、地块状态或可见信息；忙碌队员自动叠加新行动。

## 10. Open Questions

- **Q1：普通事件来电如果玩家长期不接通，是否需要过期、自动归档或一直保留？** active call 的保留、过期和 UI 清理策略需要统一规则。
- **Q2：简单携带物应展示在哪些页面？** 当前复用队员背包数据，仍需决定通讯台队员卡、队员详情、通话页中的展示范围。
- **Q3：事件选项桥接真实行动时，如果队员已经有行动，是否允许事件强制替换，还是必须先让玩家确认停止当前行动？**
