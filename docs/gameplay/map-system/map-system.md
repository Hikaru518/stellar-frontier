# 可配置地图系统

## 1. 概述（What & Why）

地图系统是玩家通过雷达 / 地图理解星球现场态势的基础系统。它由可配置尺寸的网格地图构成，默认地图为 `8 x 8`，每个地图块拥有区域名、地形、天气、特殊状态、环境属性与地块对象等层级信息。

地图保持只读态势图定位。当前实现中，PC 地图打开后直接展示完整配置地图（默认 `8 x 8`），玩家可以查看队员位置、视觉地块、地形、天气、地块对象与特殊状态；移动、待命、停止、调查四类基础行动仍通过“通讯台 → 通话”发出，剧情动作由结构化事件选项提供。

地图系统的核心目标是把“地图块是什么”定义清楚：区域名用于人物位置与叙事表达，地形用于基础环境识别，天气、特殊状态和环境属性为事件与调查扩展提供语义，地块对象承载矿床、生物巢穴、废弃医疗舱、玩家设施等可发现或可作业内容。

## 2. 设计意图（Design Intent）

地图系统要提供“远程掌握现场”的感觉。当前原型优先让玩家完整理解 `8 x 8` 地图和美术铺设效果，暂不使用战争迷雾或探索窗口限制；后续如果需要“逐步揭示地图”的体验，将作为独立视觉与玩法设计重做。区域名让队员位置表达更自然，例如“Garry 位于灰熊丘陵，正在调查异常痕迹”；地图详情再承担坐标、地形、天气和地块对象等精确信息，避免“木材（2,3）”这类把资源误显示为地点的混乱。

这个系统鼓励玩家产生判断感：完整地图提供空间结构与队员态势，隐藏的特殊状态、地块对象和环境属性仍为调查行动保留价值。当前地图可见性不负责制造探索边界；探索 / 战争迷雾留到后续单独设计。

地图系统要避免两种体验：一是地图信息层级混乱，玩家不知道“地形、资源、建筑、状态”分别意味着什么；二是地图变成直接下令界面，破坏“通讯台 → 通话”作为指令通道的核心体验。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **地图配置**：定义地图宽高、默认地图数据与地块集合的配置。默认地图为 `8 x 8`，但尺寸不写死在规则或 UI 中。
- **地图块**：地图的最小网格单位，拥有坐标，并承载区域名、地形、天气、特殊状态、环境属性与地块对象。
- **区域名**：一个或多个地图块从属的地理 / 叙事名称，如“坠毁区域”“南部平原”“米米尔高原”“灰熊丘陵”。队员位置摘要优先显示区域名。未被任何区域覆盖的地块显示为“野外”。
- **地形**：地图块的基础环境类型，如平原、荒野、森林、水域、丘陵等。地形用于表达底层环境，不用于承载矿床、遗迹或玩家设施。
- **天气**：地图块当前短期环境，如晴朗、阴天、雨天、雷雨天、离子风暴。当前地图页完整显示配置地图，天气作为基础地块信息显示；当前不参与事件结算。
- **特殊状态**：地图块正在发生的异常或事件态，如湖泊沸腾、火山爆发、高威胁生物出现。特殊状态需要调查或其他揭示手段才可见，可有持续时间；结束后回到未知 / 无已知特殊状态的表达。
- **环境属性**：地图块的可测量环境读数，如温度、湿度、磁场强度。地图数据具备这些属性，但地图页不常驻显示；调查时可在队员回报日志中记录当时数值。
- **地块对象**：地图块上的可发现、可调查、可作业或可作为行动上下文的对象总称，例如矿床、生物巢穴、废弃医疗舱、玩家建造设施。部分地块对象发现即显示，部分需要调查揭示。
- **完整地图窗口**：当前 PC 地图与通话选点使用的显示范围。它按地图配置尺寸渲染全部合法地块，默认 `8 x 8`。
- **发现状态**：地图运行时仍保存 `discovered` / `initialDiscoveredTileIds` 等字段，用于内容初始化、编辑器 overlay 与未来战争迷雾设计；当前 PC 地图展示和移动选点不再用它裁剪显示范围。
- **视觉层**：地图内容中的 `visual.layers`，由 Map Editor 使用 tileset 资源铺设。PC Phaser runtime 按 visual layer 顺序渲染 sprite，视觉 tile edge-to-edge 拼接，格子线、选框、路线和队员标记都属于 overlay。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家从通讯台或队员摘要中看到队员所在区域名与当前行动，例如“Garry 位于灰熊丘陵，正在调查异常痕迹”。
2. 玩家打开雷达 / 地图，看到完整配置地图。默认地图为 `8 x 8`，页面初始处于全局缩放层级，便于一次性理解整体空间。
3. 玩家查看某个地图块，获得区域名、坐标、地形、天气，以及已揭示的地块对象或特殊状态。
4. 玩家根据完整地图与当前队员位置判断下一步目标，并通过通讯台 / 通话请求队员移动、待命、停止或调查。
5. 队员调查后，玩家可能在日志中获得该地块的环境属性读数，或揭示原本隐藏的特殊状态 / 地块对象。
6. 玩家回到地图，用新增信息重新判断下一步探索或派遣方向。

### 4.2 典型情境

- **高光时刻**：玩家在完整地图上选择某片区域，例如“灰熊丘陵，天气雷雨”，调查后在回报日志中看到异常磁场读数，并在地图上揭示出“矿床”或“高威胁生物活动”的地块对象 / 特殊状态。
- **低谷 / 摩擦点**：玩家看到队员位置时不应被资源名误导；摘要层显示区域名，地图详情层显示坐标、地形、天气和对象，信息职责清晰分开。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

- **地图尺寸**：地图宽高由地图配置决定；默认配置为 `8 x 8`。所有坐标合法性、地图渲染范围与移动目标校验都读取配置。
- **地块发现状态**：运行时仍保存发现状态；当前 PC 地图展示与移动选点暂时把全部合法地块视为可见。战争迷雾、探索边界、未探索视觉表现与发现状态如何影响行动，需要后续单独设计。
- **特殊状态可见性**：特殊状态需要调查或其他揭示手段后才显示。特殊状态可有持续时间；持续时间结束后，该状态不再显示，地块回到“状态未知 / 无已知特殊状态”的表达。
- **地块对象可见性**：每个地块对象拥有可见性规则：发现即显示，或调查后显示。
- **环境属性可见性**：环境属性存在于地图块数据中，但不在地图页常驻显示。调查时可把当时读数写入队员回报日志，供玩家查阅。

### 5.2 规则与公式

- **当前显示范围计算**：地图页与通话移动选点按配置尺寸渲染完整地图，默认 `8 x 8`。运行时代码中保留的发现窗口算法仅作为未来战争迷雾 / 探索窗口设计的候选基础，不属于当前玩家可见规则。
- **坐标表达**：地图内部可以用配置尺寸管理边界，但玩家可见坐标以坠毁地点 / 玩家开始地点为原点 `(0,0)`，向外扩展时可出现负数坐标，避免通过坐标范围提前暴露完整地图大小。
- **区域名显示优先级**：队员摘要与通讯台位置文案优先显示区域名，例如“位于灰熊丘陵”；地图详情中再显示坐标、地形、天气、地块对象等细节。
- **地形与对象边界**：地形是底层环境；矿床、生物巢穴、废弃医疗舱、旧医疗前哨、玩家设施等均属于地块对象，不作为地形。
- **通话行动边界**：地块对象可以作为结构化地点事件的条件、线索和上下文；对象被发现或调查揭示后，事件可提供专属剧情选项。地图仍只展示对象和状态，不直接下达指令。
- **移动与地图边界的关系**：移动目标必须是地图配置内的 authored tile。当前不再因 discovered / frontier 状态限制目标；移动路径仍按上下左右相邻推进，不允许斜向移动。

### 5.3 参数与默认值

| 参数 | 默认值 / 规则 |
| --- | --- |
| 默认地图尺寸 | `8 x 8` |
| 坐标体系 | 以坠毁地点 / 玩家开始地点为 `(0,0)`；坐标可为负数；坐标用于表达相对位置，不直接暴露真实地图边界 |
| 当前显示范围 | 完整配置地图；默认 `8 x 8` 全部显示 |
| 地块基础可见信息 | 区域名、地形、天气 |
| 需揭示信息 | 特殊状态、部分地块对象 |
| 暂不常驻显示 | 环境属性 |
| 当前不接入 | 天气 / 特殊状态 / 环境属性对事件的自动影响 |

## 6. 系统交互（System Interactions）

- **依赖于**：
  - 地图配置：提供地图尺寸、地图块集合、起始坠毁地点、区域名、地形、天气、环境属性、地块对象等数据。
  - 队员系统：提供队员当前位置和当前行动；队员位置摘要由地图区域名与行动状态共同生成。
  - 调查行动：作为揭示特殊状态、隐藏地块对象与记录环境属性读数的入口。

- **被依赖于**：
  - 地图页 / 雷达：根据地图配置渲染完整地图，并展示地图块分层信息、视觉层、队员位置、选中框与候选路线。
  - 通讯台与队员卡片：展示队员所在区域名与行动状态。
  - 通话页：可根据队员所在地形与地块对象生成候选行动；行动执行仍由通话确认。

- **共享对象 / 状态**：
  - 地图配置与地图块运行时状态。
  - 队员当前位置。
  - 地块发现状态。
  - 已揭示的特殊状态与地块对象。
  - 调查得到的环境属性回报日志。

- **事件 / 信号**：
  - 地图系统不直接结算事件。
  - 天气、特殊状态、环境属性、地块对象可作为事件触发条件、事件分支或通话行动条件的语义来源。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：查看队员位置摘要**：队员处于某地图块 → 通讯台 / 队员卡片读取该地块区域名与队员行动 → 玩家看到“Garry 位于灰熊丘陵，正在调查异常痕迹”，而不是“木材（2,3）”。
- **S2：打开雷达查看完整地图**：玩家打开地图 → 系统按地图配置渲染完整 `8 x 8` 地图 → 玩家先看到全局空间，再通过缩放、平移和点按查看局部详情。
- **S3：查看地块详情**：玩家选择一个地图块 → 地图详情显示区域名、相对坐标、地形、天气，以及已揭示的特殊状态 / 地块对象 → 玩家理解该格的基础环境和可疑内容。
- **S4：调查后获得隐藏信息**：队员完成调查 → 系统记录环境属性读数到回报日志，并可揭示隐藏地块对象或特殊状态 → 玩家通过日志与地图更新理解该地块。

### 7.2 边界 / 失败场景

- **F1：点按地图边缘**：玩家在最高缩放层级平移到地图边缘 → 摄像机边界允许查看完整左上角和其他边缘地块 → 玩家不会因为缩放而无法选中边缘格。
- **F2：特殊状态过期**：某状态如“火山喷发”持续时间结束 → 地图不再显示该状态，改为“无已知特殊状态 / 状态未知” → 玩家不会把过期情报误认为当前事实。
- **F3：对象未揭示**：地块存在隐藏对象但尚未调查 → 地图详情不显示该对象，或只显示“未知信号 / 未确认对象” → 玩家需要调查才能确认。
- **F4：位置文案缺失区域名**：若某地块没有配置区域名 → 使用“野外”等明确 fallback，而不是资源名或地块对象名 → 避免资源名冒充位置。
- **F5：地图误用为指令入口**：玩家在地图查看地块和路线 → 地图只辅助选点，不直接下达移动、待命、停止、调查或剧情动作。
- **F6：临时探索限制误伤行动**：玩家希望派遣队员到完整地图上的任意合法地块 → 系统只检查目标是否在地图配置内且有 authored tile → 不再因临时 discovered / frontier 状态拒绝。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- 选了“配置驱动地图”而非“固定 8x8”，因为默认 `8 x 8` 解决地图过小的问题，而配置驱动能避免未来再次把尺寸写死，并支持不同地图规模。
- 选了“当前完整显示地图”而非“继续沿用临时战争迷雾”，因为现阶段更需要验证地图内容、视觉铺设、通话选点和 Phaser 地图基础体验。探索 / 战争迷雾会在后续作为独立系统设计。
- 选了“区域名优先显示”而非“坐标 / 资源名优先显示”，因为队员位置摘要需要服务叙事与可读性；坐标、地形、地块对象属于雷达 / 地图详情层，不应挤进一行摘要。
- 选了“地块对象”而非“兴趣点 / POI”，因为地块对象能统一矿床、生物巢穴、废弃医疗舱和玩家设施，同时比兴趣点更适合作为规则层术语。
- 选了“地图只读、通话执行”而非“地图直接执行对象行动”，因为地块对象只提供事件条件和上下文，玩家仍需要通过通讯台 / 通话确认指令。
- **要避免的反模式**：
  - 把资源名、建筑名、地块对象名显示成队员所在地点。
  - 把旧医疗前哨、矿床、生物巢穴等对象塞进地形字段。
  - 在地图页直接下达行动，绕过通讯台 / 通话。
  - 在战争迷雾尚未重新设计前，用临时 discovered / frontier 状态限制地图查看或移动选点。
  - 把天气、特殊状态、环境属性直接接入事件结算但缺少清晰可见性规则。

## 9. 参考与灵感（References & Inspiration）

- **Tiled 官方文档 - Layers**：https://doc.mapeditor.org/en/stable/manual/layers/ 。借鉴点：地图内容应分层组织，网格层与对象层承担不同职责，避免单一字段承载所有信息。
- **Godot TileMap 文档**：https://docs.godotengine.org/en/stable/tutorials/2d/using_tilemaps.html 。借鉴点：视觉层、碰撞 / 导航 / 规则层应分离；本项目中地形显示不应等同于事件或资源判定。
- **Civilization VI 官方资料**：https://civilization.2k.com/civ-vi/ 。借鉴点：terrain、resources、features、improvements 等术语分层，有助于区分地形、资源与人造设施。
- **The Sims 4 - Concurrent Interactions in The Sims**：https://www.gdcvault.com/play/1020190/concurrent-interactions-in-the-sims 。借鉴点：未来通话行动可从当前对象、角色状态与场景上下文生成。
- **Microsoft Learn - Menu flyout and menu bar**：https://learn.microsoft.com/en-us/windows/apps/develop/ui/controls/menus 。借鉴点：未来对象级行动菜单应保持层级浅、常用动作可见，不可执行动作说明原因。
- **NN/g - Contextual Menus**：https://www.nngroup.com/articles/contextual-menus/ 。借鉴点：上下文菜单需要可发现性补偿；本项目地图页保持只读，行动执行仍应在通话中完成。

## 10. Open Questions

1. **地点事件动作的扩展元数据**：当前基础行动只包含移动、待命、停止和调查。未来如果需要对象级耗时、消耗、失败率或工具需求，需要扩展结构化事件或目标模型。
2. **战争迷雾 / 探索可见性重设计**：当前实现完整显示地图并允许选择任意合法 authored tile。后续若要恢复探索限制，需要同时设计视觉表现、信息隐藏层级、移动目标规则、队员回传、事件触发与 UI 文案。
