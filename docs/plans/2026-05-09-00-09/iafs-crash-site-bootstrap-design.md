# 1. 概述（What & Why）

本轮 IAFS 先建立一个最小但可扩展的开局骨架：把 `4-4` 设为坠毁地点，在其上放置三个损坏对象 `发电机`、`维生装置`、`穿梭机核心`，并用周围一圈山体把玩家初期活动限制在坠毁点附近。三个对象都通过通话页进入“维修”交互；维修成功后本轮只改变对象状态，不立即解锁新系统或新功能。

这个增量的目的不是一次性做完整 IAFS，而是先把“开局空间布局 + 对象状态 + 通话维修入口 + 可重复的敏捷检定维修事件”这条最核心的原型链跑通。后续无论要在发电机上接电力、在维生装置上接生存压力，还是在穿梭机核心上接逃生主线，都可以在这个骨架上继续叠加。

# 2. 设计意图（Design Intent）

这一轮的重点是验证 IAFS 的开局是否能成立，而不是一次性把整套逃生系统做完。玩家进入游戏后，首先面对的应该是一个被事故打断、被地形暂时封闭、但有明确修复目标的场景：坠毁点是当前唯一安全且可操作的工作区，三个损坏对象则构成最初的交互驱动力。这样玩家一开始就有清楚的问题可处理，而不是在空白地图上无目标游走。

维修玩法在这一轮应当体现“有门槛，但不惩罚过重”。因此本轮采用属性影响成功率、失败不产生额外状态变化、允许重复尝试的软失败设计。这里的“消耗时间”仅指这次维修尝试本身就需要的行动时间，而不是失败后再追加额外罚时。玩家会直观理解“敏捷更高的人修得更稳”，但不会因为一次失败立刻遭受伤害、资源损失或永久锁死流程。这样既能验证属性参与行动结算是否有价值，也能避免首轮原型因为惩罚过重而掩盖核心体验。

在实现边界上，这一轮不追求把概率公式抽象成完整通用系统。当前事件系统还不能直接用 content 配置出“属性进入公式后得到动态成功率”的通用链路，因此本轮更合理的策略是：先为维修事件补一个最小专用实现，让它支持约定好的公式和结果结算；等后续确认 IAFS 之外也大量需要同类能力时，再决定是否把它抽象回正式事件引擎能力。这样可以让 IAFS 主线继续推进，同时避免过早为潜在需求付出系统复杂度。

# 3. 核心概念与术语（Core Concepts & Terminology）

- **坠毁地点**：位于 `4-4` 的开局核心地块，是玩家当前唯一明确的操作中心，也是三个首批维修对象的共同承载地点。
- **山体封锁圈**：围绕坠毁地点周边布置的一圈阻挡地形，用于限制玩家在最初阶段离开坠毁区域，形成清晰的开局边界。
- **损坏对象**：本轮放置在坠毁地点上的三个可交互对象：`发电机`、`维生装置`、`穿梭机核心`。它们初始都处于 `damaged` 状态。
- **对象状态**：对象当前可供系统与内容读取的离散状态。第一版至少包含 `damaged` 与 `repaired` 两种状态；本轮成功维修后只改变状态，不立刻解锁附加功能。
- **维修动作**：玩家在通话中针对某个损坏对象发起的专属交互入口。它不是基础行动，而是对象 action，会进入对应的维修事件或维修结算链。
- **维修事件**：由维修动作触发的事件流程。它负责消耗时间、执行成功率判定，并根据结果决定“对象维持损坏”还是“对象变为已修复”。
- **敏捷检定公式**：本轮维修使用的成功率规则。设计目标是以队员敏捷为主要输入，并允许每个维修事件单独配置数值参数；第一版按最小专用实现落地，不要求先抽象成通用公式系统。
- **软失败**：维修失败时不产生伤害、额外损坏、资源损失或新的状态变化；系统只结算这次维修尝试原本就会消耗的行动时间，对象维持原状态，玩家之后可以再次尝试。
- **对象锁定**：当某个队员正在维修某对象时，该对象暂时被占用；其他队员不能同时对同一对象发起维修。
- **最小验证范围**：本轮只验证“空间封锁、损坏对象、通话维修入口、时间消耗、成功率判定、对象状态变化”这条闭环，不扩展修复后的功能收益。

# 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家开局位于坠毁地点 `4-4`，周围一圈山体明确阻止当前阶段直接外出。
2. 玩家通过通讯台联系当前可行动队员，查看坠毁地点上的可交互对象。
3. 玩家在通话中针对 `发电机`、`维生装置` 或 `穿梭机核心` 选择“维修”。
4. 系统检查该对象是否已被其他队员占用；若已占用，则当前队员不能开始本次维修。
5. 若对象未占用，队员进入该对象对应的维修事件 / 维修结算流程，并消耗这次维修尝试固有的行动时间。
6. 维修结算时，系统按该对象配置读取当前队员的敏捷和维修参数，计算本次成功率并判定成功或失败。
7. 若成功，对象状态从 `damaged` 变为 `repaired`；若失败，对象状态不变，也不产生额外惩罚。
8. 玩家根据维修结果决定是否继续处理其他损坏对象，或对同一对象再次尝试。

### 4.2 典型体验

- **高光时刻**：玩家明确看到坠毁点上存在多个可修复设施，并能快速理解“先联系队员 -> 选择对象 -> 发起维修 -> 看结果”这一条主循环。
- **节奏时刻**：维修失败不会打断流程，但会自然占用行动时间，使“谁去修、先修哪个、是否重试”成为有代价的调度判断。
- **边界时刻**：当某对象已被一个队员占用时，玩家会被清楚告知当前不能由第二个队员同时维修，从而维持对象级调度的清晰性。

# 5. 机制与规则（Mechanics & Rules）

### 5.1 地图与空间规则

- 坠毁地点固定放在 `4-4`。
- `4-4` 地块至少承载三个对象：`发电机`、`维生装置`、`穿梭机核心`。
- 坠毁地点周围一圈相邻地块布置为山体阻挡区，用于阻止玩家在本轮最小验证阶段直接离开坠毁区域。
- 本轮山体的主要职责是形成明确边界，不承担额外事件、资源或调查收益。

### 5.2 对象状态规则

- 三个对象初始状态统一为 `damaged`。
- 本轮每个对象至少支持两种状态：
  - `damaged`
  - `repaired`
- 维修成功后，对象状态切换为 `repaired`。
- 本轮对象变为 `repaired` 后，不立即解锁额外功能、资源、出口或后续系统效果。
- 已经是 `repaired` 的对象，不应再显示“维修”作为主要可执行动作。

### 5.3 维修入口规则

- “维修”不是基础通用行动，而是坠毁点对象的专属 action。
- 玩家必须先进入通讯台并发起通话，才能对对象选择“维修”。
- 玩家选择维修时，目标必须明确指向某一个对象，而不是对整个坠毁点发起泛化维修。
- 每个对象可拥有独立的维修事件或独立的维修参数配置，但共享同一套结算思路。

### 5.4 对象锁定规则

- 同一对象在同一时间只允许一个队员进行维修。
- 如果对象已被某队员占用，其他队员不能同时对该对象发起维修。
- 被占用对象应向玩家显示为“不可维修 / 已被占用”的清晰状态，而不是静默消失。
- 对象锁定只针对“同一对象”，不阻止不同队员同时维修不同对象。

### 5.5 成功率与结算规则

- 本轮维修检定以队员 **敏捷** 为输入属性。
- 设计目标公式为：

```text
successRate = clamp(min, max, base + agility * ratio + bias - difficulty)
```

- 第一版默认参数建议为：
  - `base = 0`
  - `ratio = 20%`
  - `bias = 0`
  - `difficulty = 0`
  - `min = 0%`
  - `max = 100%`
- 在这组默认参数下，敏捷 `5` 可达到 `100%` 成功率。
- 每个维修对象后续可以单独配置自己的 `base / ratio / bias / difficulty / min / max`，但本轮最小验证可以先让三个对象共用同一组默认参数。
- 当前事件系统不能只靠 content 直接完成这套动态概率判定，因此本轮允许为维修结算补一个最小专用实现；但这套实现只服务当前维修链路，不扩展为通用表达式系统。

### 5.6 失败与重试规则

- 维修失败时：
  - 不造成伤害
  - 不造成额外损坏
  - 不消耗额外资源
  - 不追加新的负面状态
- 失败只结算本次维修尝试原本就会消耗的行动时间。
- 失败后对象保持 `damaged`。
- 玩家之后可以再次对同一对象发起维修尝试。

### 5.7 时间规则

- 每次维修尝试都应有固定行动时间成本。
- 这个时间成本无论成功或失败都会正常消耗。
- 本轮可以先让三个对象共用同一维修时长，后续再细分为不同对象的不同维修耗时。
- 本轮不引入“失败额外罚时”“连续失败累积惩罚”或“维修次数上限”。

# 6. 系统交互（System Interactions）

- **依赖于地图系统**：地图需要在 `4-4` 放置坠毁地点对象，并在周围相邻地块布置山体阻挡，形成当前阶段的活动边界。
- **依赖于对象系统**：`发电机`、`维生装置`、`穿梭机核心` 作为地块对象存在，并具备可读取的对象状态与对象 action 入口。
- **依赖于通讯台 / 通话系统**：玩家不能在地图直接下达维修，而是必须通过通话页对具体对象选择“维修”。
- **依赖于行动系统**：维修尝试需要占用队员的一次行动时间，并在成功或失败后正常结算行动结束。
- **依赖于事件 / 结算链**：维修动作进入维修事件或维修专用结算流程，由该流程负责对象锁定、成功率计算与结果写回。
- **依赖于角色属性数据**：维修成功率读取执行该次维修的队员敏捷值，后续若要扩展到其他属性，仍沿用同一角色属性来源。

- **写回到对象状态**：维修成功后，系统把对象从 `damaged` 改为 `repaired`；失败则保持原状态。
- **写回到行动与时间**：无论成功或失败，都只结算这次维修尝试本来就需要的行动时间，不额外追加罚时。
- **写回到通话可见性**：对象被占用时，其他队员在通话里应看到“已占用 / 当前不可维修”的状态；对象修复后不再显示维修为主要动作。
- **不依赖额外技能系统**：本轮维修不要求额外维修技能、专长标签或道具门槛。
- **不写回额外功能收益**：修好发电机、维生装置或穿梭机核心后，本轮不立即改写资源、供电、生存状态、地图出口或后续功能，只改变对象状态。

- **当前实现边界**：动态概率公式并不是现成事件 content 能直接表达的通用能力，因此本轮允许在维修链路中加入最小专用结算实现，但不改写“地图只读、通话发指令”的整体系统边界。

# 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：开局识别可修复目标**：玩家进入坠毁点后，通过通讯台联系队员，看到 `发电机`、`维生装置`、`穿梭机核心` 三个处于 `damaged` 状态的对象，并理解当前主目标是逐个尝试维修。
- **S2：单次维修成功**：玩家在通话中选择某个对象的“维修”，队员消耗一次维修尝试固有的行动时间，系统按敏捷公式结算并成功，将该对象状态改为 `repaired`。
- **S3：单次维修失败但可继续**：玩家选择维修后未通过成功率判定，系统只结算本次维修尝试应消耗的时间，不产生额外惩罚；对象继续保持 `damaged`，玩家稍后可再次尝试。
- **S4：并行处理不同对象**：如果存在多个可行动队员，玩家可以让不同队员分别处理不同损坏对象，从而同时推进多个修复目标。

### 7.2 边界 / 失败场景

- **F1：对象占用冲突**：A 队员已经在维修 `发电机` 时，玩家尝试让 B 队员也维修 `发电机`，系统应明确拒绝，并提示该对象当前已被占用。
- **F2：已修复对象重复操作**：某对象已进入 `repaired` 状态后，玩家不应再看到它作为主要“维修”目标出现，避免重复刷取无意义结果。
- **F3：封锁边界误解**：玩家尝试在本轮直接离开坠毁区域时，地图与行动层都应表现出山体阻挡的边界含义，而不是让玩家误以为这是 bug 或漏做路径。
- **F4：连续失败体验**：即使玩家连续数次维修失败，系统也不应引入额外损坏、额外罚时或隐藏惩罚；失败的唯一结果仍然是对象状态不变且本次行动时间已被消耗。

# 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- **取舍 1：先做坠毁点封闭开局，不先做外部探索内容**。这样能先验证 IAFS 的起始驱动力是否成立，把玩家注意力集中到“看见问题并尝试修复”上，而不是同时分散到外部地图、资源搜集和远距离调度。
- **取舍 2：先只做对象状态变化，不先做修复后的功能收益**。这样可以把本轮实现复杂度压在可控范围内，先确认对象、事件、时间和成功率链路跑通，再决定每个设施修好后真正带来什么系统效果。
- **取舍 3：采用软失败，不引入额外惩罚**。第一版更重视“属性参与维修是否有手感”而不是“失败是否足够残酷”。如果首轮就叠加伤害、资源损耗或连锁故障，会让玩家更难判断问题出在概率设计还是惩罚设计。
- **取舍 4：允许维修链路做最小专用实现，不强行抽象成通用概率系统**。当前事件系统没有现成的 content 级动态概率公式支持，因此本轮优先服务 IAFS 主线推进，避免为了未来可能复用的需求提前引入更大规模系统重构。
- **取舍 5：对象锁定只限制同一对象，不限制不同对象并行维修**。这样既保留了调度价值，也避免多人同时堆在同一对象上导致规则解释复杂化。

- **要避免的反模式**：
  - 把“维修”做成基础通用动作，而不是坠毁点对象的专属交互。
  - 因为要支持概率公式，就提前把事件系统扩展成任意表达式语言或通用脚本系统。
  - 失败后偷偷追加隐藏惩罚，让“软失败”名义和实际体验不一致。
  - 在对象修好后立刻顺手加上供电、生存、出口解锁等额外功能，导致本轮最小验证失焦。
- 用静默不可点击替代“已被占用 / 已修复”的显式反馈，增加玩家理解成本。

# 10. Open Questions

1. **三个对象的默认维修时长是否完全一致？**
当前策划允许先共用同一维修时长，但尚未确认发电机、维生装置、穿梭机核心是否应该在首版就体现不同复杂度。

2. **山体封锁圈在后续如何解除？**
本轮只把它作为开局边界，不定义何时、因何、以何种方式开放外出路线。

3. **维修成功后下一步功能接什么？**
本轮只改变对象状态，后续仍需分别决定发电机、维生装置、穿梭机核心修复后各自解锁的系统收益或主线推进作用。

4. **维修结果是否需要额外日志 / 通话反馈层次？**
当前只要求有成功 / 失败结算和对象状态变化，尚未决定是否要为不同对象提供更强的角色化台词、阶段化反馈或可视化状态提示。

5. **最小专用概率实现未来是否抽象回正式事件引擎能力？**
本轮刻意不把维修概率公式扩展成通用系统；后续若其他玩法也大量需要同类检定，再决定是否统一抽象。
