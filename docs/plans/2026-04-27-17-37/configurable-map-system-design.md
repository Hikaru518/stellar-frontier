# 可配置地图系统

## 1. 概述（What & Why）

地图系统是玩家通过雷达/地图理解星球现场态势的基础系统。它由可配置尺寸的网格地图构成，默认地图为 `8 x 8`，每个地图块拥有区域名、地形、天气、特殊状态、环境属性与地块对象等层级信息。地图本身保持只读态势图定位：玩家通过它查看已发现区域、外围未探索区域、队员位置与地块信息，但具体指令仍通过“通讯台 → 通话”发出。

本系统的核心目标是把“地图块是什么”定义清楚：区域名用于人物位置与叙事表达，地形用于基础环境识别，天气/特殊状态/环境属性用于未来事件与调查扩展，地块对象用于承载矿床、生物巢穴、废弃医疗舱、玩家设施等可发现或可作业内容。本轮重点是重构地图本体的数据与显示语义，涉及事件结算和通话对象行动菜单的部分只记录为后续 TODO。

## 2. 设计意图（Design Intent）

地图系统要提供“远程掌握现场”的感觉：玩家不是俯视全知的上帝，而是通过雷达、队员汇报和逐步揭示的信息拼出星球环境。区域名让队员位置表达更自然，例如“Garry 位于灰熊丘陵，正在执行采矿作业”；地图详情再承担坐标、地形、地块对象等精确信息，避免“木材（2,3）”这类把资源误显示为地点的混乱。

这个系统应鼓励玩家产生探索欲和判断感：已发现地块提供足够行动依据，外围一圈未探索区域制造下一步探索目标，隐藏的特殊状态、地块对象和环境属性则为调查行动保留价值。它要避免两种体验：一是地图信息层级混乱，玩家不知道“地形、资源、建筑、状态”分别意味着什么；二是地图变成直接下令界面，破坏“通讯台 → 通话”作为指令通道的核心体验。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **地图配置**：定义地图宽高、默认地图数据与地块集合的配置；本系统默认使用 `8 x 8`，但尺寸不写死在规则或 UI 中。
- **地图块**：地图的最小网格单位，拥有坐标，并承载区域名、地形、天气、特殊状态、环境属性与地块对象。
- **区域名**：一个或多个地图块从属的地理/叙事名称，如“坠毁区域”“南部平原”“米米尔高原”“灰熊丘陵”。队员位置摘要优先显示区域名。
- **地形**：地图块的基础环境类型，如平原、荒野、森林、水域、丘陵等。地形用于表达底层环境，不用于承载矿床、遗迹或玩家设施。
- **天气**：地图块当前短期环境，如晴朗、阴天、雨天、雷雨天、离子风暴。天气一旦地块被发现即可显示；本轮不接入事件结算。
- **特殊状态**：地图块正在发生的异常或事件态，如湖泊沸腾、火山爆发、高威胁生物出现。特殊状态需要调查或其他揭示手段才可见，可有持续时间；结束后回到未知。
- **环境属性**：地图块的可测量环境读数，如温度、湿度、磁场强度。地图数据需要具备这些属性，但地图页暂不常驻显示；调查时可在队员回报日志中记录当时数值。
- **地块对象**：地图块上的可发现、可调查、可作业或可作为行动上下文的对象总称，例如矿床、生物巢穴、废弃医疗舱、玩家建造设施。部分地块对象发现即显示，部分需要调查揭示。
- **发现区域**：玩家已掌握基础信息的地图块集合。发现后可见区域名、地形与天气。
- **外围未探索区域**：与发现区域相邻一圈、尚未发现的地图块。地图显示范围由发现区域与外围未探索区域共同形成的外接长方形决定。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家从通讯台或队员摘要中看到队员所在区域名与当前行动，例如“Garry 位于灰熊丘陵，正在执行采矿作业”。
2. 玩家打开雷达/地图，看到已发现区域与外围一圈未探索区域构成的外接长方形，而不是完整暴露整张 `8 x 8` 地图。
3. 玩家查看某个已发现地图块，获得区域名、坐标、地形、天气，以及已揭示的地块对象或特殊状态。
4. 玩家对未探索外围产生下一步探索目标，并通过通讯台/通话请求队员移动、调查或执行当前已支持的行动。
5. 队员调查后，玩家可能在日志中获得该地块的环境属性读数，或揭示原本隐藏的特殊状态 / 地块对象。
6. 玩家回到地图，用新增信息重新判断下一步探索或派遣方向。

### 4.2 典型情境

- **高光时刻**：玩家原本只知道某片区域是“灰熊丘陵，天气雷雨”，调查后在回报日志中看到异常磁场读数，并在地图上揭示出“矿床”或“高威胁生物活动”的地块对象/特殊状态。
- **低谷 / 摩擦点**：玩家看到队员位置时不应再被“木材（2,3）”这类资源名误导；摘要层显示区域名，地图详情层显示坐标、地形和对象，信息职责清晰分开。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

- **地图尺寸**：地图宽高由地图配置决定；默认配置为 `8 x 8`。所有坐标合法性、地图渲染范围与移动目标校验都读取配置，不再假设固定 `4 x 4`。
- **地块发现状态**：地图块至少分为“未可见 / 外围未探索 / 已发现”三类显示状态。外围未探索地块只用于提示探索边界；已发现地块可显示区域名、地形和天气。
- **特殊状态可见性**：特殊状态需要调查或其他揭示手段后才显示。特殊状态可有持续时间；持续时间结束后，该状态不再显示，地块回到“该状态未知 / 无已知特殊状态”的表达。
- **地块对象可见性**：每个地块对象拥有可见性规则：发现即显示，或调查后显示。本轮只定义与直线展示这些对象，不改事件系统与对象行动结算。
- **环境属性可见性**：环境属性存在于地图块数据中，但不在地图页常驻显示。调查时可把当时读数写入队员回报日志，供玩家查阅。

### 5.2 规则与公式

- **可见范围计算**：先取所有已发现地块；再加入每个已发现地块周围一圈合法地图块作为外围未探索区域；最后对两者求外接长方形。地图页只渲染该外接长方形内的格子。
- **坐标表达**：地图内部可以用配置尺寸管理边界，但玩家可见坐标以坠毁地点 / 玩家开始地点为原点 `(0,0)`，向外扩展时可出现负数坐标，避免通过坐标范围提前暴露完整地图大小。
- **区域名显示优先级**：队员摘要与通讯台位置文案优先显示区域名，例如“位于灰熊丘陵”；地图详情中再显示坐标、地形、天气、地块对象等细节。
- **地形与对象边界**：地形是底层环境；矿床、生物巢穴、废弃医疗舱、旧医疗前哨、玩家设施等均属于地块对象，不作为地形。
- **通话行动边界**：地块对象可以在设计上声明未来可提供的候选行动，如采矿、勘测、调查、修复，但本轮不改通话菜单、不新增对象级二级菜单。

### 5.3 参数与默认值

| 参数 | 默认值 / 规则 |
| --- | --- |
| 默认地图尺寸 | `8 x 8` |
| 坐标体系 | 以坠毁地点 / 玩家开始地点为 `(0,0)`；坐标可为负数；坐标用于表达相对位置，不直接暴露真实地图边界 |
| 可见扩张 | 已发现地块外围一圈，按上下左右与斜向相邻均计入“外围一圈” |
| 地块基础可见信息 | 区域名、地形、天气 |
| 需揭示信息 | 特殊状态、部分地块对象 |
| 暂不常驻显示 | 环境属性 |
| 本轮不接入 | 天气/状态/环境属性对事件的影响、对象级通话行动菜单 |

## 6. 系统交互（System Interactions）

- **依赖于**：
  - 地图配置：提供地图尺寸、地图块集合、起始坠毁地点、区域名、地形、天气、环境属性、地块对象等数据。
  - 队员系统：读取队员当前位置，并把位置摘要从“资源/对象名 + 坐标”改为优先显示区域名与当前行动。
  - 调查行动：作为揭示特殊状态、隐藏地块对象与记录环境属性读数的入口；本轮可先保持既有调查流程，只明确数据承载与日志表达方向。

- **被依赖于**：
  - 地图页 / 雷达：根据发现状态渲染“已发现区域 + 外围一圈未探索区域”的外接长方形，并展示地图块分层信息。
  - 通讯台与队员卡片：展示队员所在区域名与行动状态。
  - 通话页：未来可根据队员所在地形与地块对象生成候选行动；本轮只记录设计，不改通话菜单。

- **共享对象 / 状态**：
  - 地图配置与地图块运行时状态。
  - 队员当前位置。
  - 地块发现状态。
  - 已揭示的特殊状态与地块对象。
  - 调查得到的环境属性回报日志。

- **事件 / 信号**：
  - 本轮不改变事件系统结算。
  - 天气、特殊状态、环境属性、地块对象未来可作为事件触发条件、事件分支或通话行动条件；相关接入项进入 TODO / Later。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：查看队员位置摘要**：队员处于某地图块 → 通讯台 / 队员卡片读取该地块区域名与队员行动 → 玩家看到“Garry 位于灰熊丘陵，正在执行采矿作业”，而不是“木材（2,3）”。
- **S2：打开雷达查看探索边界**：玩家打开地图 → 系统根据已发现地块与外围一圈未探索地块计算外接长方形 → 玩家只看到当前探索范围，而不是整张 `8 x 8` 地图。
- **S3：查看已发现地块详情**：玩家选择一个已发现地块 → 地图详情显示区域名、相对坐标、地形、天气，以及已揭示的特殊状态 / 地块对象 → 玩家理解该格的基础环境和可疑内容。
- **S4：调查后获得隐藏信息**：队员完成调查 → 系统记录环境属性读数到回报日志，并可揭示隐藏地块对象或特殊状态 → 玩家通过日志与地图更新理解该地块。

### 7.2 边界 / 失败场景

- **F1：已发现区域在地图边缘**：可见范围扩张到配置地图边界 → 系统只加入合法地图块，不显示配置外坐标 → 玩家不会看到不存在的格子。
- **F2：特殊状态过期**：某状态如“火山喷发”持续时间结束 → 地图不再显示该状态，改为“无已知特殊状态 / 状态未知” → 玩家不会把过期情报误认为当前事实。
- **F3：对象未揭示**：地块存在隐藏对象但尚未调查 → 地图详情不显示该对象，或只显示“未知信号/未确认对象”（如后续需要） → 玩家需要调查才能确认。
- **F4：位置文案缺失区域名**：若某地块没有配置区域名 → 使用明确 fallback（例如“未知区域”）而不是资源名或地块对象名 → 避免再次出现资源名冒充位置。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- **取舍 1：选了“配置驱动地图”而非“固定 8x8”**，理由：默认 `8 x 8` 解决当前地图过小的问题，但配置驱动能避免未来再次把尺寸写死，并支持不同地图规模。
- **取舍 2：选了“区域名优先显示”而非“坐标/资源名优先显示”**，理由：队员位置摘要需要服务叙事与可读性；坐标、地形、地块对象属于雷达/地图详情层，不应挤进一行摘要。
- **取舍 3：选了“地块对象”而非“兴趣点 / POI”**，理由：地块对象能统一矿床、生物巢穴、废弃医疗舱和玩家设施，同时比兴趣点更适合作为规则层术语。
- **取舍 4：选了“本轮只做地图本体重构”而非“同时改事件与通话菜单”**，理由：天气、特殊状态、环境属性和地块对象都会影响事件与行动，但这些联动应在地图层级稳定后再接入，避免一次迭代牵动过多系统。
- **要避免的反模式**：
  - 把资源名、建筑名、地块对象名显示成队员所在地点。
  - 把旧医疗前哨、矿床、生物巢穴等对象塞进地形字段。
  - 在地图页直接下达行动，绕过通讯台 / 通话。
  - 用完整坐标范围暴露真实地图大小。
  - 把天气、特殊状态、环境属性直接接入事件结算但缺少清晰可见性规则。

## 10. 本轮范围与阶段拆分（Scope & Phasing for This Round）

### 10.1 MVP（本轮必做）

- 将地图尺寸从固定 `4 x 4` 改为配置驱动，默认配置为 `8 x 8`。
- 将玩家可见坐标改为以坠毁地点 / 玩家开始地点为 `(0,0)` 的相对坐标，允许负数坐标。
- 为地图块建立明确层级：区域名、地形、天气、特殊状态、环境属性、地块对象。
- 地图页只显示“已发现区域 + 外围一圈未探索区域”的外接长方形。
- 已发现地块显示区域名、地形、天气；特殊状态和部分地块对象按可见性规则显示。
- 地图块数据具备环境属性，但地图页不常驻显示；调查时可将当前读数写入回报日志。
- 队员位置摘要优先显示区域名与行动状态，修正资源名 / 对象名被当作地点名的问题。
- 保持地图页只读，不在地图页直接下达行动。

### 10.2 Later（未来再做，明确本轮不做）

- 天气、特殊状态、环境属性参与事件触发、概率修正或事件分支。
- 地块对象驱动通话中的对象级行动菜单，例如矿床 → 采矿 / 勘测 / 道具调查。
- 更完整的状态持续时间 UI、历史状态记录与情报过期提示。
- 更丰富的区域命名规则，例如大区域 + 重要单格地名。
- 地图编辑器、随机地图生成或程序化区域生成。
- 地块对象的复杂生命周期，例如建造、损坏、升级、枯竭、占领。

### 10.3 不做（Out of Scope，避免范围膨胀）

- 不改变现有事件系统结算逻辑。
- 不新增复杂对象级二级菜单或道具条件菜单。
- 不把地图页改成直接下令界面。
- 不实现天气变化模拟、气候系统或真实时间天气。
- 不实现离线探索、离线事件或离线资源补算。
- 不做完整战争迷雾 / 视野遮挡 / 雷达扫描 minigame。

## 11. 本轮验收与风险（Acceptance & Risks）

### 11.1 Player Stories / Play Scenarios（验收切片）

#### PS-001: 配置驱动地图
- **作为**：玩家
- **我能**：在默认 `8 x 8` 的地图中探索，而不是被限制在固定 `4 x 4`
- **以便**：地图规模能支撑更长的探索节奏
- **验收标准**：
  - [ ] 地图尺寸从配置读取，默认地图为 `8 x 8`
  - [ ] 合法坐标、移动目标校验与地图渲染不依赖固定 `4 x 4`
  - [ ] 玩家可见坐标以坠毁地点为 `(0,0)`，允许负数
- **不包含**：随机地图生成
- **优先级**：P0

#### PS-002: 渐进式地图显示
- **作为**：玩家
- **我能**：只看到已发现区域和外围一圈未探索区域形成的外接长方形
- **以便**：保持探索未知感，同时知道下一步可探索边界
- **验收标准**：
  - [ ] 地图页不直接显示完整 `8 x 8`
  - [ ] 外围一圈未探索区域会围绕已发现地块出现
  - [ ] 配置边界外的格子不会显示
- **不包含**：复杂视野遮挡
- **优先级**：P0

#### PS-003: 清晰的队员位置文案
- **作为**：玩家
- **我能**：在队员摘要中看到区域名与行动状态
- **以便**：快速理解队员在哪里、正在做什么
- **验收标准**：
  - [ ] 队员位置摘要优先显示区域名
  - [ ] 地图详情显示坐标、地形、天气和地块对象
  - [ ] 不再出现“木材（2,3）”这类资源名作为地点名的文案
- **不包含**：复杂地名层级系统
- **优先级**：P0

#### PS-004: 地图块层级数据
- **作为**：设计者 / 后续实现者
- **我能**：为地图块配置区域名、地形、天气、特殊状态、环境属性与地块对象
- **以便**：后续事件、调查、对象行动能基于清晰地图模型扩展
- **验收标准**：
  - [ ] 每个地图块可配置区域名、地形、天气
  - [ ] 每个地图块可承载特殊状态、环境属性、地块对象
  - [ ] 地块对象可区分发现即显示与调查后显示
  - [ ] 环境属性不在地图页常驻显示，但可作为调查回报数据来源
- **不包含**：天气/状态/属性参与事件结算
- **优先级**：P1

### 11.2 成功标准（Success Criteria）

- [ ] 地图系统文档和运行表现不再把地图固定为 `4 x 4`。
- [ ] 玩家默认体验到的是 `8 x 8` 配置地图，但不会通过坐标范围提前知道地图边界。
- [ ] 地图页、通讯台、队员摘要对位置的表达职责清晰：摘要显示区域名，地图详情显示精确信息。
- [ ] 地形、天气、特殊状态、环境属性、地块对象在设计和数据上有明确边界。
- [ ] 本轮没有引入事件系统或通话菜单的大范围改动。

### 11.3 风险与缓解（Risks & Mitigations）

- **R1：地图尺寸配置牵动旧逻辑过多**
  - **缓解**：本轮验收聚焦默认 `8 x 8` 与配置读取；旧存档兼容、更多地图尺寸可作为实现时的风险项单独处理。
- **R2：层级过多导致 UI 信息拥挤**
  - **缓解**：摘要层只显示区域名和行动；地图详情分组显示；环境属性不常驻显示。
- **R3：特殊状态有持续时间但本轮不接事件系统，可能造成假功能感**
  - **缓解**：本轮只定义数据与可见性；事件影响、状态自动生成和复杂 UI 进入 Later / TODO。
- **R4：地块对象未来要驱动通话行动，但本轮不改菜单**
  - **缓解**：在地块对象上保留候选行动设计位，但明确本轮不实现对象级菜单。

## 12. Open Questions

- **已确认：地图配置文件的最终归属**：地图配置数据应作为新的内容数据管理，后续实施时应纳入内容数据与校验体系，而不是继续只放在代码初始状态中。
- **已确认：特殊状态的来源与生命周期**：特殊状态既可以由地图配置预置，也可以由事件系统生成；持续时间结束后的状态表现仍遵循“已知状态过期后回到未知 / 无已知特殊状态”的原则。
- **已确认：环境属性的调查记录格式**：环境属性报告作为结构化调查报告输出；日志中出现对应按钮，玩家点击后弹出查看报告详情。
- **已确认：区域名的边界规则**：本轮只支持“一个区域覆盖多个地块”；未被任何区域覆盖到的地块显示为“野外”。
- **已确认：旧存档迁移策略**：从固定 `4 x 4` 切换到配置驱动 `8 x 8` 时，旧存档需要重置。
- **Q1：地块对象的候选行动模型**：地块对象未来如何声明采矿、勘测、道具调查、修复等行动，以及这些行动如何进入通话菜单，需要下一轮设计；本轮不改。
