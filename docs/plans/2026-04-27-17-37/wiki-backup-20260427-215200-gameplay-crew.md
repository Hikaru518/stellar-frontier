# 队员系统

## 1. 概述（What & Why）

队员系统负责管理队员作为行动执行者与真实伙伴的双重身份：一方面，队员有位置、状态、主行动、通话指令和行动结果；另一方面，每名队员都有独立背景、说话语气、个人日记、性格标签、专长标签与 5 维轻量能力属性。

队员是玩家与地图、资源、事件系统互动的主要媒介。玩家主要通过通讯台进入通话，再在通话中向队员下达移动、待命、调查、采集、建设和事件处理等指令；地图用于查看地块、队员位置和行动状态，不直接下达队员指令。

人物表达规则让队员不只是可替换的数值单位。能力属性用于轻量表达能力倾向；自由性格标签用于展示与写作依据，不直接参与规则判定；专长标签用于表现角色独特能力或叙事身份，可在手工定义的情境中产生规则效果。

队员行动不会瞬间完成。除明确写为即时结算的通话选择外，行动会消耗游戏时间；队员移动到目标地点后默认进入原地待命状态。游戏时间持续推进时，队员行动持续结算；游戏关闭后，队员行动不继续推进。

## 2. 设计意图（Design Intent）

队员系统的核心情绪是“这些人不是工具，而是同行的真实伙伴”。每名队员都有独立来历、表达方式、记录习惯与自我标签；他们可以讨喜，也可以固执、别扭或不符合玩家偏好。系统承认这种差异，并把它作为角色可信度的一部分。

玩家在查看队员时，先感受到“这是一个具体的人”，再理解他适合做什么。背景档案、通话语气、个人日记、性格标签与专长标签共同提供可见线索，让玩家通过少量明确信息产生脑补，而不是只阅读一张效率表。

能力属性与专长标签支持角色区分，但不能吞掉个性。角色台词、日记与标签需要共同维护差异，避免所有人说话像同一个人。性格标签不进入硬规则判定，避免把人物性格压扁成效率条件。

行动规则让位置变化有明确时间成本，避免队员在地图上瞬间切换地点。玩家在通话中做出有代价的调度决策；通讯台、通话页面和地图对同一个队员状态保持一致显示。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **队员（Crew Member）**：队员系统的基本单位；既是玩家在星球表面执行行动的单位，也是可被玩家识别和牵挂的真实伙伴。
- **背景档案（Profile）**：由模板骨架与手写短句组成，记录原世界 / 出身、原职业、经历、性格、一句话自我介绍等信息。
- **通话语气（Voice & Tone）**：队员在通讯、事件反馈与对话中的表达差异，用于体现人物个性与关系感。
- **个人日记（Personal Log）**：队员在关键节点留下的记录，可在人物详情中查看，用于补充其主观经历与情绪变化。
- **日记可见性（Log Availability）**：通讯正常时可查看已传回日记；失联后只能看到此前传回内容；找回本人 / 遗体后可看到失联后的日记。
- **5 维轻量能力属性（Attributes）**：能力属性为体能、敏捷、智力、感知、运气；主要体现人物能力倾向，不承担复杂 RPG 数值体系。
- **自由性格标签（Personality Tags）**：自由文本短标签，用于展示与写作依据；不直接参与规则判定。
- **专长标签（Expertise Tags）**：体现人物独特能力或叙事身份的标签，数量不要求固定，可具有规则效果。
- **专长效果（Expertise Effect）**：专长在特定情境中提供差异化结果，例如“拾荒者”在调查时更容易发现额外物资。
- **人物详情页（Character Detail）**：展示队员背景档案、自由性格标签、专长标签、能力属性、个人日记和日记可见性状态的界面。
- **主行动（Main Action）**：队员同一时间正在执行的主要行为，例如移动、待命、调查、采集、建设或事件处理。
- **移动（Move）**：主行动的一种。玩家在与队员通话时，可以请求队员移动到指定地点；移动按路线逐格推进。
- **待命（Standby）**：队员没有执行主行动时的默认状态，不自动结束，直到玩家下达新指令或事件触发。
- **通讯状态（Communication State）**：决定队员是否可被通讯台联系、通话内容是否可见、个人日记是否能传回。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家在通讯台看到队员头像、姓名、专长标签、当前通讯状态、行动状态和剩余时间。
2. 玩家打开人物详情，通过背景档案、自由性格标签、专长标签与 5 维轻量能力属性形成第一印象。
3. 玩家接通队员通话，从语气、用词和回应节奏感知其性格差异。
4. 玩家在通话中下达行动指令，例如移动到指定地点、待命、调查、采集、建设或处理事件。
5. 行动消耗游戏时间并同步更新通讯台、地图和通话页面显示。
6. 行动结束或关键节点触发后，系统生成个人日记，玩家在人物详情中阅读队员的主观记录。
7. 玩家反复对照“他说的话”和“他写下的事”，逐渐把队员当成具体的人，而不是数值单位。
8. 若队员失联，玩家只能查看此前已传回的日记，信息突然中断。
9. 找回本人或遗体后，玩家解锁失联期间的日记，补完这段经历。

### 4.2 典型情境

- **高光时刻**：玩家同时查看 Mike、Amy、Garry 与更多队员时，能从档案、标签、通话语气、日记口吻与能力侧重点上一眼区分这些人。
- **调度时刻**：玩家从通讯台进入通话，查看目标、路线、预计耗时与中断后果，再确认队员移动到指定地块。
- **低谷 / 摩擦点**：某名队员失联后，详情页保留他的既有记录，但新日记空缺。玩家等待找回结果，之后才读到失联期间发生了什么。
- **系统反馈时刻**：拥有特定专长的队员参与调查或事件时，结果描述、线索发现或收益提示出现轻量差异。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

#### 队员总状态

| 状态 | 说明 | 是否可接普通移动指令 |
| --- | --- | --- |
| `idle` | 原地待命，没有执行主行动。 | 是 |
| `moving` | 正在移动到目标地点。 | 否，需要先停止当前行动。 |
| `working` | 正在调查、采集或建设。 | 是，但会中断当前工作。 |
| `inEvent` | 正在处理紧急事件或剧情事件。 | 否，需先处理事件。 |
| `lost` | 失联，无法正常通讯。 | 否 |
| `dead` | 死亡或不可用。 | 否 |

队员同一时间只能执行一个主行动。下达新行动时需要检查当前状态：

| 当前行动 | 新移动指令处理 |
| --- | --- |
| 原地待命 | 直接开始移动。 |
| 移动中 | 不允许直接改派目标，需要先停止当前行动。 |
| 调查中 | 提示会中断调查，玩家确认后开始移动。 |
| 采集中 | 提示当前未完成采集轮不会结算，玩家确认后开始移动。 |
| 建设中 | 提示建设会中断，材料是否返还按建设规则处理；基础规则默认不返还。 |
| 紧急事件中 | 不能下达普通移动指令。 |
| 失联或死亡 | 不能下达移动指令。 |

#### 日记可见性状态

日记条目按可见性分为：`已传回`、`未传回`、`失联锁定`、`找回解锁`。

- 队员通讯正常时，玩家可查看所有已传回日记；新生成但尚未传回的条目不可见。
- 队员失联后，日记列表锁定在失联前已传回的内容。
- 失联期间生成的日记标记为未传回 / 失联锁定，不直接展示。
- 玩家找回本人或遗体后，失联后的日记解锁为可查看内容，用于补足其经历与情绪变化。

#### 行动模型

队员行动至少需要以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `actionId` | 字符串 | 行动唯一 ID。 |
| `crewId` | 字符串 | 执行动作的队员。 |
| `actionType` | 枚举 | 行动类型，例如 `move`、`standby`、`survey`、`gather`、`build`、`event`。 |
| `status` | 枚举 | `pending`、`inProgress`、`completed`、`interrupted`、`failed`。 |
| `startTime` | 游戏秒 | 行动开始时间。 |
| `durationSeconds` | 整数或空 | 行动总耗时。持续待命等无固定结束时间的行动可为空。 |
| `finishTime` | 游戏秒或空 | 行动预计完成时间。持续待命等无固定结束时间的行动可为空。 |
| `fromTile` | 坐标 | 行动起点。 |
| `targetTile` | 坐标 | 行动目标地块。 |
| `resultPayload` | 对象 | 行动完成后的结果。 |

移动行动需要额外字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `route` | 坐标数组 | 从起点到目标地块的逐格路线，不包含起点，包含目标。 |
| `routeStepIndex` | 整数 | 当前正在前往的路线节点索引。 |
| `stepStartedAt` | 游戏秒 | 当前格移动开始时间。 |
| `stepFinishTime` | 游戏秒 | 当前格移动完成时间。 |
| `totalDurationSeconds` | 整数 | 整条路线总耗时，应与移动行动的 `durationSeconds` 一致。 |
| `remainingSeconds` | 整数 | 当前时刻距离抵达目标的剩余时间。 |

### 5.2 规则与公式

#### 人物表达规则

角色由背景档案、能力属性、自由性格标签、专长标签与关键节点日记组成。

背景档案采用模板骨架加手写短句，包括原世界 / 出身、原职业、经历、性格、一句话自我介绍等字段。`role` 可作为列表或卡片中的短展示字段，详细身份信息进入背景档案。

能力属性采用 5 维轻量表达：体能、敏捷、智力、感知、运气。属性主要用于表达角色能力倾向，可作为事件判定、文本分支或任务适配的参考，但不围绕复杂公式构建数值体系。

自由性格标签为展示与写作依据，例如“谨慎”“嘴硬”“怀旧”，不直接参与规则判定。

专长标签不设等级、不固定数量，可带规则效果。每个专长的触发条件与结果由内容手工定义，例如“拾荒者：调查废弃地点时，更容易发现额外物资”。专长影响调查或事件结果时，由调查 / 事件 / 资源等对应系统结算，不由人物字段直接凭空产生资源。

个人日记由关键节点触发生成，每个触发点默认生成 1 条短文本，并根据通讯状态决定是否传回与可见。

背包负重、关系 / 士气、角色成长、程序生成角色不属于人物表达规则的核心范围。

#### 移动指令入口

移动指令只能通过通话页面确认：

1. 玩家从通讯台主动联系队员，或接通队员来电。
2. 通话页面出现“请求前往”或同类行动按钮。
3. 玩家打开地图查看地块信息。
4. 玩家选择目标地块后返回通话页面。
5. 通话页面展示目标、预计耗时、是否会中断当前行动。
6. 玩家确认后，系统创建移动行动。

地图页面只负责查看地点和辅助选点，不直接向队员下达移动指令。

#### 目标选择规则

目标地块必须满足以下条件：

- 地块存在于当前地图范围内。
- 地块不是完全不可达地块。
- 队员当前状态允许接收移动指令。
- 起点和目标之间存在可通行路线。
- 目标不是队员当前所在地块；如果选择当前地块，按钮显示为“已在此处”或保持禁用。

不可达情况示例：

| 情况 | 处理 |
| --- | --- |
| 目标是水域且队员没有渡水能力 | 禁用确认按钮，提示“当前无法前往水域”。 |
| 路线被危险、建筑或事件阻断 | 禁用确认按钮，提示“当前路线不可达”。 |
| 队员失联 | 禁用移动指令，提示“信号中断，无法下达指令”。 |
| 队员处于紧急事件 | 禁用普通移动指令，提示“需先处理当前事件”。 |

#### 路线规则

基础地图为 `4 x 4` 网格，每个地块用坐标表示，例如 `(1,1)` 到 `(4,4)`。

移动路线按上下左右相邻地块逐格组成，不允许斜向移动。

路线选择规则：

- 使用曼哈顿路径。
- 路线只能经过可通行地块。
- 如果有多条同等长度路线，优先选择固定顺序，避免同一指令产生不稳定结果。
- 推荐固定顺序为：先横向，再纵向。
- 若固定顺序路线被阻断，则尝试其他同等长度路线。
- 若所有路线都不可通行，则目标不可达。

曼哈顿距离计算：

```text
distance = abs(fromX - targetX) + abs(fromY - targetY)
```

路线示例：

```text
起点：(1,1)
目标：(3,2)
优先路线：(2,1) -> (3,1) -> (3,2)
距离：3 格
```

#### 逐格模拟规则

移动按路线逐格推进。队员每走完一格，`currentTile` 立即更新为该格。

示例：

```text
第 1 日 00:00:00，Garry 从 (1,1) 前往 (3,2)。
路线：(2,1) -> (3,1) -> (3,2)。
第 1 日 00:01:00，Garry 抵达 (2,1)。
第 1 日 00:02:00，Garry 抵达 (3,1)。
第 1 日 00:03:00，Garry 抵达 (3,2)，进入原地待命。
```

逐格更新带来的规则影响：

- 通讯台显示的“当前位置”应随当前格更新。
- 地图应显示队员当前所在格和最终目标格。
- 队员如果在移动中被中断，停留在最近已经抵达的地块。
- 经过地块事件可以在每次抵达新格时触发。
- 基础行动规则只要求目标地块事件；完整经过地块事件表由事件内容定义。

#### 移动耗时

每一步移动的耗时按进入的目标地块计算。

默认地形移动耗时：

| 地形 | 每格耗时 |
| --- | --- |
| 平原 | `60 秒` |
| 丘陵 | `90 秒` |
| 森林 | `120 秒` |
| 山地 | `180 秒` |
| 沙漠 | `150 秒` |
| 水域 | 默认不可步行通过 |

总耗时计算：

```text
totalDurationSeconds = sum(每一步进入地块的移动耗时)
```

示例：

```text
路线：(2,1 平原) -> (3,1 平原) -> (3,2 丘陵)
总耗时 = 60 + 60 + 90 = 210 秒
```

简化数值可以使用：

```text
每格默认 60 秒
移动耗时 = 曼哈顿距离 * 60 秒
```

如果启用地形耗时表，通话页面和通讯台显示必须使用地形计算后的实际耗时，而不是简化耗时。

#### 移动开始

玩家确认移动后，系统执行：

```text
fromTile = crew.currentTile
targetTile = 玩家确认的目标地块
route = 根据 fromTile 和 targetTile 生成逐格路线
startTime = 当前 elapsedGameSeconds
stepStartedAt = startTime
stepFinishTime = startTime + 第一格耗时
finishTime = startTime + totalDurationSeconds
crew.status = moving
crew.currentAction = move
```

如果队员原本正在执行可中断行动，先中断原行动，再创建移动行动。

#### 移动推进

游戏运行中，时间系统每秒推进并检查移动行动。

逐格推进逻辑：

```text
if elapsedGameSeconds >= stepFinishTime:
  crew.currentTile = route[routeStepIndex]
  routeStepIndex += 1

  if routeStepIndex >= route.length:
    完成移动
  else:
    stepStartedAt = stepFinishTime
    stepFinishTime = stepStartedAt + 下一格耗时
```

如果一次检查跨过多个格子的完成时间，系统需要连续处理所有已经到期的步骤，直到当前步骤尚未完成或整条路线完成。

#### 移动完成

当队员抵达路线最后一个地块时：

```text
crew.currentTile = targetTile
crew.status = idle
crew.currentAction.status = completed
crew.currentAction = null
```

移动完成后，队员原地待命，不会自动调查、采集或建设。

抵达目标地块后可以触发：

- 地块进入事件。
- 危险遭遇。
- 通讯提醒。
- 可调查或可采集提示。

#### 移动中断

移动可能被中断。中断来源包括：

- 玩家要求停止当前行动。
- 队员遭遇紧急事件。
- 路线后续地块变为不可达。
- 队员失联、受伤、死亡。

逐格模拟下的中断规则：

| 中断时机 | 队员停留位置 |
| --- | --- |
| 已完成至少一格 | 最近已经抵达的地块，即当前 `currentTile`。 |
| 第一格尚未完成 | 起点地块。 |
| 刚好到达某一格同秒中断 | 先按结算顺序处理移动到格，再处理后续中断。 |

停止行动不是瞬间完成。停止当前行动耗时为 `10 秒`，停止完成后队员在当前地块待命。

#### 待命

待命是队员没有执行主行动时的默认状态。

| 行动 | 耗时 | 说明 |
| --- | --- | --- |
| 原地待命 | 持续状态 | 不自动结束，直到玩家下达新指令或事件触发。 |
| 停止当前行动 | `10 秒` | 用于从移动、调查、采集或建设切换到待命。 |

队员移动完成后自动进入原地待命。待命期间，队员仍可能因为地块危险、剧情条件或长时间停留触发事件。

### 5.3 参数与默认值

| 参数 | 默认 / 范围 | 说明 |
| --- | --- | --- |
| 能力属性维度 | 体能、敏捷、智力、感知、运气 | 轻量表达角色能力倾向。 |
| 能力属性取值 | `1-6` | `1` 明显短板，`2` 偏弱，`3` 普通，`4` 擅长，`5` 突出，`6` 极少数人的标志性强项。 |
| 自由性格标签 | 每名角色 `3-5` 个 | 作为内容规范，用于展示与写作依据。 |
| 专长标签 | 每名角色建议 `1-3` 个 | 作为内容规范，不作为硬限制。 |
| 关键节点日记 | 每个触发点 `1` 条短文本 | 由通讯状态决定是否传回与可见。 |
| 基础地图尺寸 | `4 x 4` 网格 | 使用坐标表示地块。 |
| 基础移动方向 | 上下左右相邻 | 不允许斜向移动。 |
| 简化每格移动耗时 | `60 秒` | 未启用地形耗时表时使用。 |
| 停止当前行动耗时 | `10 秒` | 停止完成后队员在当前地块待命。 |

每个队员至少需要维护以下行动运行态字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `crewId` | 字符串 | 队员唯一 ID。 |
| `name` | 字符串 | 队员显示名，例如 `Amy`、`Garry`。 |
| `role` | 字符串 | 队员身份或职业的短展示文本。 |
| `currentTile` | 坐标 | 队员当前所在地块。逐格移动时，该字段会随着移动进度更新。 |
| `status` | 枚举 | 队员总状态，例如 `idle`、`moving`、`working`、`inEvent`、`lost`、`dead`。 |
| `currentAction` | 对象或空 | 当前正在执行的主行动。 |
| `canCommunicate` | 布尔 | 当前是否可被通讯台联系。 |
| `lastContactTime` | 游戏秒 | 最近一次成功通讯时间。 |

人物内容还需要在内容数据中维护背景档案、通话语气、个人日记、自由性格标签、专长标签与 5 维轻量能力属性。

## 6. 系统交互（System Interactions）

- **依赖于**：
  - 时间系统：提供当前 `elapsedGameSeconds`、行动开始时间、当前步骤完成时间、整体行动完成时间与行动到期判断。
  - 地图系统：提供地块、地形、可通行性和目标选择信息。
  - 通话系统：提供玩家指令、行动确认、中断提示与人物通话文本。
  - 通讯状态：决定人物详情、日记与通话内容是否可见。
  - `content JSON`：提供背景档案、通话语气、日记文本、自由性格标签、专长标签与 5 维轻量能力属性。

- **被依赖于**：
  - 人物详情页：展示档案、标签、能力、日记与可见性状态。
  - 通讯台系统：读取队员状态，展示队员卡片、剩余时间、来电状态、失联状态、专长标签与通讯状态。
  - 通话页面：读取通话语气、可通讯状态、人物文本、行动后果、预计耗时和中断提示。
  - 地图系统：展示队员当前位置、目标和路线。
  - 资源系统：在队员采集行动完成或事件结果要求时结算产出；队员系统不直接凭空增加资源。
  - 调查行动与事件系统：读取专长标签与能力属性，触发手工定义的规则效果。

- **共享对象 / 状态**：
  - 队员 ID、当前位置、目标地点、行动状态、当前行动、行动结果。
  - 通讯 / 失联 / 找回状态、已解锁日记节点、专长规则效果、能力属性值。

- **事件 / 信号**：
  - 通讯状态变化、日记关键节点解锁、人物详情查看、调查判定请求、事件效果结算。
  - 队员进入地块、行动完成、长时间待命、紧急倒计时、失联、受伤、死亡。

关系说明：

- 队员系统是角色行动状态的中心。它不直接推进时间，也不直接绘制 UI，而是接收通话指令，读取地图地块信息，按时间系统结算行动，并把队员位置、状态和行动结果同步给地图、通讯台、资源和事件系统。
- 移动子模块属于队员系统，负责路线生成、逐格推进、移动中断和抵达待命。
- 通讯台不直接下达移动指令。玩家需要点击“通话”进入通话页面后确认行动。
- 地图负责展示队员位置、目标和路线状态，不直接发起指令。

结算顺序建议沿用时间系统规则：

1. 更新全局时间。
2. 结算已经到期的队员行动步骤。
3. 更新队员当前位置和状态。
4. 处理抵达地块后的事件。
5. 刷新通讯台、地图和通话页面显示。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：首次查看队员**：玩家打开人物列表 → 查看队员的档案、语气、标签、专长、5 维属性 → 队员在背景、说话方式与能力侧重点上一眼可分辨。
- **S2：通话 / 行动后日记生成**：关键通话或行动结算后 → 系统为相关队员生成个人日记 → 日记内容的措辞、关注点与其性格标签、通话语气保持一致。
- **S3：专长影响调查**：调查场景触发队员参与 → 若队员拥有如「拾荒者」等手工定义专长 → 结果描述或可发现线索出现差异化反馈。
- **S4：确认移动指令**：玩家从通讯台主动联系队员或接通来电 → 通话页面展示目标、预计路线、总耗时、是否会中断当前行动 → 玩家确认后系统创建移动行动。
- **S5：移动完成并待命**：队员按路线逐格移动 → 抵达目标地块 → `currentTile` 更新为目标，`status` 更新为 `idle`，当前移动行动标记为完成，通讯台显示“待命中”。
- **S6：地图展示移动状态**：地图显示队员当前所在地块、目标地块、路线高亮与移动剩余时间；当队员逐格移动时，地图上的队员标记随 `currentTile` 更新。

### 7.2 边界 / 失败场景

- **F1：队员失联**：队员进入失联状态 → 玩家查看该队员日记 → 新日记不继续可见，避免通过日记泄露失联后的信息。
- **F2：自由性格标签存在**：系统读取角色标签 → 仅用于展示与文本风格参考 → 不把自由文本标签当作规则条件参与判定。
- **F3：角色内容缺失**：某角色缺少背景、日记或标签 → 玩家进入人物详情 → 明确暴露内容缺口，而不是静默显示空白。
- **F4：目标不可达**：玩家选择水域、阻断路线或其他不可达目标 → 确认按钮禁用，并给出“当前无法前往水域”或“当前路线不可达”等提示。
- **F5：移动中断**：移动中的队员遭遇紧急事件、失联、受伤、死亡或玩家要求停止 → 队员停留在最近已经抵达的地块；第一格尚未完成时停留在起点。
- **F6：移动会中断工作**：队员正在调查、采集或建设 → 通话页面提示会中断当前工作及后果 → 玩家确认后再创建移动行动。
- **F7：地图误用为指令入口**：玩家在地图查看地块和路线 → 地图只辅助选点，不直接下达移动、调查、采集或建设指令。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- 选了真实伙伴感而非纯工具 / 劳动力，因为角色需要让玩家在探索中产生牵挂，而不是只作为效率插件。反模式：把角色写成可替换的数值表。
- 选了轻量 5 维 `1-6` 属性而非复杂 RPG 数值体系，因为本系统服务于叙事判断与角色差异，不追求构筑深度。反模式：围绕属性堆公式、阈值和最优解。
- 选了专长而非领域技能 / 细分技能表，因为专长更像角色记忆点，少量高辨识度标签更适合当前表达方式。反模式：专长太多，最后变成配装和构筑游戏。
- 选了自由性格标签而非结构化 / 混合标签，因为标签主要帮助展示与写作，不直接参与规则判定。反模式：标签写成同义词堆叠，或把自由标签写进硬判定。
- 选了关键节点日记而非每日自动流水账，因为日记应记录关系转折和重大经历。反模式：所有人通话 / 日记都用同一种声音，失去角色差异。
- 选了模板骨架 + 手写短句而非完全手写或程序生成，因为它能控制成本并保留人味。反模式：用随机组合替代具体人物塑造。
- 选了通话确认行动而非地图直接下指令，因为通话能保留人物语气、行动后果提示和调度代价。反模式：地图成为万能操作面板，削弱通讯台和人物存在感。
- 选了逐格移动而非瞬间移动，因为位置变化需要有时间成本，并同步影响通讯台、地图和事件触发。反模式：队员在地图上任意跳转，导致行动风险和时间管理失效。
- 选了队员系统不直接结算资源，因为资源产出应由采集、事件或资源系统负责。反模式：专长或人物字段直接凭空加资源，破坏系统边界。

## 10. Open Questions

1. **能力数据迁移**：现有 6 个能力字段迁移为新的 5 维轻量属性「体能 / 敏捷 / 智力 / 感知 / 运气」，不保留并行的旧属性体系；旧数据到新属性的具体数值转换规则仍需定义。
2. **角色内容补全**：待补充角色的具体姓名、来源世界、职业与人格短句仍需定义。
3. **专长内容定义**：每个专长的具体触发条件、效果数值与文案，需要在内容设计中逐条定义。
