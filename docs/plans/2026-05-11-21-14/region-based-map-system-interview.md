# Interview Log

## Q1: MVP 区域职责

问题：这次新版地图系统的 MVP 里，区域主要应该承担哪一层职责？

选项：

- 语义+去重：区域负责可见描述、对象归属、探索/事件去重；移动规则保持 tile 级。
- 语义+规则：区域也进入移动、危险、调查难度等规则计算，设计和实现范围更大。
- 仅展示语义：只先改玩家看到的区域信息，不改变对象归属或事件去重。

用户：请你具体解释 1 和 2。

回答：

- `语义+去重` 是较小的 MVP：区域成为“森林 / 飞船遗骸 / 村庄”的事实源，地图详情显示区域，地图对象归属于区域，区域内任一 tile 触发过一次性事件后，同一区域不重复触发；但移动路径、每格耗时、是否能走仍然按 tile 处理。
- `语义+规则` 是更大的版本：区域除了上面这些，还会直接进入规则计算，比如“雪原区域移动更慢”“飞船遗骸区域调查更容易触发电磁危险”“村庄区域有交易/敌对风险”。它产品上更完整，但会牵动移动、事件、调查、Editor、content validation 和测试面。

追问：在这个解释下，这次 MVP 的区域职责你希望选哪一个？

用户：语义+去重。

## Q2: 重叠区域展示

问题：当一个 tile 同时属于多个区域时，玩家地图详情应该怎么展示？

选项：

- 主区域+标签：显示一个主地点名，如“霜湾村庄”，并列出环境层标签如“雪原”。
- 全部并列：把所有区域都显示出来，如“雪原 / 村庄”，语义完整但文案更拥挤。
- 按优先级只显示一个：只显示最高优先级区域，界面清爽但会隐藏重叠语义。

用户：全部并列。

## Q3: 区域形状 MVP

问题：区域形状的 MVP 需要支持到什么程度？

选项：

- Box+Tiles：支持矩形 bounding box 和显式 tile 列表，覆盖大区域和不规则遗骸。
- 再加填充：额外支持连续填充/画刷式区域，编辑体验更好但工具复杂度更高。
- 只做 Box：实现最小，但飞船遗骸、村庄边缘等不规则区域会表达得很粗。

用户：请你解释一下 1 和 2。

回答：

- `Box+Tiles` 是数据模型优先的 MVP：区域可以是一个矩形范围，也可以是一组明确 tile。优点是实现清楚、验证容易、能表达“森林大片区域”和“飞船遗骸不规则覆盖”；缺点是手工维护大面积不规则区域会比较累。
- `再加填充` 是编辑体验更好的版本：策划可以像画图一样用画刷/填充生成连续区域，数据层仍可保存成 tiles 或压缩后的 shapes。优点是更适合 256 x 256 的内容生产；缺点是会明显增加 Map Editor 的工具量，包括选择、擦除、预览、撤销、重叠区域检查。

追问：在这个解释下，区域形状的 MVP 你希望选哪一个？

用户：我有一个问题，我们是不是可以认为就是一个连续图案？box 只是其中一个例外。

回答：可以，这个抽象更准确：区域本质上可以被理解为一张覆盖 tile 网格的“连续图案 / footprint / mask”。`box` 不是区域本质，只是快速生成或压缩这种 mask 的一种方式。

追问：那我们是否把区域的核心模型定义为“连续 footprint”，box 只是创建/压缩 footprint 的一种方式？

用户：连续 Footprint。

## Q4: 区域探索状态

问题：队员在区域内任一 tile 完成调查后，区域内应该共享哪些探索结果？

选项：

- 事件去重+对象可见：一次性事件不再重复，区域对象变为已发现/已调查；单个 tile 环境读数仍只记录当前点。
- 只做事件去重：区域内事件不重复，但对象和调查报告仍按具体 tile 处理。
- 整个区域全调查：区域内所有 tile 都视为已调查，反馈最强但会损失逐点探索价值。

用户：对于上一个问题，你所说的“画刷”如果是指地图编辑器，那么是需要画刷的。对于这个问题，请你详细解释每个选项。

回答：

- `事件去重+对象可见`：如果队员在“飞船遗骸”区域任一 tile 调查，系统记录这个区域已被调查过，因此遗骸区域的一次性事件不再在其他 tile 重复触发；同时属于该区域的对象，例如“破损反应堆”“散落补给”，可以变为可见或可交互。具体环境读数仍只来自队员调查的那个 tile，避免一次调查把整片区域所有细节都扫完。
- `只做事件去重`：区域只负责“别重复触发同一个事件”。对象可见性仍绑在具体 tile 上，调查 A 点不会揭示区域对象。实现更小，但和“地图对象归属于区域”的目标不完全一致。
- `整个区域全调查`：调查区域内任一 tile 后，区域覆盖的所有 tile 都变成已调查。玩家反馈最强，但会让大区域探索变得过快，也可能让 256 x 256 地图的空间细节失去意义。

追问：在这个解释下，队员在区域内任一 tile 完成调查后，区域内应该共享哪些探索结果？

用户：事件去重+对象可见。

## Q5: 地图对象是否也是区域

问题前讨论：用户提出，这个 tile 可能归属于多个区域，每个区域都有自己的需要被调查触发的对象。地图对象是否也可以被抽象成一种特殊的区域，只需要判断自己是否在“地图对象”这个区域内并调查，而不是“地图对象属于某一个区域”。这样可以避免在一个 tile 调查时，把所有从属区域的可调查内容全部显示出来。

回答：建议不要把所有东西都叫“区域”，而是抽象成上层对象 `map feature / 地图特征`。它们都拥有连续 footprint，都可以覆盖多个 tile，都可以重叠；“雪原”“村庄”“飞船遗骸”“破损反应堆”只是不同 `kind`，不是父子归属关系。

问题：我们是否把“区域”和“地图对象”统一成同一种带 footprint 的地图特征模型？

选项：

- 统一 Feature：雪原、村庄、遗骸、设备都用 footprint 表达，靠 kind 和调查规则区分。
- 两类平级：区域和地图对象仍是两种模型，但都可有 footprint 且互不从属。
- 对象归属区域：保留区域包含对象的模型，再额外设计过滤规则避免批量揭示。

用户：统一 Feature。

## Q6: 调查命中规则

问题：当队员所在 tile 同时命中多个可调查 Feature 时，调查行动应该如何选择目标？

选项：

- 通话中选择目标：通话页列出可调查 feature，如调查村庄/遗骸；玩家明确选一个。
- 系统优先级自动选：按 priority 自动调查最高优先级 feature，操作少但可能不符合玩家意图。
- 一次广域扫描：一次调查会扫描多个 feature，但只结算轻量发现，不直接完成所有事件。

用户：有优先级的设定，如果优先级相同，那么在通话中选择目标，优先级是一个 1-100 的整数。

## Q7: 低优先级 Feature 可见性

问题：不同优先级的 Feature 重叠时，低优先级 Feature 在通话中应该怎样处理？

选项：

- 作为背景可见：低优先级仍显示在地图详情/通话上下文，但调查按钮默认只给最高优先级。
- 完全隐藏：低优先级在高优先级未处理前不可见，悬念更强但可能让地图信息不完整。
- 仍可手动选择：所有可调查 Feature 都能选，priority 只负责排序和默认项。

用户：作为背景可见。

## Q8: 可调查性

问题：哪些 Feature 应该能成为调查目标？

选项：

- 显式标记：Feature 用 investigatable 标记决定；雪原可只是背景，也可被标记为异常雪原调查目标。
- 按类型固定：biome 永远背景，site/object/hazard 永远可调查，规则简单但弹性少。
- 全部可调查：所有 Feature 都能调查，表达自由但通话按钮容易过多。

用户：显式标记。

## Q9: Feature 调查结果与状态

问题：调查某个 Feature 成功后，运行时应该标记什么？

选项：

- Feature已调查：标记该 Feature 已调查/已触发，解锁其内容；tile 只记录本次调查报告。
- 只记事件历史：不存 Feature 状态，只用事件历史去重；模型轻但 UI 难显示进度。
- Footprint全调查：Feature 覆盖的所有 tile 都变成已调查；反馈强但会扫掉太多空间细节。

用户：对于一个可调查的 feature 来说，他应当承载现在的地图对象的功能。也就是说会有状态。这个状态是一个字符串。你具体可以参考现在的地图对象的代码中的做法。

## Q10: Feature 状态模型

说明：当前 `mapObject` 已经有 `status_options`、`initial_status`，运行时在 `map.mapObjects[objectId].status_enum` 里保存字符串状态。新版可以把这套能力上移到 `Feature`：每个可调查 Feature 有 footprint、priority、visibility、investigatable、status options、initial status 和 actions；运行时按 `featureId` 保存当前字符串状态，而不是按 tile 保存对象状态。

问题：Feature 的“是否可见/可调查”和“当前状态字符串”应该怎么关系？

选项：

- 分开建模：visibility/investigatable 控制出现和按钮；status 字符串表达 damaged/repaired/searched 等内容状态。
- 全放状态里：hidden/discovered/damaged 都是同一个 status 字符串，灵活但通用 UI 判断更难。
- 只保留状态：不做 visibility/investigatable，所有逻辑都靠事件条件解释。

用户：我觉得这种可调查的内容还是应当单据建模，我认为可以被认为是一种特殊的“map feature”。

记录理解：普通 `MapFeature` 提供玩家可见语义、标签、footprint、priority；可调查 `InvestigatableFeature` 是特殊 `MapFeature`，承载现在 `mapObject` 的能力，包括字符串状态、可见性、行动、事件触发和去重。

## Q11: Feature 内容归属

问题：新版 Feature 的静态定义和 footprint 应该主要放在哪里？

选项：

- 地图文件内：每张 map 自己定义 features 和 footprint，Editor 最直接，适合地图专属内容。
- 独立 Feature 库：放在 `content/map-features/*.json`，地图只引用；复用强但编辑链路更复杂。
- 定义库+地图放置：通用定义在库里，footprint/priority 在 map 里；最规范但 MVP 成本最高。

用户：地图文件内。

## Q12: 旧字段迁移

问题：旧的 `tile.areaName` / `tile.objectIds` / `radar.regions` 在新版设计里应该如何处理？

用户：分别解释一下这个三个内容在原本的系统中是做什么的？

回答：

- `tile.areaName`：每个 tile 上的一段地点文案。队员位置、地图 focus label、通话里的目标位置会读它，例如“IAFS坠毁点 (0,0)”。它不是一个真正的区域定义，只是重复写在每个 tile 上的名字。
- `tile.objectIds`：这个 tile 上有哪些地图对象的引用。对象定义本体在 `content/map-objects/*.json`，运行时状态在 `map.mapObjects[objectId].status_enum`。当前对象是否可见、是否能提供行动，基本都是从“队员当前 tile 的 objectIds”查出来的，而且校验脚本还禁止同一个 object id 被多个 tile 复用。
- `radar.regions`：雷达显示层的区域形状，有 `circle / box`、label、priority、tone。现在它主要服务地图视觉和 label fallback，例如 tile 没有具体 areaName 时显示 region label；它不承担对象归属、调查状态、事件去重或运行时状态。

用户：那新版本为什么不需要这些字段呢。

回答：

- `tile.areaName` 的信息会被 `features` 覆盖关系取代。一个 tile 属于“雪原 / 村庄”，地图详情直接从覆盖该 tile 的 Feature 列表生成地点文案，不需要在 65,536 个 tile 上重复写 `areaName`。
- `tile.objectIds` 会被“可调查 Feature”取代。飞船遗骸或破损反应堆本身就是一个带 footprint、priority、status 的 Feature；队员站在哪个 tile，只要这个 tile 命中该 Feature，就能调查它。这样对象天然可以跨多个 tile，也不会被迫归属某一个 tile。
- `radar.regions` 的能力会并入 `features`。现在 radar region 已经有 shape、label、priority、tone，但它只是显示层；新版 Feature 也有 footprint、label、priority，并且还能承载 gameplay 语义。继续保留一个独立 `radar.regions` 作为正式模型，会造成“显示区域”和“玩法区域”两套形状不同步。

追问：你是否接受把这三个旧字段降级为兼容 fallback，新地图语义统一由 `map.features` 承担？

用户：前两个需要一处。对第三个 radar 是这样的，我们现在说的所有内容都是玩法层面的，而 radar 是负责渲染层的显示，我们先不管。两套状态不同步不要紧。等之后再来修复。

确认：我理解“前两个需要一处”是指 `tile.areaName` 和 `tile.objectIds` 在新版玩法模型里应该移除，不再作为正式 source of truth；`radar.regions` 属于渲染层，先不纳入这次玩法设计，允许它和玩法 Feature 暂时不同步。

追问：这个理解是否正确？

用户：正确，移除前两个。
