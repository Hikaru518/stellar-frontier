# 任务系统策划案

## 1. 概述（What & Why）

任务系统是玩家的叙事行动面板，用来追踪“现在发生了什么、下一步该做什么、哪些关键事实已经确认”。  
它以“任务 → 子任务 → 待办事项”三层结构组织主要任务与次要任务；主要任务围绕剧情主线“找到回家的办法”。  
任务只负责聚合与展示，不替代 event / objective 的规则判定，也不改变“通讯是指令通道、地图只读”的项目约束。  
任务完成状态只分为未完成 / 已完成，并由事件显式推进。

## 2. 设计意图（Design Intent）

任务系统让玩家在失联、分散行动和线索不完整的压力下，仍能抓住“下一步该追什么”的方向感。  
它强调叙事行动感：任务像通讯台外的行动面板，汇总任务状态与关键事实，而不是替玩家解释世界。  
玩家应通过任务看见主线目标、支线牵挂和队员当前处境之间的关系，尤其是“找到回家的办法”这条长期牵引。  
任务系统避免变成百科、待办软件或万能导航；它不替代事件判定，不绕过通讯台和地图的行动约束。  
任务的完成与推进来自事件的显式结果，让玩家相信进展来自探索、通话和选择，而不是界面自动结算。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **任务系统**：承载叙事目标与行动线索的常驻行动面板，用来聚合玩家当前该关注的事情，不替代事件或目标判定。
- **主要任务**：剧情主线任务，当前核心方向是“找到回家的办法”，用于维持长期叙事牵引。
- **次要任务**：非主线但值得追踪的目标，用于记录支线发现、局部问题、人物诉求或可选探索方向。
- **任务**：任务系统的最高层级条目，可为主要任务或次要任务，包含若干子任务与当前节点描述。
- **子任务**：任务下的中层目标，用于拆分一个任务的阶段性目标；是否完成由事件显式推进，不自动由待办事项汇总决定。
- **待办事项**：子任务下的最小提示项，用于展示具体线索、地点、对象或下一步关注点，不直接执行行动。
- **当前节点描述**：作者为任务或子任务当前进度编写的固定文案，用于解释上下文、叙事状态和玩家为何应关注它。
- **任务状态**：任务、子任务、待办事项只使用“未完成 / 已完成”两种状态，不设置失败、过期、隐藏完成等额外状态。
- **任务推进事件**：由剧情事件、地点事件、通话选项或其他系统结算显式发出的推进信号，用于完成指定任务、子任务或待办事项。
- **任务侧边栏**：PC UI 中常驻展示任务的侧栏组件，默认提供折叠摘要，展开后展示任务详情、层级与当前节点描述。
- **导航入口**：任务详情中指向相关页面、队员、地图位置、通讯对象或事件上下文的入口，只负责引导玩家前往，不直接下达行动或结算结果。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家在主界面看到常驻任务侧边栏；折叠状态下显示未完成任务数量、主线优先项和最近更新摘要。
2. 玩家展开侧边栏，按状态（未完成 / 已完成）和类型（主任务 / 次任务）筛选任务，快速判断当前叙事行动面板里的可推进目标。
3. 玩家选择一个任务，查看三层结构：任务摘要、当前节点描述、独立任务内容；当前节点描述由作者固定，不随运行时自动改写。
4. 当事件产生显式推进条件时，系统把相关任务标记为可推进或已更新，并在侧边栏摘要中提示变化。
5. 玩家点击任务详情中的导航入口，系统带玩家前往相关页面、角色、地点或事件上下文，但不自动执行调查、移动、通话选项或事件结算。
6. 玩家在目标页面完成实际行动后，事件系统显式推进任务节点，任务状态从未完成更新为下一节点或已完成。
7. 玩家回到任务侧边栏，确认主 / 次任务进展，并决定下一步优先处理的叙事行动。

### 4.2 典型情境（可选）

- **高光时刻**：玩家结束一次关键通话后，主任务在侧边栏即时高亮更新；展开后能看到新的固定节点描述，清楚知道剧情进入了下一阶段。
- **高光时刻**：玩家在多个次任务之间筛选，发现某个任务详情提供地点导航入口；点击后直接定位到相关上下文，但最终行动仍由玩家手动决定。
- **低谷 / 摩擦点**：如果任务只显示标题而缺少当前节点描述，玩家会不知道该去哪里、为什么去；因此每个当前节点必须有作者固定描述。
- **低谷 / 摩擦点**：如果导航入口直接执行任务动作，玩家会失去决策感并混淆“查看任务”和“推进事件”；因此任务详情只负责导航，不负责执行。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

任务系统固定为三层结构：任务 → 子任务 → 待办事项。层级深度不可配置，不支持第 4 层或任意树状嵌套。任务、子任务、待办事项都只允许两种状态：`未完成`、`已完成`。

任务状态只能由事件系统或任务推进信号显式写入；不得因玩家查看、导航、筛选、展开侧边栏、进入地图、打开通讯台等 UI 行为改变状态。父任务 / 子任务不因其下级条目全部完成而自动完成，除非收到针对该父任务 / 子任务的显式完成推进事件。

### 5.2 规则与公式

- **主 / 次任务**：每个任务必须标记为主要任务或次要任务；该标记用于展示与筛选，不改变事件判定逻辑。
- **当前节点描述**：作者为每个任务或子任务的当前阶段固定编写描述；系统不自动拼接、不根据 objective 状态生成摘要。
- **完成推进**：任务完成由事件显式推进；任务系统只记录和展示结果，不替代 event、objective、地图对象、队员行动的判定。
- **独立任务内容**：任务文本、层级、主次标记、当前节点描述作为独立任务内容维护，不从事件标题或地图对象说明隐式推导。
- **导航入口**：任务条目可提供跳转入口，如前往地图、通讯台或相关页面；导航只改变界面位置，不执行移动、调查、通话选项或任何游戏行动。
- **侧边栏交互**：任务侧边栏支持折叠 / 展开；折叠仅隐藏详情，不影响状态、筛选条件或任务推进。
- **筛选**：任务列表至少支持按完成状态与主 / 次任务筛选；筛选只影响可见列表，不改变任务数据。

### 5.3 参数与默认值

| 参数 | 默认值 | 规则 |
| --- | --- | --- |
| 层级深度 | 3 层 | 固定为任务 → 子任务 → 待办事项 |
| 任务状态 | 未完成 / 已完成 | 无进行中、失败、隐藏等额外状态 |
| 任务类型 | 主要任务 / 次要任务 | 每个任务必填 |
| 完成方式 | 显式推进事件 | 不允许由子项全完成自动推导 |
| 侧边栏状态 | 展开 | 玩家可折叠 / 展开，持久化可选 |
| 默认筛选 | 全部状态 + 全部类型 | 玩家可按状态与主次组合筛选 |

## 6. 系统交互（System Interactions）

- **依赖于**：`event/objective` 提供任务推进与完成判定；`world_flags` / `history` 提供任务可见性、分支状态与已发生事实；地图提供地点、区域与目标定位；队员行动系统提供移动、调查、待命、停止等执行状态；通讯台 / 通话提供下达行动与剧情选择的唯一正式入口；`content/quests` 提供任务定义、展示文案、分组与追踪规则；存档提供任务可见性、追踪状态与历史快照恢复；手机 companion 可接收任务相关提示，但不持有权威状态。

- **被依赖于**：UI 依赖任务系统聚合当前目标、已完成目标、任务来源与相关地点；通讯台 / 通话可引用任务摘要帮助玩家理解当前通话上下文；地图可显示任务相关地点标记与目标提示；手机 companion 可展示私密任务提醒或任务相关来电摘要。

- **共享对象或状态**：任务系统只读取并聚合 `quest`、`objective`、`event`、`world_flags`、`history`、队员位置 / 行动状态、地图地点 ID、通话上下文与存档状态；任务不复制 objective 判定，不替代 event 结算，不直接修改队员行动结果。玩家可选择追踪任务、查看任务详情、查看关联目标与地点；任何实际行动仍必须经通讯台 / 通话或既有事件选项触发。

- **事件或信号**：订阅 `objective_updated`、`objective_completed`、`event_resolved`、`world_flag_changed`、`history_appended`、`crew_action_started`、`crew_action_completed`、`crew_arrived_at_tile`、`call_started`、`call_option_selected`、`save_loaded`、`save_written`、`mobile_companion_connected` 等信号；输出 `quest_visibility_changed`、`quest_updated`、`quest_completed`、`quest_tracked_changed`、`quest_notification_requested` 等展示层信号。这些输出只驱动 UI、地图标记、通知与 companion 展示，不绕过通讯台下达指令，也不作为 event/objective 的权威判定来源。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1**：坠毁序章完成 → 系统创建主线任务“重整态势”，并在任务侧边栏置顶显示 → 玩家明确当前首要目标与下一步待办。
- **S2**：玩家通过事件选项维修发电机成功 → 任务系统收到显式进度事件并勾选对应待办 → 任务详情显示发电机待办已完成，父任务进度同步更新。
- **S3**：玩家打开任务侧边栏 → 切换“未完成 / 已完成”筛选 → 列表只显示对应状态任务，且不改变任务本身状态。
- **S4**：玩家在任务详情点击“前往地图”或“打开通讯台” → 页面导航到对应入口 → 系统只定位目标上下文，不自动下达移动、调查、通话选项等行动。

### 7.2 边界 / 失败场景

- **F1**：某任务所有子项都已勾选，但父任务未收到显式完成事件 → 系统保持父任务为未完成或待结算状态 → 避免任务被隐式自动完成，等待后续事件确认。
- **F2**：任务缺少当前节点描述 → 详情页显示简短兜底文案 → 玩家仍能看到任务标题、状态、待办与可用导航。
- **F3**：任务导航目标不存在或当前不可用 → 玩家点击导航动作 → 系统显示不可用提示并留在当前页面，不执行任何行动。
- **F4**：手机 companion 不在线 → 任务侧边栏与详情仍在 PC 可用 → 私密提醒或手机侧同步不阻塞任务查看、筛选与导航。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- **叙事行动面板 vs 完整任务管理器**：我们选了叙事行动面板而不是完整任务管理器，因为任务系统应服务“当前可做什么、为什么值得做”的决策，而不是变成待办清单软件。
- **任务聚合 vs 任务自带规则判定**：我们选了任务聚合而不是任务自带规则判定，因为行动合法性、事件结算和状态变化仍由既有行动 / 事件 / 地图 / 队员系统负责，任务只负责整理目标与入口。
- **三层固定 vs 无限树**：我们选了三层固定结构而不是无限任务树，因为主线 / 任务 / 步骤已经足够表达叙事推进，且能避免 UI、存档和内容编辑复杂度失控。
- **显式完成 vs 自动汇总完成**：我们选了显式完成而不是自动汇总完成，因为任务完成应是可感知的叙事节点，避免玩家错过结算文本、状态变化和下一步目标。
- **常驻折叠侧边栏 vs 独立页面**：我们选了常驻折叠侧边栏而不是独立页面，因为任务信息应伴随通讯、地图和行动决策出现，而不是把玩家拉离当前操作语境。
- **导航入口不执行 vs 一键行动**：我们选了导航入口不执行而不是一键行动，因为任务面板只能把玩家带到相关页面或对象，实际行动仍必须经过原系统确认。
- **只分主次 vs 多分类标签**：我们选了只分主次而不是多分类标签，因为当前阶段只需要表达叙事优先级，过多标签会制造管理负担和内容维护成本。

- **要避免的反模式**：把任务系统做成第二套行动系统，绕过通话、地图、事件或队员状态的既有规则。
- **要避免的反模式**：让任务文本替代事件内容，导致同一段叙事在任务、通话和事件日志中重复维护。
- **要避免的反模式**：用大量标签、筛选器和排序规则包装少量任务，造成 UI 重于玩法。
- **要避免的反模式**：任务自动完成但没有明确反馈，使玩家不知道自己完成了什么、为什么推进、下一步去哪里。

## 10. 本轮范围与阶段拆分（Scope & Phasing for This Round）

### 10.1 MVP

本轮 MVP 只实现可验证的任务系统基础闭环：

- 独立定义任务内容数据，不把任务写死在页面或事件代码中。
- 提供少量手写任务，用于验证任务列表、详情、状态变化与事件推进链路。
- 任务只能由事件或明确的系统入口显式推进，不做自动扫描式进度推导。
- PC 端提供常驻任务侧边栏，支持折叠 / 展开。
- 提供任务列表与任务详情视图，展示任务标题、描述、目标、状态与关联线索。
- 支持主线 / 支线筛选。
- 支持完成状态筛选。
- 控制中心或现有导航中提供任务入口，但入口本身不执行任务、不推进状态。
- 任务状态进入存档，并能在刷新后恢复。

### 10.2 Later

以下内容留到后续轮次，不进入本轮 MVP：

- 任务失败、过期、隐藏任务与发现规则。
- 复杂标签、全文搜索、任务追踪 / 置顶。
- 手机端任务详情展示与多设备同步体验。
- 任务编辑器与可视化 authoring 工具。
- 任务奖励，包括物品、资源、关系、声望或剧情奖励。
- 自动进度推导、数字进度条、计数型目标。
- 无限任务树、复杂并行节点、parallel join / sync 等高级流程结构。

### 10.3 不做

本轮明确不做以下设计：

- 不允许任务直接下达队员行动；行动仍必须走通讯台 / 通话 / 事件选项等既有渠道。
- 不允许地图直接完成任务；地图只能提供信息、位置与导航上下文。
- 不用任务系统替代事件系统；任务只记录目标、状态与可见进度，事件仍负责剧情分支与结算。
- 不做离线时间推进或离线任务结算。
- 不让手机端任务信息阻断主线推进；PC 仍是权威状态与主流程入口。

## 11. 本轮验收与风险（Acceptance & Risks）

### 11.1 Player Stories / Play Scenarios（验收切片）

#### PS-001: 侧边栏查看任务
- **作为**：正在控制中心、通讯台、地图或通话页面浏览局势的玩家
- **我能**：从常驻侧边栏看到当前任务摘要、任务状态和关键待办
- **以便**：不用离开当前页面也能判断下一步该关注什么
- **验收标准**：
  - [ ] 侧边栏在 PC 主要页面可见，并支持折叠 / 展开
  - [ ] 每条任务至少显示标题、主要 / 次要类型、未完成 / 已完成状态
  - [ ] 空任务状态有明确占位文案，不显示报错或空白列表
- **不包含**：任务奖励、全文搜索、复杂标签
- **优先级**：P0

#### PS-002: 筛选任务列表
- **作为**：拥有多个任务的玩家
- **我能**：按完成状态与主要 / 次要任务筛选列表
- **以便**：快速区分当前未完成目标、已完成记录和主线优先级
- **验收标准**：
  - [ ] 支持查看全部 / 未完成 / 已完成
  - [ ] 支持查看主要任务 / 次要任务
  - [ ] 筛选切换只影响展示，不改变任务或事件状态
- **不包含**：全文搜索、复杂排序、自定义标签
- **优先级**：P1

#### PS-003: 事件推进待办
- **作为**：通过通话或事件选项推进剧情的玩家
- **我能**：在事件推进后看到任务待办同步更新
- **以便**：理解事件结果带来的下一步目标，而不是只依赖一次性对话文本
- **验收标准**：
  - [ ] 事件显式推进后，相关任务 / 子任务 / 待办状态更新
  - [ ] 已完成的待办不会继续作为当前行动建议突出显示
  - [ ] 父任务 / 子任务不会仅因下级全部完成而自动完成，除非收到显式完成事件
- **不包含**：独立任务脚本系统、任务反向驱动事件结算
- **优先级**：P0

#### PS-004: 导航入口不直接执行行动
- **作为**：从任务侧边栏寻找下一步入口的玩家
- **我能**：点击任务入口跳转到相关页面或上下文
- **以便**：更快抵达通话、地图或通讯台，但仍由既有系统执行真实行动
- **验收标准**：
  - [ ] 任务入口只能导航到相关页面、角色、地点或事件上下文
  - [ ] 点击任务入口不会直接发出移动、调查、通话选项或事件选择
  - [ ] 所有行动仍必须通过既有通讯台 / 通话 / 事件选项确认
- **不包含**：一键执行任务、自动寻路、自动接通队员
- **优先级**：P0

#### PS-005: 存档恢复任务展示
- **作为**：关闭并重新打开游戏的玩家
- **我能**：看到任务侧边栏恢复到与存档状态一致的任务进度
- **以便**：继续理解当前目标，不因重启丢失任务上下文
- **验收标准**：
  - [ ] 重新加载存档后，任务列表与当前事件、队员和游戏状态一致
  - [ ] 已完成任务不会回退为未完成任务
  - [ ] 未完成任务能在刷新后保持正确状态与当前节点描述
- **不包含**：离线任务推进、跨设备存档同步、云存档
- **优先级**：P0

### 11.2 成功标准（Success Criteria）

- [ ] PC 主要页面出现任务侧边栏，且不遮挡现有核心操作区域。
- [ ] 至少一个现有事件链能显式推进任务待办，并可在 UI 中观察到变化。
- [ ] 任务筛选切换后，任务数据本身不发生改变。
- [ ] 任务导航入口只改变页面 / 选中上下文，不直接执行任何 gameplay action。
- [ ] 刷新页面或重新进入游戏后，任务展示与存档中的事件进度保持一致。
- [ ] 新增任务内容通过独立任务内容数据维护，不需要在多个 UI 文件重复硬编码同一任务文案。

### 11.3 风险与缓解（Risks & Mitigations）

- **R1**：任务侧边栏侵入现有 PC 页面布局，影响控制中心、通讯台、地图等页面的可读性。
  - **缓解**：本轮只做轻量侧边栏；优先复用现有低保真控制台视觉语言，并验证不遮挡核心按钮。

- **R2**：任务系统变成事件系统之外的第二事实源，导致任务状态与事件结算不一致。
  - **缓解**：任务状态只由显式任务推进事件更新；任务 UI 不独立结算事件，也不直接写入行动结果。

- **R3**：任务文案过少，玩家看到状态但无法理解下一步为什么要做。
  - **缓解**：每条任务至少配置标题、简短说明和当前节点描述；MVP 优先覆盖少量高质量任务。

- **R4**：范围膨胀为完整 RPG 任务系统，引入奖励、复杂条件、追踪地图标记或任务脚本。
  - **缓解**：本轮只实现查看、筛选、显式推进和导航入口；奖励、复杂标签、编辑器和任务脚本全部留到 Later。

- **R5**：内容维护成本上升，任务文案散落在事件、UI 和代码中。
  - **缓解**：任务定义集中在独立任务内容数据中；UI 只负责展示，不复制任务描述和待办规则。

## 12. Open Questions

- **Q1（实施时复议）**：任务 content schema 的最小字段集如何定稿？需要确认 `id`、`title`、`description`、`status`、`priority`、`category`、`sourceEventId`、`targetCrewId`、`objectiveRefs`、`createdAt`、`updatedAt`、`completedAt` 等字段中哪些进入 MVP，哪些保留为 Later。
- **Q2（实施时复议）**：任务推进事件的命名与契约如何定义？需要统一任务创建、状态变更、目标进度更新、完成、失败、取消等事件的 event name、payload shape 与幂等规则。
- **Q3（实施时复议）**：PC 侧边栏在控制中心、通讯台、地图、通话页的布局适配是否使用同一任务摘要组件？需要确认窄屏、滚动、折叠、优先级排序与空状态规则。
- **Q4（实施时复议）**：任务与 `event_log` / global log 的引用关系如何落地？需要确认任务是否只保存 log entry id 引用，还是允许任务拥有独立历史记录并反向关联事件日志。
- **Q5（实施时复议）**：tracked task 是否需要持久化为 `GameState` 的一等字段？需要确认刷新页面、重置存档、事件回放与未来存档迁移时的行为。
- **Q6（下一轮 brainstorm）**：手机端未来是否展示任务列表或仅展示与私人来电相关的任务提示？需要单独确认手机端信息权限、已读状态与 PC 权威状态同步边界。
- **Q7（下一轮 brainstorm）**：任务编辑器未来边界是什么？需要确认是否只编辑任务模板 / schema，还是允许编辑任务触发条件、事件绑定、测试预览与内容校验规则。
