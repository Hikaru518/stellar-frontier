# Dual Device Play Interview

**Q:** 是否需要在动手访谈前，先探索现有 docs（core-ideas、gameplay、ui-designs），找出与“双设备联动”相关的既有约束？

**A:** 扫描现有 docs（推荐）

**Q:** 以下哪些方向值得做互联网研究？可以多选。

**A:** 二屏互动游戏案例, 扫码配对与低延迟, 手机即游戏内设备

**Q:** 手机端在这款游戏里应该是什么定位？ Options: 可选增强（推荐）, 核心必需, 章节/模式必需.

**A:** 可选增强（推荐）

**Q:** 在“电脑端完整可玩、手机端可选增强”的前提下，手机端最核心的独特体验应该是什么？ Options: 私密通讯终端（推荐）, 信号扫描/解码仪, 应急确认器.

**A:** 私密通讯终端（推荐）

**Q:** 为了“扫码最简单 + 延迟极低”，你能接受 MVP 引入一个轻量联网 relay 服务吗？ Options: 接受 relay（推荐）, 只要本地直连, 混合方案.

**A:** 接收 relay，但是可以探索一下 webrtc 之类的，只要在同一个局域网就建立连接

**Q:** 第一条 MVP playable slice 最应该验证哪种双设备体验？ Options: 配对 + 私密来电（推荐）, 紧急事件双端处理, 信号扫描/解码任务.

**A:** 配对 + 私密来电（推荐）

**Q:** 手机断线、锁屏、没电或玩家不想用手机时，游戏应该如何处理？ Options: 安全回退（推荐）, 剧情化轻摩擦, 风险会影响结果.

**A:** 安全回退（推荐）

**Q:** 这套电脑 + 手机联动默认面向哪种玩家关系？ Options: 单人双设备（推荐）, 本地双人协作, 两者都支持.

**A:** 单人双设备（推荐）

**Q:** 这次策划案应该把双设备联动设计成 Stellar Frontier 的正式子系统，还是先作为通用新游戏概念？ Options: Stellar 子系统（推荐）, 通用新游戏概念, 平台能力原型.

**A:** 你是没活了吗？问我这么多明显的问题，你就出一些完整的方案，如果你觉得实在是有纠结的地方再来问我；而且这个调研的重中之重不是玩法，而是技术可行性，可访问性，便利性，低延迟，我希望这个双端不要成为玩家的负担。
