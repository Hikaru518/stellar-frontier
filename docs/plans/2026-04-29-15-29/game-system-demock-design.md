## 1. 一句话总结

本次重构要把游戏 UI、运行时内容和数据模型重新对齐：删除所有会让玩家误判已实现机制的 mock/legacy 内容，把队伍缩减为 Mike、Amy、Garry 三人，并建立一条明确规则：UI 只能展示真实状态、正式内容资产、或不虚构世界状态的中性占位。

## 2. 背景与问题（As-is）

- **当前做法**：原型阶段的 UI 把真实 `GameState`、正式 `content` 资产、演示文案、legacy 兼容层和硬编码剧情句混在一起展示。通讯台、初始日志、设施摘要、角色 summary、编辑器 legacy event 展示和地图 legacy 投影都存在类似问题。
- **痛点/成本**：玩家无法判断哪些内容是真实机制，哪些只是氛围或旧资产。例如“最近一次通讯”“天线偏移”“信号噪声”“脚步声”会暗示不存在或未接入的系统。
- **为什么现在做**：项目已经从原型展示进入内容和系统收敛阶段，继续保留 mock/legacy 会污染后续设计判断、测试样例和内容生产规则。

## 3. 目标（Goals）

- 删除 UI 和代码中所有硬编码剧情、台词、假系统数值和假机制暗示。
- 审计 `content` 中没有真实机制或清晰语义归属的文本字段；发现后删除字段、schema/type 和 UI 读取路径，而不是清空值后保留。
- 队伍缩减为三人：只保留 Mike、Amy、Garry；删除 Lin Xia、Kael 及其专属事件、call template、fixture、地图引用和正式文档例子。
- 删除所有 legacy 内容和 legacy 命名路径，包括 legacy event、legacy action dispatch、地图 legacy 投影和 content/schema 中的 legacy 字段。
- 允许 UI 保留纯标签、空状态、调试说明和页面结构占位，但禁止虚构世界状态。
- 修改后的 docs、content、UI、测试都应表达同一个当前事实：没有 mock 机制、没有旧资产兼容、没有未实现内容伪装成游戏状态。

## 4. 非目标（Non-goals）

- 本轮不新增新的通讯、天线、信号质量、最近通话、脚步声、异常探测等机制来“解释”旧 mock 文案；没有机制支撑的内容直接删除。
- 本轮不保留 Lin Xia、Kael 的运行时内容，也不把他们的个人剧情改写给 Mike、Amy、Garry。
- 本轮不做旧存档、旧 content、旧 editor 资产的兼容迁移；依赖 legacy 字段或 legacy 资产形态的内容必须被清理或删除。
- 本轮不追求让页面保持原有信息密度；如果真实模型没有对应数据，页面可以显示中性空状态。

## 5. 目标用户与使用场景

### 5.1 用户画像

- **P1：项目设计者/开发者**。需要从 UI、content 和 docs 中准确判断当前游戏已经实现什么，避免被原型文案误导。
- **P2：后续内容作者或 agent**。需要知道哪些文本可以写进 content，哪些字段没有机制支撑必须删除。

### 5.2 典型场景

- **S1**：玩家打开通讯台，只看到真实来电、真实连接状态或中性空状态，不再看到虚构信号、天线偏移或硬编码求救。
- **S2**：开发者查看角色列表，只看到 Mike、Amy、Garry，以及有清晰语义归属的档案字段。
- **S3**：内容作者新增事件或地图对象时，不再看到 legacy event、legacy 字段或五人旧样例作为参考。

## 6. 用户旅程（To-be）

1. 用户进入控制中心或通讯台。
2. 系统从当前 `GameState` 和正式 content 资产读取可展示信息。
3. 如果存在真实来电、真实行动、真实事件或真实日志，UI 展示对应内容，并能追溯到状态或 content 来源。
4. 如果没有对应数据，UI 展示中性空状态或调试说明，例如“暂无来电”“未接入真实状态”“无可显示日志”。
5. 用户进入角色、地图、事件相关页面时，只看到 Mike、Amy、Garry，以及当前结构化模型支持的字段。
6. 内容作者或开发者查阅 schema、fixtures、docs 时，不再看到 legacy 字段、legacy event、Lin Xia、Kael 或 mock 机制样例。

### 6.1 失败路径与边界

- **F1**：如果某段 UI 文案无法追溯到 `GameState`、content 正式资产或中性 UI 说明，则视为失败，应删除或改为中性空状态。
- **F2**：如果 content 字段表达了不存在的机制，例如“最近通讯”但系统没有通话记录模型，则视为失败，应删除字段及其读取路径。
- **F3**：如果实现后仍有 legacy 命名字段、legacy event 或 legacy editor 展示入口，则视为失败，应继续清理。
- **F4**：如果页面因为真实数据不足而变空，这不是失败；只要空状态明确、中性、无虚构世界状态，即符合本轮目标。

## 7. 约束与假设

### 7.1 约束（Constraints）

- **C1：content 是运行时内容事实源**。角色、事件、地图、物品等游戏资产应从 `content` 进入运行时；代码只能保留 UI 标签、空状态、调试说明和结构性占位。
- **C2：UI 不能伪造游戏状态**。信号噪声、天线偏移、最近通讯、异常数量、脚步声、求救等世界状态，必须由真实模型或事件产生，否则不能展示。
- **C3：本轮不保留旧资产兼容**。删除 legacy 字段、legacy event、legacy editor 展示和旧队员内容时，不为旧存档或旧内容做适配层。
- **C4：正式文档必须同步**。`docs/game_model`、`docs/gameplay`、`AGENTS.md` 等当前事实文档不能继续描述五人队伍或 legacy 资产。

### 7.2 假设（Assumptions）

- **A1：当前原型可以接受信息密度下降**。删除 mock 后，部分页面会更空。验证方式：审核 UI 是否仍能清楚表达真实状态和可用操作。
- **A2：Lin Xia 与 Kael 相关内容不需要迁移**。验证方式：删除后运行内容校验、测试和文档审计，确认没有残留引用。
- **A3：所有 legacy 命名路径都能被替换或删除**。验证方式：实现阶段用全文搜索、schema 校验和测试确认没有 `legacy` 残留。

## 8. 方案选择

### 选择的方案：严格清理旧模型，保留中性 UI 占位

- **做法**：对 UI、代码、content、schema、fixtures、tests 和 docs 做一次一致性清理。删除 mock 机制暗示、无机制文本字段、五人旧内容和全部 legacy 命名路径；保留必要的中性 UI 说明。
- **优点**：玩家和开发者能直接从界面与数据判断当前系统真实能力，后续内容生产不会继续沿用旧样例。
- **缺点/风险**：删除面较大，短期内页面可能变空；去掉 legacy 地图投影和事件兼容层会触发较多测试与文档更新。
- **选择理由**：用户明确要求“多余内容应该被删掉而不是为了保持兼容性而留下”。因此，本轮不选择分阶段保留兼容层。

### 选择与理由（Decision）

- **采用**：严格清理 + 中性占位。所有世界状态必须有真实来源；没有机制支撑的文本字段删除；旧队员和 legacy 内容删除。
- **放弃**：保留 legacy event 只读展示、保留地图 legacy 投影、把 Lin Xia/Kael 剧情改写给三人、用新机制解释旧 mock 文案。

### 方案的比较

| 方案 | 优点 | 问题 | 结论 |
| --- | --- | --- | --- |
| 只删代码硬编码 | 工作量小 | content 里仍会保留误导性字段，如“最近一次通讯” | 不采用 |
| 分阶段删除 legacy | 风险较低 | 继续保留旧资产语义，违背本轮目标 | 不采用 |
| 严格清理 + 中性占位 | 当前事实最清楚 | 影响面大，需要同步测试和文档 | 采用 |

## 9. 核心对象/数据

- **角色资产（Crew）**
  - **来源/归属**：`content/crew/crew.json`。
  - **关键字段**：只保留三名角色及有清晰静态语义的档案字段；删除没有机制支撑的文本字段，例如容易伪装成动态记录的 `summary`。
  - **生命周期**：Lin Xia、Kael 及其引用在本轮删除，不归档到运行时内容。

- **事件资产（Events）**
  - **来源/归属**：结构化事件定义、call templates、presets 和 handler registry。
  - **关键字段**：删除 `content/events/events.json`、legacy event schema、legacy editor 类型和展示入口；删除依赖 Lin Xia、Kael 或 legacy 字段的事件资产。
  - **生命周期**：不迁移旧 legacy event；需要保留的机制样例必须以当前结构化事件重新定义，但本轮不默认改写旧剧情。

- **地图资产（Map）**
  - **来源/归属**：当前地图配置、地图对象 schema 和运行时 `GameState.map`。
  - **关键字段**：删除 `legacyResource`、`legacyBuilding`、`legacyInstrument`、`legacyDanger` 等 legacy 字段和 `deriveLegacyTiles` 兼容投影。
  - **生命周期**：地图 UI 与事件读取应面向当前地图模型，不再通过 legacy tile 投影表达事实。

- **UI 文案**
  - **来源/归属**：页面结构、真实状态、content 正式资产或中性占位。
  - **关键字段**：允许“暂无来电”“未选择队员”“未接入真实状态”等中性说明；禁止写死剧情、台词、假数值和世界状态。
  - **生命周期**：发现无法追溯来源的文案即删除或改为中性空状态。

## 10. 范围与阶段拆分

### 10.1 MVP（本次必须做）

- 审计并清理 PC UI、mobile UI、editor UI 中的 mock 世界状态、硬编码剧情、硬编码台词和假系统数值。
- 审计并清理 `content` 中没有机制或清晰语义归属的文本字段；删除字段定义、schema、类型和 UI 读取路径。
- 将队伍缩减为 Mike、Amy、Garry；删除 Lin Xia、Kael 的 content、事件、call template、manifest、fixtures、地图引用、测试引用和正式文档引用。
- 删除 legacy event、legacy event schema、legacy event runtime、legacy editor 类型/展示和相关测试。
- 删除 `legacy.<verb>` action dispatch、地图 legacy 投影和 content/schema 中所有 legacy 字段。
- 更新 docs，使当前 wiki 与代码事实一致。
- 运行内容校验、lint 和测试，确保删除后没有残留引用。

### 10.2 Later（未来可能做，但本轮不做）

- 设计真正的通讯记录模型、信号质量模型、天线校准模型或异常探测模型。
- 为三人团队补充新的正式剧情、事件和通话内容。
- 重新提高页面信息密度，但必须基于真实模型或正式内容资产。
- 如果未来需要旧资产导入，再单独设计一次显式迁移工具；本轮不做隐式兼容。

## 11. User Stories（MVP）

### US-001：清理 UI 中的 mock 世界状态

- **作为**：玩家或开发者
- **我想要**：页面只展示真实状态或中性空状态
- **以便**：我不会把原型文案误认为已实现机制
- **验收标准**：
  - [ ] 通讯台不再展示硬编码信号噪声、天线偏移、求救、脚步声或固定最近通讯。
  - [ ] 控制中心、移动端和 editor 不再展示“演示”性质的虚构世界状态。
  - [ ] 所有保留的占位文案都是中性的 UI 说明、空状态或调试说明。
- **不包含**：新增真实信号、天线或通讯记录机制。
- **优先级**：P0
- **依赖**：无机制文本审计规则。

### US-002：审计并删除无机制 content 文本字段

- **作为**：内容作者
- **我想要**：content 字段只表达真实资产语义
- **以便**：后续不会继续写入误导性 mock 文本
- **验收标准**：
  - [ ] `crew` 中类似“最近一次通讯”的字段被删除，而不是改写成另一段假状态。
  - [ ] schema、类型、初始状态和 UI 读取路径同步删除。
  - [ ] 内容校验能阻止已删除字段再次出现。
- **不包含**：批量创作新角色文案。
- **优先级**：P0
- **依赖**：content schema 更新。

### US-003：三人化队伍

- **作为**：玩家或开发者
- **我想要**：运行时、content、docs 和 tests 都只引用 Mike、Amy、Garry
- **以便**：当前原型聚焦三人团队
- **验收标准**：
  - [ ] Lin Xia、Kael 从 crew content 中删除。
  - [ ] 两人的事件、call templates、manifest、地图引用和 fixtures 被删除。
  - [ ] 正式 wiki、AGENTS 和测试不再把五人队伍当成当前事实。
- **不包含**：把两人的个人剧情改写给其他角色。
- **优先级**：P0
- **依赖**：事件资产清理。

### US-004：删除 legacy 事件与地图兼容层

- **作为**：开发者
- **我想要**：事件和地图系统只保留当前结构化模型
- **以便**：后续实现不再被旧资产兼容层牵引
- **验收标准**：
  - [ ] `content/events/events.json`、legacy schema、legacy runtime、legacy editor 类型和展示入口被删除。
  - [ ] `legacy.<verb>` action dispatch 被删除。
  - [ ] 地图 legacy 投影和 legacy 字段从 content、schema、代码和 docs 中删除。
  - [ ] 全仓搜索不再出现当前运行时或正式文档中的 `legacy` 命名路径。
- **不包含**：旧资产迁移工具。
- **优先级**：P0
- **依赖**：地图与事件模型的当前字段确认。

## 12. 成功标准（如何判断做对了）

- [ ] 用户打开游戏 UI 时，不再看到无法追溯到真实状态或正式内容资产的世界状态描述。
- [ ] `content` 中不存在无机制支撑的文本字段；相关 schema/type/UI 读取路径已删除。
- [ ] 队伍只剩 Mike、Amy、Garry；Lin Xia、Kael 不出现在运行时内容、测试 fixture 或正式当前文档中。
- [ ] legacy event、legacy dispatch、地图 legacy 投影和 legacy content/schema 字段全部删除。
- [ ] 全仓搜索 mock 示例词、旧角色 id 和 `legacy` 命名路径时，没有当前事实残留。
- [ ] `npm run validate:content` 通过；涉及客户端/editor/dual-device 的改动通过 `npm run lint` 和 `npm run test`。

## 13. 风险与缓解

- **R1：删除 legacy 地图投影影响事件读取**
  - **缓解**：先确认当前结构化地图模型的读取路径，再删除投影；测试应覆盖地图对象、行动和事件触发。
- **R2：删除字段后出现大量类型与测试失败**
  - **缓解**：把失败视为清理索引，逐项删除旧读取路径，而不是添加兼容 fallback。
- **R3：页面信息变少，用户误以为功能损坏**
  - **缓解**：使用中性空状态和调试说明解释“当前没有数据”或“该机制未接入”，但不补虚构世界状态。
- **R4：docs 与代码清理不同步**
  - **缓解**：把正式 wiki、AGENTS、schema 和测试作为同一批验收对象；实现完成后运行文档关键词审计。

## 14. 未决问题（Open Questions）

- 当前没有阻塞本轮设计的产品问题。
- 实现阶段需要逐文件确认：哪些地图字段是当前模型字段，哪些只是 legacy 投影字段。
- 实现阶段需要确认 editor 是否仍需要事件资产浏览能力；若需要，应只浏览当前结构化事件资产。
