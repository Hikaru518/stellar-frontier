## Q1：去 mock 的边界
问题：去 mock 的边界采用哪一种？推荐 A：严格事实源，UI 只能展示 GameState 真实状态、content JSON 中的正式资产、或中性 UI 标签/空状态；删除所有硬编码剧情句、假系统数值和假机制暗示。
选项：
- A. 严格事实源：UI 只能展示 GameState 真实状态、content JSON 中的正式资产、或中性 UI 标签/空状态；删除所有硬编码剧情句、假系统数值和假机制暗示。
- B. 保留少量装饰文案：允许无机制承诺的氛围文本，但必须明确标注为装饰/占位；风险是仍可能混淆。
- C. 建立白名单：逐条决定哪些 mock 文案删除、迁入 content、或保留；风险是规则更复杂，后续容易反复。
用户回答：B。保留少量装饰文案。

## Q2：legacy 删除范围
问题：本轮“不要 legacy 内容”应覆盖到什么范围？推荐 B：先完整删除 legacy event 与旧资产展示，再把地图 legacy 投影作为单独技术债处理。
选项：
- A. 只删除 legacy event：删除 content/events/events.json、旧 eventSystem、legacy_event 编辑器展示和相关 schema/test/doc。
- B. 删除 legacy event，并清理 legacy.<verb> action dispatch；地图 deriveLegacyTiles/legacyResource 等投影暂不作为本轮主目标，但要在设计里明确为后续债务。
- C. 所有 legacy 命名一并删除：包括地图 legacy 投影、legacyResource/legacyBuilding/legacyDanger 等 content/schema 字段；范围最大，风险也最大。
用户回答：C。所有 legacy 命名一并删除。

## Q4：正式 content 文本展示规则
问题背景：解释选项 B：content JSON 里的正式文本可以保留，但 UI 必须按它的真实语义展示。角色背景/性格/专长说明只放角色档案；事件台词/求救/通话文本只在事件或通话真实触发后显示；日志文本只在效果真实结算后写入日志；像“最近一次通讯：非常不礼貌的求救”这种容易被误读成动态机制的 summary，要改成静态档案口吻或移到真实通话记录里。
用户追问：这不就会变成还是会存在硬编码的内容吗？
回答：这句话不应该以硬编码形式继续存在。“改成静态档案口吻”不是把它搬到代码里，而是指如果确实需要角色摘要，也只能作为 `content/crew/crew.json` 里的正式档案字段，并且内容不能暗示一次已经发生的动态通讯。
用户追问：是 A 更严格还是 B 更严格呢？
回答：B 更严格。A 只管“不要在代码里硬编码剧情/台词”，但只要文本在 content 里就允许展示；B 还会继续限制 content 文本的展示位置和含义。
用户追问：content 中哪里有“最近一次通讯”这样的内容呢？
回答：在 `content/crew/crew.json` 的 Amy 档案里，`summary` 字段现在是 `最近一次通讯：非常不礼貌的求救。`。
用户回答：B。而且希望进行审计，需要找到这种没有机制的文本（即使在 content 中），如果有这种情况出现，需要删掉这个字段。例如 crew 的最近通话事实上没有机制，需要删掉。

## Q5：无机制文本字段处理
问题：审计发现“没有机制支撑的文本字段”时，应如何处理？推荐 A：删除字段及 schema/type/UI 读取路径。
选项：
- A. 删除字段及 schema/type/UI 读取路径：如果字段本身没有清晰机制或语义归属，就从 content、schema、类型、UI 中一起删除。
- B. 清空或删除现有值，但暂时保留字段定义；风险是后续又被写入类似 mock 文本。
- C. 只有当字段有明确真实归属时才迁移，例如移动到真实通话记录/事件资产；否则删除字段及读取路径。
用户回答：A。删除字段及 schema/type/UI 读取路径。

## Q6：允许的占位文案边界
问题：在 UI 里允许的“装饰/占位文案”应限定到什么程度？推荐 B：保留必要 UI 引导，但禁止任何虚构世界状态。
选项：
- A. 只允许纯 UI 标签和空状态，例如“暂无来电”“未选择队员”“功能未接入”。
- B. 允许纯 UI 标签、空状态、调试说明和明确的页面结构占位；不允许虚构信号、天线、异常、求救、脚步声等世界状态。
- C. 允许少量明确标注为“占位/装饰”的氛围文案；但不能被用于状态栏、日志、角色摘要或机制反馈。
用户回答：B。允许纯 UI 标签、空状态、调试说明和明确的页面结构占位；不允许虚构世界状态。
