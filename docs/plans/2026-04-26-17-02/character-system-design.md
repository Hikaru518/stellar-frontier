# 人物系统设计

## 1. 概述（What & Why）

人物系统定义队员作为“真实伙伴”的个体差异：每名队员都有独立背景、说话语气、个人日记、性格标签、专长标签与 5 维轻量能力属性。  
能力属性用于轻量表达人物能力倾向；性格由自由文本标签承载，只用于展示与写作依据，不直接参与规则判定。  
系统允许未来队员数量增加时，玩家自行分配注意力，但仍保证每个队员都有可被识别的个性与世界位置。  
当前验证集合覆盖现有 3 名队员与新增 2 名队员，用于检验角色差异、世界观承载与人物详情入口的可读性。

## 2. 设计意图（Design Intent）

人物系统的核心情绪是“这些人不是工具，而是同行的真实伙伴”。每名队员都有独立来历、表达方式、记录习惯与自我标签；他们可以讨喜，也可以固执、别扭或不符合玩家偏好。系统承认这种差异，并把它作为角色可信度的一部分。

目标体验是：玩家在查看队员时，先感受到“这是一个具体的人”，再理解他适合做什么。背景档案、通话语气、个人日记、性格标签与专长标签共同提供可见线索，让玩家通过少量明确信息产生脑补，而不是只阅读一张效率表。

系统避免把队员压扁成可替换劳动力。能力属性与专长标签用于支持区分，但不能吞掉个性；自由性格标签只作为展示与写作依据，不作为硬规则条件。角色台词、日记与标签需要共同维护差异，避免所有人说话像同一个人。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **队员（Crew Member）**：人物系统的基本单位；每名队员被视为真实伙伴，而不是可替换数值单位。

- **背景档案（Profile）**：由模板骨架与手写短句组成，记录原世界/出身、原职业、经历、性格、一句话自我介绍等信息。

- **通话语气（Voice & Tone）**：队员在通讯、事件反馈与对话中的表达差异，用于体现人物个性与关系感。

- **个人日记（Personal Log）**：队员在关键节点留下的记录，可在人物详情中查看，用于补充其主观经历与情绪变化。

- **日记可见性（Log Availability）**：通讯正常时可查看已传回日记；失联后只能看到此前传回内容；找回本人/遗体后可看到失联后的日记。

- **5 维轻量能力属性（Attributes）**：能力属性为体能、敏捷、智力、感知、运气；主要体现人物能力倾向，不在本节承诺复杂公式。

- **自由性格标签（Personality Tags）**：自由文本短标签，用于展示与写作依据；不直接参与规则判定。

- **专长标签（Expertise Tags）**：体现人物独特能力或叙事身份的标签，数量不要求固定，可具有规则效果。

- **专长效果（Expertise Effect）**：专长在特定情境中提供差异化结果，例如“拾荒者”在调查时更容易发现额外物资。

- **验证角色集合（Validation Cast）**：用于验证人物差异的初始角色组，包括现有 Mike、Amy、Garry 与新增 2 名队员，重点覆盖不同世界、背景与个性表达。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家在通讯台看到 5 名队员的头像、姓名、专长标签与当前通讯状态。
2. 玩家打开人物详情，先通过背景档案、自由性格标签与 5 维轻量能力属性形成第一印象。
3. 玩家接通队员通话，从语气、用词和回应节奏感知其性格差异。
4. 行动结束后，系统在关键节点生成个人日记，玩家在人物详情中阅读队员的主观记录。
5. 玩家反复对照“他说的话”和“他写下的事”，逐渐把队员当成具体的人，而不是数值单位。
6. 若队员失联，玩家只能查看此前已传回的日记，信息突然中断。
7. 找回本人或遗体后，玩家解锁失联期间的日记，补完这段经历。

### 4.2 典型情境（可选）

高光时刻：玩家同时查看 Mike、Amy、Garry 与新增 2 名队员时，能从档案、标签、通话语气和日记口吻上一眼区分 5 个人。

低谷 / 摩擦点：某名队员失联后，详情页保留他的既有记录，但新日记空缺。玩家等待找回结果，之后才读到失联期间发生了什么。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

本节不新增复杂人物状态机，仅定义日记可见性与通讯状态的关系。

日记条目按可见性分为：`已传回`、`未传回`、`失联锁定`、`找回解锁`。队员通讯正常时，玩家可查看所有已传回日记；新生成但尚未传回的条目不可见。队员失联后，日记列表锁定在失联前已传回的内容，失联期间生成的条目标记为未传回 / 失联锁定，不直接展示。若玩家找回本人或遗体，失联后的日记解锁为可查看内容，用于补足其经历与情绪变化。

### 5.2 规则与公式

角色由背景档案、能力属性、自由性格标签、专长标签与关键节点日记组成。

能力属性采用 5 维轻量表达：体能、敏捷、智力、感知、运气。属性主要用于表达角色能力倾向，可作为事件判定、文本分支或任务适配的参考，但本节不承诺复杂公式。

自由性格标签为展示与写作依据，例如“谨慎”“嘴硬”“怀旧”，不直接参与规则判定。

专长标签不设等级、不固定数量，可带规则效果。每个专长的触发条件与结果由内容手工定义，例如“拾荒者：调查废弃地点时，更容易发现额外物资”。

背景档案采用模板骨架加手写短句，包括原世界 / 出身、原职业、经历、性格、一句话自我介绍等字段。

日记由关键节点触发生成，每个触发点默认生成 1 条短文本，并根据通讯状态决定是否传回与可见。

背包负重、关系 / 士气、角色成长不属于本节规则。

### 5.3 参数与默认值

能力属性数值范围暂定为 `1-6`，用于轻量表达人物能力倾向，而不是构建复杂 RPG 数值体系。数值含义建议为：`1` 明显短板，`2` 偏弱，`3` 普通，`4` 擅长，`5` 突出，`6` 极少数人的标志性强项。

每名角色默认配置 `3-5` 个自由性格标签。专长标签建议 `1-3` 个作为内容规范，不作为硬限制。关键节点日记默认每个触发点 `1` 条短文本。

## 6. 系统交互（System Interactions）

- **依赖于**
  - `crew` 队员对象：承载人物身份、当前状态与详情入口。
  - `content JSON`：提供背景档案、通话语气、日记文本、自由性格标签、专长标签与 5 维轻量能力属性。
  - 通讯状态：决定人物详情、日记与通话内容是否可见。

- **被依赖于**
  - 人物详情页：展示档案、标签、能力、日记与可见性状态。
  - 通讯台与通话页：读取通话语气、可通讯状态与人物文本。
  - 调查行动与事件系统：读取专长标签与能力属性，触发手工定义的规则效果。
  - 调试菜单可用于验证时间相关日记 / 通讯表现，但不改变人物规则。

- **共享对象 / 状态**
  - 队员 ID、通讯 / 失联 / 找回状态、已解锁日记节点、专长规则效果、能力属性值。

- **事件 / 信号**
  - 通讯状态变化、日记关键节点解锁、人物详情查看、调查判定请求、事件效果结算。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：首次查看队员**：玩家打开人物列表 → 查看 Mike、Amy、Garry 与新增 2 名队员的档案、语气、标签、专长、5 维属性 → 5 人在背景、说话方式与能力侧重点上一眼可分辨。
- **S2：通话 / 行动后日记生成**：关键通话或行动结算后 → 系统为相关队员生成个人日记 → 日记内容的措辞、关注点与其性格标签、通话语气保持一致。
- **S3：专长影响调查**：调查场景触发队员参与 → 若队员拥有如「拾荒者」等手工定义专长 → 结果描述或可发现线索出现差异化反馈。

### 7.2 边界 / 失败场景

- **F1：队员失联**：队员进入失联状态 → 玩家查看该队员日记 → 新日记不继续可见，避免通过日记泄露失联后的信息。
- **F2：自由性格标签存在**：系统读取角色标签 → 仅用于展示与文本风格参考 → 不把自由文本标签当作规则条件参与判定。
- **F3：角色内容缺失**：某角色缺少背景、日记或标签 → 玩家进入人物详情 → 明确暴露内容缺口，而不是静默显示空白。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- 选了真实伙伴感而非纯工具 / 劳动力，因为角色需要让玩家在探索中产生牵挂，而不是只作为效率插件。反模式：把角色写成可替换的数值表。
- 选了轻量 5 维 `1-6` 属性而非复杂 RPG 数值体系，因为本系统服务于叙事判断与角色差异，不追求构筑深度。反模式：围绕属性堆公式、阈值和最优解。
- 选了专长而非领域技能 / 细分技能表，因为专长更像角色记忆点，建议 `1-3` 个即可。反模式：专长太多，最后变成配装和构筑游戏。
- 选了自由性格标签而非结构化 / 混合标签，因为标签主要帮助展示与写作，不直接参与规则判定。反模式：标签写成同义词堆叠，或把自由标签写进硬判定。
- 选了关键节点日记而非每日自动流水账，因为日记应记录关系转折和重大经历。反模式：所有人通话 / 日记都用同一种声音，失去角色差异。
- 选了模板骨架 + 手写短句而非完全手写或程序生成，因为它能控制成本并保留人味。关系 / 士气、成长、程序生成、背包负重放入 Later / 不做，避免本轮失焦。

## 10. 本轮范围与阶段拆分（Scope & Phasing for This Round）

### 10.1 MVP（本轮必做）

- 覆盖 5 名角色：保留并补齐 Mike / Amy / Garry，新增 2 名角色，用于验证不同世界背景与个性差异是否可被玩家感知。
- 每名角色必须具备：背景档案、通话语气、个人日记、`3-5` 个自由性格标签、`1-3` 个专长标签、5 维轻量属性。
- 背景采用“模板骨架 + 手写短句”，可在人物详情中查看。
- 日记只做关键节点日记；在人物详情查看，并根据通讯、失联、找回状态控制可见性。
- 属性为 5 维，每维取值 `1-6`；本轮只展示与写作参考，不参与复杂规则判定。
- Debug toolbox 是本轮调试 / 作弊辅助项，不是人物系统正式规则；仅包含时间倍率 `1x` / `2x` / `4x` / `8x` 与重置存档。

### 10.2 Later（未来再做，明确本轮不做）

- 背包负重容量与背包规则改动。
- 性格标签、专长标签、属性参与自动规则判定。
- 角色成长、升级、训练或属性变化。
- 关系系统、士气系统、好感度或团队氛围联动。
- 程序生成角色、随机背景或随机性格组合。

### 10.3 不做（Out of Scope）

- 不把 Debug toolbox 设计成人物系统的一部分，也不要求其影响角色状态。
- Mike 到达后不能调查、调查结束似乎无事件的问题，作为独立技术事项处理，不纳入本策划 MVP。

## 11. 本轮验收与风险（Acceptance & Risks）

### 11.1 Player Stories / Play Scenarios

**PS-001：区分 5 个队员**  
作为玩家，我能通过姓名、头像 / 代号、背景摘要、专长与日记语气区分 5 个队员，以便在调查与事件选择中形成基本人物记忆。  
验收标准：5 名队员在列表与详情页均可识别；每人至少有 1 个清晰差异点；玩家不需要依赖数值成长来区分角色。  
不包含：背包、成长、关系 / 士气、程序生成角色。  
优先级：P0。

**PS-002：查看关键节点日记与失联可见性**  
作为玩家，我能在人物详情中查看关键节点日记，并在队员失联后看到符合规则的可见 / 不可见状态，以便理解角色经历与当前信息缺口。  
验收标准：日记入口清晰；关键节点日记可按时间或阶段浏览；失联角色不会继续暴露不应被玩家知道的新内容。  
不包含：无限制日记生成、关系驱动日记。  
优先级：P0。

**PS-003：专长影响调查 / 事件结果**  
作为玩家，我能通过队员专长看到调查或事件结果差异，例如拾荒者带来额外物资线索或不同描述，以便理解“派谁去”有轻量但可感知的意义。  
验收标准：至少 1 个专长在事件 / 调查结果中产生可观察差异；差异以文本或结果提示体现；专长不形成复杂构筑或长期养成路线。  
不包含：职业树、装备搭配、数值 build。  
优先级：P1。

**PS-004：Debug toolbox 辅助验收**  
作为测试者，我能使用游戏内 Debug toolbox 调整时间倍率 `1x` / `2x` / `4x` / `8x` 与重置存档，以便快速验证人物系统在不同时间推进下的表现。  
验收标准：工具入口明确标注 Debug / 作弊；仅包含时间倍率与重置存档；不被包装为正式玩法。  
不包含：人物系统正式规则、平衡性承诺。  
优先级：P2（辅助验收）。

### 11.2 成功标准

- [ ] 玩家能在一次浏览后区分 5 个队员的身份与核心差异。
- [ ] 人物详情页能展示基础信息、专长、背景摘要与关键节点日记。
- [ ] 失联状态下，人物日记与信息可见性符合已定义规则。
- [ ] 至少一个专长能在调查 / 事件结果中产生可观察差异。
- [ ] Debug toolbox 可用于 `1x` / `2x` / `4x` / `8x` 时间倍率与重置存档测试。
- [ ] 本轮不引入背包、成长、关系 / 士气、程序生成系统。

### 11.3 风险与缓解

- 角色文本同质化：用“身份标签 + 专长 + 日记语气 + 关键经历”建立差异，先保证 5 人可识别。
- 自由标签无法被系统使用：标签仅作为展示与写作辅助；可被规则引用的内容收敛到专长与状态。
- 专长过多变成构筑系统：本轮限制为轻量结果差异，不做职业树、成长或装备联动。
- 日记内容成本过高：只写关键节点日记，不做高频动态文本。
- Debug toolbox 被误解为正式玩法：界面明确标注 Debug / 作弊，文档中列为辅助验收。
- Bug 修复影响体验：相关 bug 作为独立技术事项跟进，不纳入人物系统规则范围。

## 12. Open Questions

1. **实施前确认**：现有 6 个能力字段将迁移为新的 5 维轻量属性「体能 / 敏捷 / 智力 / 感知 / 运气」，不保留并行的旧属性体系。实施时需要定义旧数据到新属性的具体数值转换规则。

2. **内容设计时定义**：新增 2 名角色的具体姓名、来源世界、职业与人格短句尚未定义。本策划只确认其用途：验证不同世界背景与个性差异。

3. **内容设计时定义**：每个专长的具体触发条件、效果数值与文案，需要在后续内容设计中逐条定义。

4. **实施前确认**：Debug toolbox 作为游戏内作弊菜单，其入口位置与是否需要二次确认弹窗，可在 UI 实施时最终决定。

5. **后续技术事项**：Mike 调查入口、调查事件反馈由后续技术任务处理，不在本策划内解决；背包、成长、关系、程序生成也不属于本轮范围。
