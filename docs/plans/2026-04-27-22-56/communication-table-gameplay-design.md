# 通讯台 Gameplay

## 1. 概述（What & Why）

通讯台 gameplay 是玩家与队员、地图、事件系统之间的主要行动入口。玩家通过主动通话给队员下达调查、移动、原地休息和对象交互等基础指令；队员在行动、抵达、调查完成或遭遇事件后，也会以事件来电的形式把世界变化反馈给玩家。这个系统的目标是把地图探索、队员行动、地块对象和事件触发串成一个可在网页上验证的基础 gameplay 闭环。

## 2. 设计意图（Design Intent）

通讯台要让玩家感觉自己是在远程指挥一支真实行动中的探索队，而不是在菜单里切换角色状态。日常体验的重点是“打电话 → 了解队员处境 → 下达一个清楚的行动 → 在地图和状态中看到行动发生 → 行动结果反过来触发报告或事件”。

这个系统应优先制造三种体验：第一，玩家能用通话稳定推进探索；第二，地图上的未知对象会因为调查和行动逐步变成可互动内容；第三，事件来电会把世界的变化重新拉回通讯台。需要避免的体验是：地图直接变成操作面板、事件只写日志不改变 play、不同队员依赖各自硬编码按钮。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **普通通话**：玩家主动联系队员的通话入口，用于查看队员当前处境并下达基础行动指令。通讯台按钮显示为“通话”。
- **事件来电**：由事件系统产生的队员主动联系玩家的入口。普通事件和紧急事件都可以显示为“接通”，但通过标签、阻塞状态和倒计时区分严重度。
- **基础行动**：所有队员都可用的通用行动，包括调查当前区域、去往另一个区域、原地休息，以及由已揭示对象提供的采集、采矿、扫描等对象行动。
- **地块对象**：地图地块上的可发现内容。对象在未调查前可能隐藏；被发现或调查揭示后，才会为普通通话提供额外动态按钮。
- **原地休息**：停止当前行动并进入待命；如果队员已经待命，则刷新或记录待命开始时间，供 `idle_time` 类事件读取。
- **简单携带物**：对象交互获得的轻量物品列表，挂在队员身上；本轮不包含重量、容量、交付回基地或完整背包规则。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家从通讯台选择一名队员，点击“通话”主动联系。
2. 通话页显示队员当前位置、当前行动状态、地块已知信息，以及可用行动。
3. 若队员空闲，玩家可选择调查当前区域、去往另一个区域、原地休息，或点击已揭示地块对象生成的动态按钮。
4. 若玩家选择移动，通话页进入地图选点流程；地图只展示候选目的地，最终派遣仍在通话语境中确认。
5. 队员开始真实行动，通讯台、地图和队员状态展示行动进度。
6. 行动完成后，系统结算位置、调查披露、携带物或地块对象状态，并发出 `arrival`、`action_complete` 或 `idle_time` 等事实信号。
7. 事件系统根据事实信号判断是否生成事件来电；玩家从通讯台“接通”并处理普通或紧急事件。
8. 事件选项若要求后续行动，应桥接回同一套真实行动，而不是只写日志。

### 4.2 典型情境

- **高光时刻**：玩家让 Garry 调查矿床并执行采矿，采矿完成后获得简单携带物，同时触发矿床异常来电，玩家能看到对象交互和事件反馈连接在一起。
- **低谷 / 摩擦点**：队员正在移动或调查时，玩家不能直接追加新任务，只能查看当前行动并选择停止/原地休息，避免行动并行造成状态混乱。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

- 队员普通状态至少包含：待命、移动中、调查中、对象交互中、事件阻塞中。
- 队员同一时间只能执行一个主行动；移动、调查、采集、采矿、事件处理互斥。
- 玩家主动通话时，如果队员待命，则显示基础行动和可用对象行动；如果队员忙碌，则显示当前行动状态，并只提供停止/原地休息。
- 普通事件来电不阻塞队员行动；紧急事件来电会使相关队员进入事件阻塞中，并要求玩家在倒计时或有效期内处理。

### 5.2 规则与公式

- 普通通话的基础行动由 App 层通用行动生成负责，不按角色硬编码。
- “调查当前区域”创建 `survey` 行动；完成后揭示当前地块的 `onInvestigated` 对象/状态，并发出 `action_complete`。
- “去往另一个区域”进入地图选点；地图只读，最终移动必须回到通话流程确认。
- “原地休息”中断当前行动并进入待命；若已待命，则刷新待命开始时间，可供 `idle_time` 触发。
- 已揭示地块对象的 `candidateActions` 会直接映射成动态按钮，例如“采集木材”“采矿”“扫描残骸”。
- 对象交互完成后，把产物加入队员简单携带物列表，并可更新地块对象状态或触发 `action_complete`。
- 事件系统监听 `arrival`、`action_complete`、`idle_time`、`call_choice` 等事实信号；事件选项如果要求行动，必须桥接到真实行动创建，而不是仅写日志。

### 5.3 参数与默认值

- 调查行动默认耗时：沿用现有行动系统默认值，本轮不新增复杂公式。
- 对象交互默认耗时：MVP 可按对象配置或行动类型给出短时长，优先保证网页可验证。
- 停止/原地休息：应立即或按现有停止动作规则生效；本轮不新增疲劳恢复数值。
- 简单携带物：无重量、无容量、无自动交付基地规则。

## 6. 系统交互（System Interactions）

- **依赖于**：通讯台与通话页负责入口和行动确认；地图系统提供队员位置、地块已知信息、对象可见性和移动目标；队员行动系统负责移动、调查、对象交互、停止和待命；事件系统负责监听行动事实并生成来电。
- **被依赖于**：事件内容依赖此系统把选项落成真实行动；地图探索依赖此系统推动地块调查和对象交互；队员详情和通讯台列表依赖此系统展示当前行动、简单携带物和来电状态。
- **共享对象 / 状态**：`GameState` 中的队员位置、队员 `activeAction`、地图 runtime state、地块对象可见性、事件 runtime call、队员简单携带物。
- **事件 / 信号**：移动抵达发出 `arrival`；调查、采集、采矿、扫描等完成发出 `action_complete`；原地休息/待命可产生 `idle_time`；玩家在事件通话中选择选项产生 `call_choice`。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1：主动调查地块**：队员待命在某地块 → 玩家主动通话并选择“调查当前区域” → 队员进入调查行动，完成后揭示隐藏对象，地图详情和通话选项发生变化。
- **S2：通话派遣移动**：队员待命 → 玩家通话中选择“去往另一个区域” → 地图进入选点模式 → 玩家确认目的地 → 队员按路线移动，并在抵达后触发 `arrival`。
- **S3：对象交互**：地块对象已揭示且声明 `candidateActions` → 通话页显示“采矿/采集/扫描”等动态按钮 → 队员执行对象行动 → 完成后获得简单携带物或更新对象状态，并触发事件检测。
- **S4：事件来电**：行动完成或抵达触发事件 → 通讯台显示“接通”入口和普通/紧急标签 → 玩家进入通话处理事件选项 → 选项结果写入日志、状态或桥接真实行动。
- **S5：紧急事件处理**：Amy 在森林触发野兽事件 → 队员进入事件阻塞中并出现倒计时 → 玩家接通并选择应对方案 → 事件结果解除阻塞、创建后续行动或造成后果。

### 7.2 边界 / 失败场景

- **F1：队员忙碌中**：队员正在移动、调查或采集 → 玩家主动通话 → 页面只展示当前行动和停止/原地休息，避免直接叠加新行动。
- **F2：地块对象未揭示**：地块存在隐藏对象但未调查 → 普通通话不显示对象交互按钮，避免提前泄露信息。
- **F3：普通事件未处理**：普通事件来电出现后，队员行动不被阻塞；玩家可稍后接通，但事件不应丢失。
- **F4：紧急事件超时**：紧急来电在有效期内未处理 → 系统按事件定义结算失败或自动后果，并将结果反馈到队员/地图/日志。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- **取舍 1**：选择 App 层通用行动生成，而不是把所有普通动作都事件图化。理由：调查、移动、休息和对象交互是高频基础操作，应先保证全员一致、链路短、可验证；事件系统负责监听结果和升级反馈。
- **取舍 2**：选择已揭示对象动态按钮，而不是本轮先做对象子菜单。理由：MVP 地块对象数量有限，直接按钮更容易让玩家看见“调查后解锁交互”的变化。
- **取舍 3**：选择简单携带物，而不是完整背包/负重/交付链。理由：对象交互需要有获得感，但完整背包会把本轮目标从通讯 gameplay 扩展成资源运输系统。
- **取舍 4**：选择所有事件都可表现为“接通”，而不是只有紧急事件进入通话。理由：本轮希望事件反馈回到通讯台；普通/紧急的差异由标签、阻塞状态和倒计时承担。
- **要避免的反模式**：继续为 Mike、Amy、Garry 分别写死通话按钮和地块 id；地图上直接下达指令；事件选项只追加日志但不改变队员行动、地块状态或可见信息；忙碌队员自动叠加新行动。

## 10. 本轮范围与阶段拆分（Scope & Phasing for This Round）

### 10.1 MVP（本轮必做）

- 所有队员在普通通话中拥有统一的基础行动：调查当前区域、去往另一个区域、原地休息。
- 队员待命时展示完整普通行动；队员忙碌时只展示当前行动状态和停止/原地休息。
- 调查行动能揭示当前地块的隐藏对象或状态，并让后续通话出现对应对象行动。
- 已揭示对象的 `candidateActions` 能映射成动态按钮，至少支持采集/采矿类对象交互。
- 对象交互完成后产出简单携带物，并能触发事件检测。
- 所有事件都能以“接通”形式回到通讯台；普通/紧急通过标签、阻塞状态和倒计时区分。
- Mike、Amy、Garry 各有一个稳定可复现事件：Mike 调查坠毁区发现残骸线索；Amy 在森林触发野兽紧急来电；Garry 采矿触发矿床异常报告。
- 事件选项如果表达移动、调查、等待或处理目标，必须桥接到真实队员行动或地块状态变化。

### 10.2 Later（未来再做，明确本轮不做）

- 对象交互数量增加后的“与当前地块对象互动”子菜单。
- 更复杂的事件严重度层级、未读消息箱、普通报告队列。
- 完整资源运输链：携带物带回基地、转换为基地资源、交付失败或丢失。
- 队员疲劳、恢复、休息收益、长期待命风险。
- 角色差异化行动文案、专长影响行动效率或事件选项。

### 10.3 不做（Out of Scope，避免范围膨胀）

- 完整背包、负重、容量、丢弃和交付资源链。
- 关系、士气、好感度或角色成长系统。
- 地图上直接下达指令。
- 离线时间推进或离线行动结算。
- 可视化事件编辑器或复杂脚本语言。

## 11. 本轮验收与风险（Acceptance & Risks）

### 11.1 Player Stories / Play Scenarios（验收切片）

#### PS-001: 全员普通通话基础行动
- **作为**：玩家
- **我能**：主动联系任意队员，并看到调查当前区域、去往另一个区域、原地休息等基础行动
- **以便**：不依赖角色硬编码也能推进日常探索
- **验收标准**：
  - [ ] 任意队员待命时，普通通话都显示三类基础行动。
  - [ ] 任意队员忙碌时，普通通话显示当前行动并只允许停止/原地休息。
  - [ ] 地图仍只用于查看和选点，不直接下达指令。
- **不包含**：角色专属技能、关系/士气影响、地图直接操作。
- **优先级**：P0

#### PS-002: 调查揭示对象并解锁交互
- **作为**：玩家
- **我能**：让队员调查当前地块，调查完成后看到隐藏对象被揭示，并在后续通话中出现对象行动按钮
- **以便**：理解调查如何把地图未知信息变成可操作内容
- **验收标准**：
  - [ ] 调查前不显示隐藏对象交互。
  - [ ] 调查完成后地图详情展示新对象/状态。
  - [ ] 已揭示对象的 `candidateActions` 生成动态按钮。
- **不包含**：对象子菜单、复杂对象编辑器。
- **优先级**：P0

#### PS-003: 三个稳定事件切片
- **作为**：玩家
- **我能**：通过 Mike、Amy、Garry 分别触发一个可复现事件
- **以便**：验证普通行动、紧急来电、对象交互都能接入事件系统
- **验收标准**：
  - [ ] Mike 调查坠毁区后触发残骸线索事件。
  - [ ] Amy 在森林触发野兽紧急事件，队员阻塞并出现倒计时/必须处理。
  - [ ] Garry 采矿后触发矿床异常报告，并保留采矿所得简单携带物。
- **不包含**：大规模随机事件、复杂剧情树。
- **优先级**：P0

### 11.2 成功标准（Success Criteria）

- [ ] 玩家能仅通过通讯台和通话页完成“联系队员 → 下达行动 → 看到行动推进 → 获得行动结果 → 接通事件”的闭环。
- [ ] 普通行动生成不再依赖 Mike/Amy/Garry 的角色硬编码分支。
- [ ] 事件选项的结果能改变真实行动、队员状态、地块状态、对象可见性或简单携带物。
- [ ] 普通事件和紧急事件都能在通讯台以“接通”出现，并通过标签/阻塞/倒计时表达差异。
- [ ] 本轮 MVP 可在网页上用前三名队员稳定复现。

### 11.3 风险与缓解（Risks & Mitigations）

- **R1**：所有事件都显示“接通”可能削弱紧急事件的特殊感。
  - **缓解**：紧急事件必须额外具备阻塞、倒计时和明确标签，普通事件不阻塞。
- **R2**：动态对象按钮后续可能膨胀。
  - **缓解**：MVP 限制对象数量；Later 再引入对象互动子菜单。
- **R3**：简单携带物可能被误解为完整背包。
  - **缓解**：文案和 UI 明确“携带物记录”而非背包系统，不显示重量/容量/交付。
- **R4**：事件选项桥接真实行动时可能与队员忙碌状态冲突。
  - **缓解**：事件创建行动前必须检查队员状态；紧急事件可先阻塞并由选项明确解除或替换当前行动。

## 12. Open Questions

- **Q1**：普通事件来电如果玩家长期不接通，是否需要过期、自动归档或一直保留？本轮可先保留，但实施时需要决定 UI 行为。
- **Q2**：紧急事件倒计时的默认时长、失败后果和暂停/离开页面时的表现，需要由具体事件内容定义还是系统给默认值？
- **Q3**：简单携带物展示在哪些页面：通讯台队员卡、队员详情、通话页，还是只在队员详情展示？
- **Q4**：事件选项桥接真实行动时，如果队员已经有行动，是否允许事件强制替换，还是必须先让玩家确认停止当前行动？
- **Q5**：Mike、Amy、Garry 的三个稳定事件是否允许重复触发，还是每个存档只触发一次？
