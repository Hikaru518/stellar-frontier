
# 通讯台的玩法实现

现在已经有了 event 机制，也有了地图，也有了人物的基本内容。在这次迭代中，我期望可以把这一切都串联成一个基本的 gameplay。

要完成下面的一些硬约束。

## UI


## 玩家主动通话

1. 玩家主动找角色通话，而不是角色遇到事件通话。此时 UI 显示的时通话，而不是接通。
2. 角色在一个地方，一定有下面的选项
   - 调查当前区域
   - 去往另一个区域
   - 原地休息
3. 在 2 的基础之上，当地图上存在对象的时候可以有别的选项。
4. 为以后的扩展提供程序可能性

### 调查当前区域

调查当前区域应当是和地块绑定的，会影响到当前的地块的信息披露。
一开始，我们只能知道一个地方的地形和天气。但是随着调查我们可以知道这个地方有什么对象。

调查前：
- 区域 坠毁区域
- 地形 平原
- 天气 阴天

调查后：
- 坠毁残骸

### 去往另一个区域

当角色不在紧急事件中的时候（例如遇到了熊），此时玩家可以让角色去往另一个区域。

完成派遣之后，需要真的看到角色在按照路线移动，且最终移动到了那个区域。

### 原地休息

当角色不在紧急事件中的时候（例如遇到了熊），此时玩家可以让角色原地休息。

例如，角色可能原本在移动，那么此时玩家可以让角色休息，此时角色会停下来，进入休息状态。

## 角色 call 玩家

只有角色在紧急情况的时候才会 call 玩家的时候，这不是一个普通的通话。会根据 event 来触发。

但是这个 event 的结果必须要呈现在真实的 play 中。例如有一个事件，举例 Mike 看到了远处的火山喷发，如果决定要让他过去看看，那么他会开始移动，然后最终移动到火山附近。

## 在网页上可玩

上述的这些我们需要一个 MVP，我需要：

1. 前三个队员（Mike，Amy，Garry需要明确的事件），你可以移动他们的初始位置和改变地图来满足这些条件。
2. 所有的队员都要满足上面的条件，例如任意一个角色移动到了地块，都可以满足上面的交互逻辑。

## 实现计划

### 如何使用当前系统的实现（或者需要改动的内容）

当前系统已经有一部分可复用的基础能力，但还不能直接满足上面的完整 MVP。核心判断是：**地图、队员移动、调查披露、事件 runtime call 的底层能力已经存在；普通通话里的行动选项和行动结算还没有通用化，目前仍然大量依赖 Mike / Amy / Garry 的硬编码分支。**

#### 当前可以直接复用的内容

1. **通讯台与通话入口**
   - `CommunicationStation` 已经支持从通讯录主动联系队员。
   - `CommunicationStation` 也能展示 `active_calls` 里的事件通话，并通过“接通”进入 `CallPage`。
   - 因此“玩家主动找角色通话”和“角色因事件 call 玩家”这两个入口都已有 UI 基础。

2. **通话中选择移动目标**
   - `CallPage` 已经支持从通话进入地图选点。
   - `MapPage` 在选点模式下只标记候选目的地，不直接下达指令，符合“地图只读、指令必须经通话确认”的设计。
   - `createMovePreview` / `startCrewMove` / `advanceCrewMovement` 已经能计算路线、显示预计耗时、逐格推进，并在抵达后把队员放回待命状态。

3. **地图发现与调查披露**
   - `content/maps/default-map.json` 已经给地块配置了地形、天气、对象、特殊状态，以及对象的 `visibility`。
   - `completeInvestigation()` 已经能在调查完成后揭示 `onInvestigated` 对象，并生成调查报告。
   - 因此“调查前只知道区域 / 地形 / 天气，调查后知道对象”可以沿用当前地图模型实现。

4. **事件模型与 runtime call**
   - `content/events/definitions/*`、`content/events/call_templates/*`、`eventEngine`、`graphRunner` 已经能支持事件定义、事件图、runtime call、选项、wait、objective、event_log、world_history。
   - 事件触发点已经能从 `arrival`、`action_complete`、`idle_time` 等上下文进入事件候选。
   - 因此紧急事件和普通发现事件不需要重新设计模型，重点是把它们接进当前可玩的行动循环。

#### 当前必须补的内容

1. **把普通通话选项通用化**
   - 现在 `CallPage` 对普通通话仍然是角色特判：Garry 有 `move / build / standby / survey`，其他队员大多落到 Mike 的硬编码选项。
   - 需要新增一个普通通话行动生成层，根据队员状态、当前地块、已揭示对象和是否处于事件占用，动态生成：
     - 调查当前区域
     - 去往另一个区域
     - 原地休息 / 停止当前行动
     - 当前地块对象带来的额外交互，例如采集、采矿、建设、修复、扫描

2. **把调查、休息、采集、建设做成所有队员可用的通用行动**
   - 现在调查和建设主要在 `resolveDecision()` 中为 Garry / Mike 写死。
   - 需要把这些动作收敛成通用 `ActiveAction` 创建逻辑：
     - `survey`：目标为当前地块，完成后调用 `completeInvestigation()`，再触发 `action_complete`。
     - `standby`：中断当前行动，清空 `activeAction`，队员状态变成待命，并可触发 `idle_time`。
     - `gather`：来自当前地块已揭示的 resource object，完成后给队员或基地增加对应资源。
     - `build` / `repair`：来自当前地块已揭示的 facility / structure object，完成后更新地块对象或特殊状态。

3. **让地块对象真正影响通话选项**
   - `default-map.json` 中对象已经有 `candidateActions`，但通话页面还没有读取这些字段来生成按钮。
   - 需要在普通通话选项生成时读取当前地块 runtime state：
     - 只使用已经发现 / 已调查后可见的对象。
     - 对 `candidateActions` 映射到行动按钮。
     - 未调查前不展示隐藏对象交互，避免玩家提前知道地块内容。

4. **补齐事件选项到真实行动的桥接**
   - 新事件引擎可以生成 runtime call 和 option，但事件选项目前更多是在 runtime event 内部推进。
   - 如果事件选项表达“让队员前往某地”“处理某个目标”“等待一段时间”，需要把事件结果桥接到当前页面能看到的队员行动：
     - 事件 option 可以创建或请求 `move / survey / gather / event_waiting` 行动。
     - 队员在地图和通讯台上必须显示真实行动进度。
     - 行动完成后再用 `action_complete` 或 `objective_completed` 推进 parent event。

5. **为 MVP 准备前三名队员的明确事件**
   - Mike、Amy、Garry 需要各自至少有一个能稳定触发、能在网页上验证的事件。
   - 可以通过调整初始位置和地图对象降低触发成本，但事件本身应走统一系统：
     - Mike：调查坠毁区 / 湖边异常，揭示对象或触发普通发现。
     - Amy：森林地块调查或待命触发紧急野兽来电。
     - Garry：铁矿地块调查或采矿触发矿床异常报告。

6. **保留现有事件模型，不把新玩法写成更多角色特判**
   - 旧的 `eventSystem.ts` 里 `getEmergencyEventDefinition()`、`getEmergencyChoices()`、`triggerEvents()` 基本是空实现或旧接口，不应继续扩展为主要路线。
   - 新增玩法应优先接入 `events/eventEngine.ts`、`events/graphRunner.ts` 和 `content/events/definitions/*`。
   - `resolveDecision()` 里的角色硬编码可以作为临时参考，但 MVP 应逐步迁移到“通用行动 + 内容配置 + 事件触发”的结构。

#### MVP 的最小落地路径

1. 先做普通通话行动生成：所有队员默认都有“调查当前区域 / 去往另一个区域 / 原地休息”。
2. 再做通用行动结算：`survey`、`standby`、`gather` 至少可被任意队员执行。
3. 然后把地块对象的 `candidateActions` 映射成额外按钮，例如铁矿床出现“采矿”，木材带出现“采集木材”。
4. 最后补 Mike / Amy / Garry 的稳定事件，并确保事件选项会改变真实队员行动或地块状态，而不只是写日志。
