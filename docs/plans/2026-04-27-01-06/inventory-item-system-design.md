# 背包与道具系统策划案

## 1. 概述（What & Why）

背包与道具系统是队员在探索、采集、事件与通话决策中获得、携带、查看并被动使用道具的基础系统。  
资源类道具默认进入触发队员背包，事件也可以显式把道具送入基地库存；标签道具则在需要玩家响应的事件或通话选项中提供可选择的解决手段。  
这个系统让探索结果从一次性文本反馈变成可延续的物品状态，并让“谁带着什么”成为通话决策和风险处理的一部分。

## 2. 设计意图（Design Intent）

背包与道具系统的核心意图，是让探索中的发现不止停留在日志里，而是延续为某名队员“带着什么继续前进”的状态。  
玩家应感受到道具是队员经历的一部分，并在事件或通话响应中，通过是否使用标签道具来做出轻量但有意义的判断。  
MVP 的背包窗口只承担查看职责，避免把道具管理变成频繁整理、搬运或计算负重的负担。  
本系统当前不追求复杂资源经营、容量优化、交换、合成或装备构筑，避免削弱通讯决策与探索叙事的主轴。

## 3. 核心概念与术语（Core Concepts & Terminology）

- **背包**：队员个人携带的道具列表；当前仅记录“谁携带了哪些道具与数量”，不实现负重、容量或格子管理。
- **道具**：可被队员携带、被事件/通话选项检查或产生效果的内容数据；定义来源为 `content/items/items.json`。
- **可消耗道具**：道具定义中标记为使用后会扣除数量的道具，例如一次性补给、照明耗材或医疗用品。
- **不可消耗道具**：道具定义中标记为使用后不扣除数量的道具，例如可重复使用的信号设备、线索物或工具型物品。
- **资源类道具**：语义上代表可累计物资的道具，例如木材、矿石、食物、水等；MVP 中现场获得时默认进入触发队员背包。
- **基地库存**：基地侧的共享资源/物资状态；只有事件明确配置为送入基地库存时，现场获得物才不进入队员背包。
- **道具标签**：附加在道具上的功能标签，用于事件判定与 UI 说明；MVP 词表为食物、照明、医疗、信号、线索。
- **标签道具**：通过标签被规则识别的一组道具；事件或通话响应可检查“是否拥有某类标签道具”，而不必绑定具体道具 ID。
- **使用**：队员在通话选项或事件响应中调用某个道具以产生效果；使用是否扣除数量由道具自身字段决定。
- **消耗**：可消耗道具被使用后从背包中扣除数量；不可消耗道具被使用后保留在背包中。
- **事件/通话响应**：玩家面对事件或通话选项时做出的选择，可触发检查道具、添加道具、使用道具、扣除可消耗道具、更新状态或写入日志。
- **背包专用事件语义**：事件规则中与背包直接相关的条件和效果，例如检查具体道具、检查标签道具、向背包添加道具、使用标签道具。
- **携带者**：当前拥有某个背包道具的队员；道具效果通常只作用于携带者或其正在处理的事件。
- **库存转移**：道具在队员背包与基地库存之间移动；MVP 不提供自由整理或上交流程，只允许事件显式配置转移结果。

## 4. 核心循环与玩家体验（Core Loop & Player Experience）

### 4.1 玩家旅程

1. 玩家在控制中心查看当前资源、系统日志与队员状态，判断是否有新的风险或机会需要处理。
2. 玩家进入通讯台，查看队员通讯录、来电与各队员当前状态。
3. 玩家选择一名队员发起通话，通过对话了解其所在位置、任务进展、携带物与可执行选项。
4. 玩家下达移动、调查、采集、建设、撤离或应急指令，队员开始执行行动。
5. 队员抵达地块或完成行动后，系统触发事件或通话响应。
6. 若事件/通话响应需要道具辅助，玩家在当前语境中选择是否使用符合标签的道具，例如食物、照明、医疗、信号或线索。
7. 若玩家选择使用道具，系统先按简单确定规则从队员背包中选取符合条件的道具；该道具是否消耗由道具字段决定。
8. 行动结果更新队员状态、地块状态、日志、背包与后续可选项；新获得物默认进入执行队员的背包。
9. 玩家可打开背包窗口查看队员携带物，但背包窗口只负责查看，不直接使用、转移或整理道具。
10. 玩家根据结果继续安排下一步行动，在探索收益、队员风险、通讯中断与有限道具之间做取舍。

### 4.2 典型情境

- **高光时刻**：队员发现关键线索物并带在身上；紧急通话中可选择使用医疗、照明、信号等标签道具改变风险、奖励或通讯局面；之前的探索收获在后续事件中产生明确回报。
- **低谷 / 摩擦点**：背包窗口只能查看；多个同标签道具由系统按简单确定规则选取；道具只能在事件/通话响应中使用；MVP 不处理复杂分配、转移、负重或容量管理。

## 5. 机制与规则（Mechanics & Rules）

### 5.1 状态与状态机

- **背包条目**：以“队员 + itemId + quantity”为最小状态单位；同一队员背包内相同 `itemId` 合并为一条数量。
- **道具定义**：每个道具由内容数据定义，至少包含名称、描述、标签、是否可在事件/通话响应中使用、使用后是否消耗数量、展示分类。
- **获得流程**：事件或行动结算产生道具奖励；若未显式指定送入基地库存，则默认加入触发该事件/行动的队员背包；随后写入日志并刷新背包展示。
- **使用流程**：事件/通话进入需要玩家响应的阶段；系统检查队员背包是否存在符合标签且可用于响应的道具；玩家选择是否使用；若使用，系统按简单确定规则选出具体道具；是否扣除数量由道具自身字段决定；事件进入对应结果分支。
- **移除流程**：可消耗道具使用后数量减少 1；数量降为 0 时，从背包条目中移除。

### 5.2 规则与公式

- **默认归属**：事件/采集获得物默认进入触发队员背包；事件显式声明“送入基地库存”时才进入基地库存。
- **数量合并**：`新数量 = 原数量 + 获得数量`。MVP 不执行 `maxStack`，不拆分堆叠，不拒绝超量获得。
- **标签检查**：可用标签道具必须满足 `quantity > 0`、包含事件要求标签、允许用于事件/通话响应。MVP 只做“命中任一指定标签”，不做复杂多标签表达式。
- **玩家响应**：系统不自动使用道具；相关事件应设置为需要玩家响应的类型，让玩家选择“使用对应标签道具”或“不使用”。
- **具体选物**：当多个道具满足同一标签时，MVP 按 `itemId` 字典序/升序选择最小者，保证结果稳定可复现；更合理的选择方式留到后续讨论。
- **消耗判定**：事件只触发“使用”；若道具定义为可消耗，则数量 -1；若定义为不可消耗，则数量不变。事件本身不临时覆盖该规则。
- **反馈**：获得时显示道具名、数量、进入谁的背包或基地库存；使用时显示使用了什么、是否消耗、事件结果如何改变，并写入系统日志。

### 5.3 参数与默认值

| 参数 | MVP 默认值 |
| --- | --- |
| MVP 标签词表 | 食物、照明、医疗、信号、线索 |
| 获得默认归属 | 触发队员背包 |
| 基地库存归属 | 仅由事件显式指定 |
| 同 itemId 处理 | 合并数量 |
| maxStack | 字段保留但不执行 |
| 多个同标签道具 | 按 itemId 升序选择第一个 |
| 背包窗口能力 | 查看队员道具名称、数量、标签、描述、是否可消耗 |
| 背包窗口限制 | 不主动使用、不转移、不丢弃、不拆分、不交易 |
| 暂不启用 | 负重/容量、队员间交换、基地上交流程、合成/拆解/交易、耐久/腐坏/维护费、复杂多标签表达式 |

## 6. 系统交互（System Interactions）

- **依赖于**：`content/items` 的道具定义、队员 `inventory` 状态、事件系统、队员行动/地块状态、时间系统。背包窗口、通讯台、通话页打开时不暂停全局时间。
- **被依赖于**：通话页在事件/决策选项中读取可用道具；通讯台提供队员背包查看入口；事件系统读取背包作为条件、效果目标与结果分支依据；日志系统记录获得、使用与消耗。
- **共享对象 / 状态**：队员当前携带物清单、道具静态定义、队员状态、地块状态、系统日志与存档状态。系统日志只记录变化，不作为背包状态来源。
- **事件 / 信号**：行动完成、通话选项选择、紧急事件升级、背包变更、时间推进都会触发或影响背包展示与结算。地图页面保持只读，不通过地图直接触发背包操作或队员指令。

## 7. 关键场景（Key Scenarios）

### 7.1 典型场景

- **S1 采矿获得矿石**：队员在矿床地块完成采集 → 系统结算矿石奖励 → 矿石作为资源类道具加入触发队员背包，日志显示获得数量与携带者。
- **S2 森林获得木材**：队员在森林地块完成调查并触发找到木材事件 → 玩家确认事件结果 → 木材加入触发队员背包，地块状态与日志同步更新。
- **S3 洞穴使用照明**：队员触发洞穴探索事件且背包有照明标签道具 → 玩家在事件响应中选择是否使用照明 → 使用后进入更安全或更高奖励的结果分支；若该道具为可消耗道具则扣除数量。
- **S4 信号辅助通讯**：队员处于通讯受限或信号不稳定事件，且背包有信号标签道具 → 玩家选择使用信号道具 → 通讯成功率、倒计时压力或额外信息结果得到改善；是否消耗由道具字段决定。

### 7.2 边界 / 失败场景

- **F1 没有符合标签道具**：事件需要某标签 → 系统检查背包无匹配道具 → 不显示使用选项或显示不可用原因，事件按无道具路径结算。
- **F2 多个同标签道具**：背包中多个道具匹配同一标签 → 玩家只选择是否使用该标签 → 系统按 `itemId` 升序选取一个具体道具，避免重复叠加。
- **F3 可消耗道具用完**：使用后数量降为 0 → 系统移除该背包条目 → 后续事件不再把它视为可用道具。
- **F4 背包窗口只查看**：玩家打开背包窗口 → 只能查看名称、数量、标签、描述与是否可消耗 → 不能直接使用、丢弃、转移、拆分或交易。

## 8. 取舍与反模式（Design Trade-offs & Anti-patterns）

- **取舍 1：背包优先，而非基地库存优先**：现场获得物默认进入触发队员背包，因为本系统要服务队员处境、通话决策与事件结算；基地库存只在事件显式指定时接收。
- **取舍 2：事件 / 通话使用，而非背包主动使用**：背包窗口负责查看，道具使用发生在事件或通话响应中，避免背包界面取代通讯链路。
- **取舍 3：道具字段决定消耗，而非事件临时决定**：事件只触发“使用”，是否扣数量由道具自身定义，保证同一道具在不同事件中的玩家预期一致。
- **取舍 4：简单 `itemId` 选物，而非玩家选择具体道具**：多个同标签道具可用时，MVP 先按 `itemId` 升序选取，换取实现稳定与流程轻量；更合理选取方式留到后续。
- **取舍 5：不做 `maxStack` / 负重 / 容量**：本阶段不引入堆叠上限、重量、背包格数、容量惩罚等规则，避免背包系统提前变成资源管理游戏。
- **要避免的反模式**：无事件/通话/行动用途的无意义库存；玩家无确认时自动消耗关键物；过早引入交易、合成、物流、仓储经济；让事件绕过道具定义临时改变消耗规则；让背包界面成为主要操作入口。

## 10. 本轮范围与阶段拆分（Scope & Phasing for This Round）

### 10.1 MVP（本轮必做）

- 补齐通讯台队员背包窗口：查看道具名称、数量、标签、描述、是否可消耗。
- 事件/采集获得道具默认加入触发队员背包，并记录日志与 UI 反馈。
- 新增清晰的背包专用事件语义：检查具体道具、检查标签道具、向队员背包添加道具、使用标签道具。
- 事件/通话响应可基于食物、照明、医疗、信号、线索五类标签提供“是否使用道具”的玩家选择。
- 道具是否消耗由道具自身字段决定；可消耗道具使用后扣数量，不可消耗道具使用后保留。
- 多个同标签道具可用时，MVP 按 `itemId` 升序选择具体道具。
- 不执行 `maxStack`，相同 `itemId` 只合并数量。
- 覆盖四个验收场景：采矿获得矿石、森林获得木材、洞穴使用照明、信号辅助通讯。

### 10.2 Later（未来再做，明确本轮不做）

- 更合理的具体道具选择规则，例如玩家手动选择、优先消耗低价值物、按上下文优先级选择。
- `maxStack` 的实际堆叠执行逻辑。
- 基地仓库/上交/回收/统一库存的完整流程。
- 队员之间交换物品。
- 背包负重与容量规则。

### 10.3 不做（Out of Scope，避免范围膨胀）

- 不实现背包负重、超重惩罚或容量限制。
- 不实现队员间手动或自动物品交换。
- 不实现背包界面的主动使用按钮。
- 不实现合成、拆解、买卖、交易价格。
- 不实现耐久下降、腐坏、维护费或定期损耗。
- 不实现复杂多标签表达式、道具品质评分或多件组合使用。
- 不把 Debug toolbox 或地图页面变成道具操作入口。

## 11. 本轮验收与风险（Acceptance & Risks）

### 11.1 Player Stories / Play Scenarios（验收切片）

#### PS-001: 采矿获得矿石
- **作为**：通过通讯指挥队员采集资源的玩家
- **我能**：让队员在矿床地块完成采集后获得矿石道具
- **以便**：确认事件/行动奖励能进入触发队员背包，而不是只显示日志或进入基地库存
- **验收标准**：
  - [ ] 采矿结算后，触发队员背包新增或叠加矿石道具。
  - [ ] 通讯台背包窗口能看到矿石名称、数量、标签与描述。
  - [ ] 系统日志记录获得数量与携带者。
- **不包含**：矿石精炼、自动采矿、多人协作采矿。
- **优先级**：P0

#### PS-002: 森林调查获得木材
- **作为**：派遣队员调查森林地块的玩家
- **我能**：在森林调查事件中获得木材道具
- **以便**：确认调查事件也能发放背包奖励
- **验收标准**：
  - [ ] 森林调查完成后可触发找到木材事件。
  - [ ] 木材加入触发队员背包，并按相同 `itemId` 合并数量。
  - [ ] 地块状态与系统日志能体现调查/获得结果。
- **不包含**：森林资源枯竭、生态影响、木材加工链。
- **优先级**：P0

#### PS-003: 洞穴使用照明
- **作为**：处理洞穴探索风险的玩家
- **我能**：在洞穴事件中选择是否使用照明标签道具
- **以便**：通过已携带道具改变风险、奖励或结果文本
- **验收标准**：
  - [ ] 队员有照明标签道具时，事件/通话响应出现“使用照明”的选择。
  - [ ] 玩家选择使用后，系统按 `itemId` 升序选取具体道具。
  - [ ] 若该道具为可消耗道具，数量 -1；否则数量不变。
  - [ ] 事件结果和日志明确说明照明发挥了作用。
- **不包含**：实时光照、洞穴迷宫、持续燃料管理。
- **优先级**：P0

#### PS-004: 信号辅助通讯
- **作为**：依赖远程通讯进行指挥的玩家
- **我能**：在通讯受限事件中选择是否使用信号标签道具
- **以便**：改善通讯成功率、倒计时压力或获得额外信息
- **验收标准**：
  - [ ] 通讯受限事件能检查信号标签道具。
  - [ ] 玩家选择使用后，事件结果体现通讯改善。
  - [ ] 是否消耗由道具字段决定，并写入日志。
- **不包含**：通信网络建设、信号塔布局、实时信道管理。
- **优先级**：P1

### 11.2 成功标准（Success Criteria）

- [ ] 背包不再只是预设展示，而能被事件/行动实际新增、读取与更新。
- [ ] 事件/采集获得物默认进入触发队员背包。
- [ ] 食物、照明、医疗、信号、线索标签可作为事件/通话响应的判定依据。
- [ ] 玩家能明确选择是否使用标签道具；系统不自动偷用道具。
- [ ] 道具是否消耗由道具字段决定，并在 UI/日志中可理解。
- [ ] 背包窗口只查看，不引入主动使用、转移、丢弃、拆分或交易。

### 11.3 风险与缓解（Risks & Mitigations）

- **R1 标签词表失控**：标签可能出现同义词、重复词或只服务单个事件的临时词。  
  - **缓解**：MVP 固定五类标签；新增标签需另开讨论。
- **R2 道具被错误消耗**：检查条件、预览选项与实际使用可能混淆。  
  - **缓解**：区分“检查可用”和“玩家确认使用”；只有确认使用后才按道具字段扣除。
- **R3 背包 / 基地边界混乱**：资源类道具可能被误送基地或误入背包。  
  - **缓解**：默认进触发队员背包；只有事件显式写明才送基地。
- **R4 UI 信息不足**：玩家可能不知道缺少什么、用了什么、是否消耗。  
  - **缓解**：事件选项、背包窗口和日志都显示标签、数量、消耗结果与不可用原因。
- **R5 范围膨胀**：背包容易扩展成容量、生产、交易或通信网络系统。  
  - **缓解**：本轮只验收四个切片，Later 内容记录但不实现。
