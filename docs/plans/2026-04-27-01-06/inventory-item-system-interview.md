**Q1:** 是否需要在动手访谈前，先探索现有 docs（core-ideas、gameplay/、ui-designs/）？
**A1:** 扫描现有 docs

**Q2:** 以下哪些方向值得做互联网研究？（可多选）
**A2:** 事件驱动物品经济, 标签化道具设计

**Q3:** 第一个关键决策：事件/采集获得的“资源类道具”（木材、矿石、食物、水等）在 MVP 中应该进入哪里？我推荐“背包优先，事件可显式送基地”，因为它最贴合你提出的“事件获得道具加入背包”，也保留基地库存的未来兼容性。
**A3:** 背包优先（推荐）

**Q4:** 第二个关键决策：当事件需要某个道具标签（例如“照明”）且队员背包里有可用道具时，MVP 应该如何处理“使用/消耗”？我推荐“显式事件选项 + 低价值自动选择”，这样既能让玩家理解道具发挥了作用，又不需要做复杂背包操作。
**A4:** 不如将这一类事件始终设置为需要玩家响应的类型，会让玩家来做选择是否使用对应的道具，而道具的选取可以先用最简单的方式实现（比如始终选取id较低的物品），我们后续可以讨论更合理的方式

**Q5:** 第三个关键决策：道具“使用”后是否一定会被消耗？我推荐把“使用但不消耗”和“使用并消耗”作为事件配置里的两种明确效果，因为工具类与消耗品的玩家预期不同。
**A5:** 配置决定（推荐）

**Q6:** 第四个关键决策：背包 UI 的 MVP 形态应该是什么？我推荐补齐“通讯台查看背包”的独立二级窗口，同时在通话/事件选项里显示可用或将消耗的道具标签；这样符合现有 UI 文档，也不会让背包操作压过通话决策。
**A6:** 独立背包窗口（推荐）

**Q7:** 第五个关键决策：MVP 不做负重/容量时，是否要执行已有的 `stackable / maxStack` 堆叠上限？我推荐暂时“只做数量合并，不执行 maxStack”，避免把本轮从事件道具闭环拖进库存容量细节。
**A7:** 不执行 maxStack（推荐）

**Q8:** 第六个关键决策：事件系统的内容语言应该如何表达背包相关条件与效果？我推荐新增清晰的背包专用语义，而不是继续把 `addResource/removeResource` 混用成背包操作，这样资源库存和队员背包边界更清楚。
**A8:** 新增背包语义（推荐）

**Q9:** 第七个关键决策：MVP 的功能性道具标签词表先覆盖哪些？我推荐先控制在少量、会被事件实际使用的标签，避免标签词表失控。
**A9:** 食物, 照明, 医疗, 信号, 线索

**Q10:** 第八个关键决策：MVP 中玩家是否可以在背包窗口里“主动使用”道具，还是只能在事件/通话响应里使用？我推荐“背包窗口只查看，使用发生在事件/通话选项中”，这样符合指令必须经通讯与通话发出的原则，也避免无上下文使用道具。
**A10:** 只在事件/通话使用（推荐）

**Q11:** 第九个关键决策：本轮 MVP 的验收切片应该优先覆盖哪些具体场景？我推荐至少覆盖“获得物品”和“标签使用改变事件结果”两个闭环，才能证明背包系统不是纯展示。
**A11:** 采矿获得矿石（推荐）, 森林获得木材（推荐）, 洞穴使用照明（推荐）, 信号辅助通讯

**Q12:** 最后一个范围决策：以下哪些明确放到 Later / Out of Scope，避免本轮范围膨胀？我推荐全部暂缓；本轮聚焦“事件获得、事件使用、标签判断、背包展示”。
**A12:** 负重/容量限制, 队员间交换物品, 基地仓库/上交流程, 合成/拆解/交易, 耐久/腐坏/维护费, 背包主动使用
