## Q1：这份全局日志主要是给谁看的？（决定颜粒度与叙事风格）

选项：

- 主要给玩家看：叙事回顾 — 类似"航行日志"，玩家读得懂的摘要句；带口吻/重要性等表达，内容量竟体。
- 主要给开发者：调试/产品分析 — 全量记录玩家动作 + 事件 trigger / effect / 结果，以便复现 bug、调数值、看玩家行为。
- 主要给 LLM / AI 复读 — 供 AI 读取作为记忆 / 叙事总结；需要结构化严谨 + 可读项零混淆。
- 三者都要，分层 — 同一条时间线上同时产出叙事层（面向玩家）与调试层（面向开发者/AI）；实现复杂度高。

**用户答**：主要给开发者：调试/产品分析

---

## Q2：你希望这份开发日志只存在哪里？（决定载体 / 授权 UX / 跨浏览器可用性）

选项：

- 用户可见的本地文件（FSA） — 在 Chrome/Edge 里选一个路径，后续 append 到这个 .jsonl 文件。优：玩家能直接打开 / 发开发者。劣：仅 Chromium、每次启动要重新授权、Safari/Firefox 不能用。
- 隐形持久化 + 手动导出（推荐） — 写入 IndexedDB 或 OPFS（玩家看不到），在开发者面板提供"导出本局日志 .jsonl"按钮。优：跨浏览器、零授权、高性能。劣：需手动调出文件。
- 两者都要，双写 — 热区写 IndexedDB，同时双写一份到 FSA 文件（设置中开启）。优：实时可见 + 跨浏览器兜底。劣：两套路径、双写一致性复杂度。
- 仅内存 + 退出时一次性下载 — 运行期写入内存数组，玩家点 "下载日志" 或关闭前一次性导出。优：最 MVP。劣：浏览器 / 标签崩溃则丢全部、与"增量写入"字面说法不符。

**用户答**：首先我需要一个持久化存储的文件，然后在debug toolbox里添加在页面上展示日志的功能

## Q3（澄清）："持久化存储的文件"具体是哪种？

选项：

- 用户磁盘上的 .jsonl 文件（FSA） — 首次启动调出"选择保存位置"，后续运行期不断 append。你能直接用 VSCode / tail / less 打开。仅 Chrome/Edge，每次页面刷新要重新授权（点一下按钮）。
- OPFS 文件 + 导出按钮（推荐） — 文件存在浏览器私有文件系统里（跨会话保留、零授权、高性能）；DebugToolbox 里提供"导出当前存档日志"按钮拿到 .jsonl。跨浏览器均可用。
- 两者都要 — 运行期 OPFS 热写保证不丢；同时双写 / 定期同步到 FSA 用户可见文件。复杂度明显上升，需处理双写一致性。

**用户答**：OPFS 文件 + 导出按钮（推荐）

## Q4：你希望日志记到多细？（决定条目数量级 + 接入点数量）

选项：

- 全量记录玩家指令 + 事件发生/走向/结果（推荐） — 每个玩家决策（通讯选项/移动/调查/待命/停止）一条；每个事件 trigger、节点跳转、结果一条；复用现有 EventLog 为叙事层。量级：每局数百条。
- 只记高价值事件（现有 EventLog 层面） — 仅复用 event_logs 现有输出，给 EventLog 加个"型制信封" + 索引后写着。量级：每局几十条。优：改动极小。劣：换不回玩家指令、调试价值低。
- 进一步加"系统低层事件" — 在选项 A 基础上另记资源变化、行动状态机跳转、effect 逐条应用。量级：每局数千条。优：可动 replay。劣：存储 / 展示冲击明显。

**用户答**：全量记录玩家指令 + 事件发生/走向/结果（推荐）

## Q5：重置存档时，旧日志怎么办？

选项：

- 保留在 OPFS 里（推荐） — 旧文件重命名为带 run_id / 时间戳的归档名（如 run-2026-05-01-0244.jsonl），OPFS 继续保存；DebugToolbox 列出历史运行供查看/导出。
- 直接丢弃，只保留当前运行 — 重置后旧文件删除，只留当前一份。极简，但调试价值损失、不能事后查上局出了什么。
- 保留 N 份后滚动丢弃 — 默认保留11 份运行（当前 + 近 N 次）。兑底不炸存储，也能横向对比几局。需要一个 N 数值以及"判定哪份是最老"的规则。

**用户答**：保留 N 份后滚动丢弃

## Q6（首次问 schema envelope）：每条日志用多重的 envelope？（决定未来 replay / 导入能力）

选项：

- 最小信封（推荐） — 每行固定包含：seq（单调递增）、log_version、game_version、run_id、occurred_at_game_seconds、occurred_at_real_time、type、source、payload。未来加字段不带走老日志。代价极小（8 个固定字段）。
- 极简：只记 type + payload + time — 不加 seq / 版本 / run_id，到需要时再补。优：上手快。劣：后续升级要老新日志兑底（upcaster），事件源圈公认事后难补。
- 完整 CloudEvents 信封 — 上业界标准：specversion / id / source / type / time / subject / data。优：将来可接外部消费侧。劣：项目还是单机状态，重型。

**用户答**（反问）：这个日志大概是长什么样的？为什么有这么多字段

（主 agent 解释了字段含义、最小信封长什么样、每个字段为什么存在、极简方案的代价、调用方只填 type 和 payload。然后重新提问。）

## Q6'（重问）：看完了，选哪种 envelope？

选项：

- 最小信封（8 个固定字段，推荐） — seq / log_version / game_version / run_id / occurred_at_game_seconds / occurred_at_real_time / type / source / payload。调用侧只填 type + payload，其他自动填。
- 极简三字段：type + payload + time — 现在轻，将来加字段要与老日志兼容。如果你认为这份日志只会在一两个版本内使用，这样也 OK。
- 加上"git_commit"与"debug_session_id"两个额外字段 — 在推荐信封基础上，加 git_commit（某次 build 的 commit hash，调试可以精确定位代码版本） + debug_session_id（断点续写 / 跨标签区分会话）。

**用户答**：最小信封（8 个固定字段，推荐）

## Q7：每条日志什么时候落盘？（决定崩溃丢失范围 + 性能）

选项：

- 批量 flush（推荐） — 内存缓冲，每 500ms 或满 50 条写一次 OPFS；事件结算 / reset / 退出前强制 flush。优：性能好。劣：标签猛关可能丢最后 <500ms 的几条。
- 每条立即写 — 现写现 close，崩溃只会丢当前这一条。劣：每帧多条事件时 出现写入风暴（但 OPFS sync handle 在 Worker 里其实还好）。
- 定期落盘（1 秒 / 心跳） — 在 settleGameTime 的每秒心跳末尾一次性 flush 本帧产生的所有日志， + reset/退出强制 flush。优：节奏与游戏心跳同步。

**用户答**：批量 flush（推荐）

## Q8（合并）：滚动保留多少份运行？

选项：

- 保留 10 份（推荐） — 当前 run + 近 9 次重置存档。对 OPFS 配额几乎无压力，足以横向对比几局。
- 保留 3 份 — 当前 + 近 2 次。最节约，能对比"上一局 / 上上局"。
- 保留 30 份 — 保留很多历史，适合长跨度调试。文件累计体积明显，要在 DebugToolbox 里列表扫也略繁。

**用户答**：保留 10 份（推荐）

---

## Q8（合并）：DebugToolbox 里的日志面板要哪些能力？（多选）

选项：

- 实时展示当前 run 的日志流 — Tail 式展示最新 N 条，自动滚动到底。基础能力。
- 按 type / source 过滤 — 例如只看 player.* 或只看 event.resolved；调试常用。
- "导出当前 run .jsonl" 按钮 — 从 OPFS 拿出当前文件并触发浏览器下载。
- "查看历史 run 列表" 按钮 — 列出 OPFS 里存在的 run 归档，点任意一份可查看 / 导出 / 删除。

**用户答**：实时展示当前 run 的日志流, 按 type / source 过滤

## Q9（澄清）：导出与历史查看这两个能力反过来同步一下：

选项：

- MVP 只要 "导出当前 run" 按钮，历史 run 列表放 Later — MVP 能拿到当前这局的 .jsonl 就够调试了，旧 run 只静默在 OPFS 里滚动保留。
- 导出与历史都进 MVP — 一次性把"导出当前 run"与"历史 run 列表（能查看/导出/删除任一历史份）"都做了。
- 两个都放 Later，MVP 只依靠实时面板 — 调试全靠 DebugToolbox 实时面板，OPFS 只作"不丢"的兑底，事后拿文件需要手写小工具。不推荐。

**用户答**：导出与历史都进 MVP
